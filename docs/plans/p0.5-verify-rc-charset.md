## 1. Context

`amnezigo` is a Go CLI that generates AmneziaWG configuration files. CPS (custom packet specifier) templates use tags like `<b 0xHEX>`, `<r N>`, `<rc N>`, `<rd N>`, `<t>` to describe synthetic packets used for I-packets.

The `<rc N>` tag means **"emit N random characters from the letters alphabet"**. The reference implementation `amneziawg-go/device/obf_randchars.go` uses an alphabet of **52 letters only** — `[a-zA-Z]`. The Habr article describing AWG 2.0 incorrectly states `[a-zA-Z0-9]` (52+10 = 62 chars). If `amnezigo`'s documentation, comments, or any future generator function follow the article instead of the source code, the on-the-wire bytes drift from real userspace output, and the mimicry quality drops because digits are statistically distinguishable from letter-only strings.

**Critical clarification about `amnezigo`'s architecture**: `BuildCPSTag("rc", "8")` returns the **literal string** `"<rc 8>"` — `amnezigo` does **not** generate the random character bytes itself. Byte generation happens at packet-emission time inside `amneziawg-go` userspace. Therefore this PR is **not** about asserting bytes against a regex; it is about (a) **documenting the contract** in code, (b) **anchoring the contract to a named constant**, and (c) **adding a regression test** that fails loudly if the contract drifts (e.g., a future contributor edits the comment/constant to include digits because they read the Habr article).

## 2. Goal

Lock the `<rc>` semantics to `[a-zA-Z]` (52 letters, **no digits**, **no symbols**) and prevent future drift:

- Introduce a single source-of-truth constant `cpsRcAlphabet` containing the exact 52-character alphabet from `amneziawg-go/device/obf_randchars.go`.
- Make `BuildCPSTag`'s docstring explicit: `<rc N>` → `N` random letters from `[a-zA-Z]` (52 chars), produced by the AmneziaWG userspace receiver — `amnezigo` only emits the tag literal.
- Add a regression test that validates the constant against the canonical reference alphabet, catching any future drift (e.g., accidental addition of digits) at CI time.
- Document, in code comments, that **digits go through `<rd>` and combinations through tag concatenation** (e.g., `<rc 4><rd 2>` for 4 letters + 2 digits) — answers the roadmap brainstorm question.

## 4. Decision Matrix — Scenarios A vs B

The roadmap requested coverage of two scenarios. Both are documented here for posterity; the chosen path is **Scenario A**.

### Scenario A — Current code is correct (chosen)

**Trigger:** No `amnezigo` code generates `<rc>` random bytes; the existing emission is only the tag literal `"<rc N>"`. Documentation is loose ("ASCII chars") but does not contradict the spec.

**Plan:**

1. Add the named constant `cpsRcAlphabet` (single source of truth for the contract).
2. Tighten the `BuildCPSTag` docstring at `cps.go:38` to specify `[a-zA-Z]` (52 letters) and reference `amneziawg-go/device/obf_randchars.go`.
3. Add a regression test that asserts the constant equals the canonical reference alphabet exactly (52 chars, no digits, no symbols, sorted lowercase-then-uppercase).
4. Document the **digit-mixing pattern** (`<rc 4><rd 2>`) in the docstring and in the regression test's comment, satisfying the roadmap brainstorm question.

### Scenario B — Current code is wrong (NOT applicable here)

**Trigger:** Some `amnezigo` function would have generated bytes from a wrong alphabet (e.g., `[a-zA-Z0-9]`). Would also apply if the docstring explicitly said "alphanumeric" or "letters and digits".

**Plan (would be):**

1. Replace the wrong charset with `[a-zA-Z]` at every callsite.
2. Then everything from Scenario A.

**Why not applicable:** verified by direct inspection (§ 3) — no byte-generator exists, and the existing docstring uses the vague-but-not-wrong term "ASCII chars". Tightening the docstring is sufficient.

## 6. Step-by-Step Implementation (TDD)

### Step 1 — Test first (Red)

Add a new regression test to `cps_test.go` that asserts the contract. Place it next to the other CPS tests (right after `TestBuildCPSTagRandomChars`, around `cps_test.go:31`).

**Test name**: `TestRcAlphabetMatchesReference`.

**Why this name**: describes the behavior verified ("the local alphabet matches the upstream reference"), no internal references / no ticket numbers (per `code-quality.md`).

**Test body shape** — the test references the **constant that does not yet exist** (`cpsRcAlphabet`), so it fails to compile, satisfying the Red step:

```go
func TestRcAlphabetMatchesReference(t *testing.T) {
    // Reference alphabet from amneziawg-go device/obf_randchars.go:
    // 52 ASCII letters, lowercase first, then uppercase, sorted alphabetically.
    // Source of truth — DO NOT modify without verifying upstream first.
    const reference = "abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ"

    if cpsRcAlphabet != reference {
        t.Errorf("cpsRcAlphabet drift: got %q, want %q", cpsRcAlphabet, reference)
    }

    // Defensive subtests — guard against partial drift that string-equality
    // would catch but which produce harder-to-read failure output.

    t.Run("length is 52", func(t *testing.T) {
        if got := len(cpsRcAlphabet); got != 52 {
            t.Errorf("cpsRcAlphabet length = %d, want 52 (26 lowercase + 26 uppercase)", got)
        }
    })

    t.Run("no digits", func(t *testing.T) {
        for _, r := range cpsRcAlphabet {
            if r >= '0' && r <= '9' {
                t.Errorf("cpsRcAlphabet contains digit %q — must be letters only per amneziawg-go/device/obf_randchars.go", r)
            }
        }
    })

    t.Run("only ASCII letters", func(t *testing.T) {
        for _, r := range cpsRcAlphabet {
            isLower := r >= 'a' && r <= 'z'
            isUpper := r >= 'A' && r <= 'Z'
            if !isLower && !isUpper {
                t.Errorf("cpsRcAlphabet contains non-letter %q — must be [a-zA-Z]", r)
            }
        }
    })
}
```

**Brainstorm-question coverage**: the docstring of this test (or a sibling comment in `cps.go` near the constant) must mention the digit-mixing pattern: "**For mixed letter+digit fields, use tag concatenation**: `<rc 4><rd 2>` produces 4 letters then 2 digits. Do not introduce a separate alphanumeric tag."

### Step 2 — Code change (Green)

**`cps.go:13-26`** — extend the const block. Append after the existing constants (do **not** reorder or rename existing ones; that is out of scope):

```go
// cpsRcAlphabet is the canonical alphabet used by amneziawg-go to fill
// <rc N> tags at packet emission time. 52 ASCII letters, lowercase first
// then uppercase, sorted alphabetically.
//
// Source of truth: amneziawg-go device/obf_randchars.go.
//
// IMPORTANT — common pitfall: the Habr article describing AWG 2.0
// incorrectly states the alphabet is [a-zA-Z0-9]. It is NOT.
// For mixed letter+digit fields, use tag concatenation:
//   <rc 4><rd 2>  →  4 letters followed by 2 digits.
// Do NOT introduce a separate alphanumeric tag.
//
// amnezigo itself only emits the tag literal "<rc N>"; the bytes
// are produced by the AmneziaWG receiver. This constant exists to
// document the contract and to anchor the regression test.
cpsRcAlphabet = "abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ"
```

## 10. Out of Scope (Explicit)

The following are **deliberately excluded** from this PR:

- Adding a runtime `<rc>` byte-generator to `amnezigo` (it doesn't need one; the userspace receiver does it).
- Adding `<ds>`, `<dz N>`, `<d>` tags (those are P1.1 / P2.2).
- Removing `<c>` tag (P0.1).
- Fixing `<t>` size (P0.2).
- Pairwise size collision validation (P0.3).
- H1-H4 type-id forbiddance (P0.4).
- Refactoring unrelated code in `cps.go` (e.g., the brittle `<r[c\|d]?>` regex, comment-style cleanup, helper extraction).
- Fixing any template misuse of `<rc>` if discovered (file a follow-up issue; do not fix here — § 6 Step 3).
- Adding a property-style 1000-iteration test (no runtime generator → would be vacuous; see Step 1 footnote).
- Renaming or relocating the constant (`cpsRcAlphabet`) into a separate file or sub-package.
- Updating `docs/roadmap.md` to reflect that P0.5 is done — that happens in a sweep PR after all P0 items land.

If any of these come up during implementation, **do not include them**. Note them for the next PR or an issue.
