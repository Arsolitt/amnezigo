# P0.1 ‚Äî Remove `<c>` (Counter) Tag

> Atomic execution plan for roadmap item **P0.1** (see `docs/roadmap.md`).
>
> **Status**: Ready for implementation
> **Estimated diff**: ~50 LOC across 3 source files + 2 test files + 1 doc file (deletions, no new features)
> **Depends on**: **P0.2** must be merged first (introduces `cpsCounterSize` constant and the split `<c>` regex/loop branch ‚Äî this PR removes them as dead code).
> **Blocks**: nothing strictly. P0.3, P0.4, P0.5 are on independent code paths, but P0.3's collision validation will be cleaner without `<c>` muddying the size accounting picture.

## 1. Context

`amnezigo` is a Go CLI that generates AmneziaWG configuration files. CPS (custom packet specifier) templates use tags like `<b 0xHEX>`, `<r N>`, `<rc N>`, `<rd N>`, `<t>` to describe synthetic packets used for I-packets.

Historically the generator also supported `<c>` (counter). However, `<c>` exists **only** in `amneziawg-linux-kernel-module/src/junk.c`. The reference Go-userspace implementation (`amneziawg-go`) never accepted it ‚Äî and as of commit `e7ef4339` (2026-03-23) it was removed even from the public Go reference. Configs containing `<c>` fail to load on:

- iOS / iPadOS (AmneziaVPN client)
- Android (AmneziaVPN client)
- Windows (AmneziaVPN GUI)
- macOS (AmneziaVPN GUI)
- Any deployment using `amneziawg-go` userspace

The error surfaced is `unknown tag <c>`. Since mobile compatibility is mandatory (decision 1 in roadmap), `<c>` must be excised from the generator.

The good news: `<c>` is **not actually used** by any current protocol template (`quic.go`, `dns.go`, `dtls.go`, `stun.go`) ‚Äî verified via grep during inventory. Random-mode tag generation in `cps.go:178` already excludes `"c"` from `allTagTypes`. So the practical user impact is purely API surface: `BuildCPSTag("c", "")` returns a string that breaks mobile clients, and `mapTagType("counter")` returns `"c"` which then propagates the same poison through templates that opt in.

## 2. Goal

Eliminate every codepath that can emit `<c>` into a generated config:

- Remove the `"c"` case from `BuildCPSTag`.
- Remove the `"counter" ‚Üí "c"` mapping from `mapTagType`.
- Remove `"c"` from comments listing supported tag types.
- Remove `cpsCounterSize` constant and the `<c>` regex branch in `calculateCPSLength` (introduced in P0.2 as a bridge ‚Äî now dead code).
- Delete `<c>`-only tests; rewrite `<c>`-mixed tests to drop the `<c>` portion.
- Manually verify generated configs load cleanly on `amneziawg-go`.

Out of scope (deferred to other roadmap items):

- Pairwise size-collision validation ‚Äî P0.3.
- Forbidding H1-H4 ranges containing 1-4 ‚Äî P0.4.
- Verifying `<rc>` charset ‚Äî P0.5.

## 3. Key Finding from Inventory

**Protocol templates do not currently use `<c>`.** A grep for `"counter"` and `<c>` across `quic.go`, `dns.go`, `dtls.go`, `stun.go` returned **zero matches**. This means the brainstorm question "pure `<r N>` or context-appropriate `<rd N>` for replacement?" is **moot for the current codebase** ‚Äî there is nothing to replace in templates.

The plan therefore drops the "template replacement" workstream entirely and replaces it with a simpler "verification grep" step: confirm zero `<c>` template usages survive into the merged tree.

This finding also means random-mode (`generateRandomTags`) already excludes `<c>` (`cps.go:178`: `allTagTypes := []string{"b", "r", "rc", "rd", "t"}`). No change needed there.

The only places where `<c>` can leak into generated output today are:

1. A direct caller invoking `BuildCPSTag("c", "")` ‚Äî public API.
2. A future template author using `{Type: "counter", Value: ""}` in `[]TagSpec` ‚Äî funnelled through `mapTagType`.

Removing both eliminates the leak.

## 7. Constants Naming Decision

**Decision**: remove `cpsCounterSize` outright. Do **not** keep it "for future use".

**Rationale:**

- The constant was introduced in P0.2 as a bridge to delete it cleanly here without leaving a magic literal `8`.
- After this PR, no code path consumes `<c>`; a constant for a non-existent feature is dead weight that pollutes IntelliSense and invites accidental re-introduction.
- If kernel-module-only deployments ever become a supported feature again, a separate code path with its own clearly-scoped constant should be introduced ‚Äî bolting onto a leftover `cpsCounterSize` would be worse than a fresh design.
- `cpsTimestampSize` survives unchanged.

## 8. Risks and Edge Cases

| Risk | Assessment |
| --- | --- |
| User has hand-written `<c>` in a custom config and runs `amnezigo` | **Cannot happen via parser today.** `parser.go:149-158` shows `I1`-`I5` parsing is commented out ‚Äî there is no codepath that reads CPS strings from disk. Custom configs are not a supported input. If a user has a hand-rolled `client.conf` with `<c>`, they edited it themselves; `amnezigo` will not touch it. |
| Future template author uses `{Type: "counter", ...}` | After this PR, `mapTagType("counter")` returns `""`, which causes `BuildCPSTag("", "")` to also return `""`. | Mobile clients that *previously* worked with `<c>` (kernel module Linux only) | **None affected.** Kernel-module Linux accepts both `<c>` and equivalent random sequences. Removing `<c>` does not break it; replacement (none needed since templates already don't use `<c>`) is moot. |
| `cpsCounterSize` referenced from outside the package | **Verified impossible.** Constant is unexported (lowercase `c`); package boundary precludes external use. |
| Random-mode silently changes behavior | **No change.** `generateRandomTags` already excludes `"c"` from `allTagTypes` (`cps.go:178`). Random output is byte-identical before/after this PR. |
| Tests that pass *vacuously* after `<c>` deletion | **Mitigated.** Every `<c>`-bearing test row is either deleted or rewritten with a meaningful replacement (`bytes_counter_timestamp` ‚Üí `bytes_timestamp`, `mixed` retains its mixed-tag intent). |

## 11. Commit and PR Plan

**Single atomic commit.** Code deletion + test cleanup + documentation update = one logical unit ("remove `<c>` tag end-to-end"). Splitting into "delete code" and "delete tests" commits would leave one of them in a broken state on the branch (bad for `git bisect`). Combined diff stays under ~50 LOC of meaningful change; well within atomic-PR guidelines.

- **Branch**: `feat/cps-remove-counter-tag`
- **Commit title**: `feat(cps): remove <c> (counter) tag for mobile compatibility`

  Rationale for `feat(cps):` over `fix(cps):` or `refactor(cps):`: this is a public API change (removes `BuildCPSTag("c", ...)` and `mapTagType("counter")` from accepted inputs) with user-visible behavioral shift (configs no longer fail on mobile). `feat` for a removal that enables a new platform set is the conventional-commits norm; releases tooling can then surface it as a breaking-change candidate if needed.

- **Commit body** (single paragraph, no Claude attribution, no issue refs):

  ```text
  Remove the <c> (counter) tag from the CPS generator. The tag is
  recognised only by amneziawg-linux-kernel-module; the Go-userspace
  reference (amneziawg-go) rejects it with "unknown tag <c>", causing
  generated configs to fail on iOS, Android, Windows, and macOS.
  Mobile compatibility is mandatory per the project roadmap. Drops
  the "c" branch from BuildCPSTag, the "counter" mapping from
  mapTagType, the cpsCounterSize constant introduced as a bridge in
  P0.2, the <c> regex in calculateCPSLength, and all <c>-bearing
  test cases. Protocol templates already do not use <c>; random-mode
  generation already excludes it. Roadmap P0.1.
  ```
