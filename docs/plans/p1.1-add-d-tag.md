## 1. Context

`amnezigo` is a Go CLI that generates AmneziaWG configuration files. CPS (custom packet specifier) strings drive the synthetic I1-I5 packets used to mimic real protocols. Today the generator supports five tags: `<b 0xHEX>`, `<r N>`, `<rc N>`, `<rd N>`, `<t>`.

AmneziaWG 2.0 added a sixth tag — **`<d>`** — described in the project roadmap (`docs/roadmap.md` § P1.1) as "the most powerful mimicry primitive". Unlike the other five, `<d>` does **not** carry data on its own and contributes **zero bytes** to the CPS byte budget at generation time. Instead, it is a **passthrough marker** the AmneziaWG userspace receiver expands at packet-emission time by reusing a value previously emitted in an earlier I-packet position.

Reference implementation: `amneziawg-go/device/obf.go`. The `<d>` tag is recognised by the receiver's CPS state machine; the writer side simply emits the literal bytes captured from a prior interval. **Scope is implicit by position**: there is no scope identifier in the `<d>` tag itself — the runtime resolves it via state-machine state, not via a value carried in the tag (verify in source as part of step 0).

**Why this matters for mimicry**: a typical real-world handshake reuses the same connection identifier across multiple packets (QUIC's DCID, TLS's session-id, DNS's transaction ID, etc.). Without `<d>`, every I-packet of a single AWG flow contains independently-random bytes — itself a fingerprint, since real protocol flows correlate. With `<d>`, `i1` can emit a fresh DCID and `i2 = <b 0x...><d>` reuses that same DCID, making the synthetic flow look like a continuation of the same simulated session.

`amnezigo` does **not** generate `<d>` bytes itself — same architectural pattern as `<rc>` / `<rd>` / `<t>` (verified during P0.5: `amnezigo` is a tag-emitter, not a payload-byte-generator). This PR therefore adds:

- A new public-API path: `BuildCPSTag("d", "")` → `"<d>"`.
- A new user-facing type name: `TagSpec{Type: "data", Value: ""}` → `<d>` via `mapTagType`.
- A documented zero-byte semantic anchored in `calculateCPSLength`.
- One protocol template (`quic.go`) that exercises the chaining behaviour.
- A regression net so that future contributors cannot silently perturb the contract.

This is **purely additive**: no existing callsite emits `<d>`, no existing test asserts its absence, no existing parser surface is affected. The risk surface is small.

## 2. Goal

Introduce `<d>` end-to-end through the generator with the **zero-byte passthrough** semantic locked in:

- Add the `"d"` case to `BuildCPSTag` (no value accepted; passthrough is value-less by spec).
- Add the `"data" → "d"` row to `mapTagType` so template authors use the human-readable form.
- Document in `calculateCPSLength`'s docstring that `<d>` contributes 0 bytes — and prove it via test, since the function already returns 0 for unknown literals (the test pins the **intended** zero, not an accidental one).
- Update `quic.go` to chain `<d>` from I1's random DCID into I2's slot, exercising the runtime-passthrough behaviour.
- Confirm `ValidatePacketSizes` (`validation.go:76`) is unaffected — `<d>` is invisible to the size-collision invariant by design (zero-byte intervals do not contribute to padded-size collisions; collisions can still occur from other tags in the same interval).
- Update user-facing docs (`docs/obfuscation.md` tag table, `AGENTS.md` tag list) to advertise the new tag and its kernel-module caveat.

**Out of scope**:

- Adding `<ds>` and `<dz N>` companion tags — deferred to P2.2.
- Generator awareness of "scope" (which earlier interval `<d>` borrows from) — implicit by position; the generator does not need state.
- Preset overrides for templates that mix-and-match `<d>` placement — deferred to P1.5.
- Kernel-module fallback ("emit `<d>` only if userspace") — deferred to a future item; for now `<d>` is always emitted and the docs warn users that the legacy Linux kernel module will reject it.
- Parser support for `<d>` in arbitrary user-supplied configs — `parser.go` does **not** parse CPS strings today (`I1`-`I5` parsing is commented out, verified during P0.1 inventory). The roadmap's "round-trip" criterion is therefore reframed in § 8 as **tag emission + length-accounting agree** (`BuildCPSTag` → `calculateCPSLength`); a true round-trip would require a `<d>`-aware parser (`parseCPSString`/`tagsToCPS`) and is out of scope for this PR.
- Multi-protocol `<d>` chaining (using `<d>` across DNS, DTLS, STUN templates) — deferred to P1.2 alongside any new templates.

## 4. Brainstorm Resolutions (from roadmap)

The roadmap posed four open questions. Resolved here so the implementer does not re-litigate them at code-review time.

1. **Does the generator need to track `<d>` "scope"?** **No.** `device/obf.go`'s state machine resolves passthrough by position implicitly — the receiver knows which prior interval to borrow from based on its own packet counter, not from any identifier in the tag. The generator emits the literal `<d>` and stops. **Confirmation step**: read `amneziawg-go/device/obf.go`'s `<d>` case before final commit; if the source shows the tag carries scope, halt and revise this plan.
2. **Which templates benefit most?** **QUIC** is the canonical case (DCID reuse across initial + handshake packets is real-protocol behaviour). DNS uses transaction-ID reuse but the existing template already handles it via fixed bytes; introducing `<d>` would change the on-the-wire shape unnecessarily. STUN/NTP do **not** benefit (typed-and-short messages, no natural cross-packet reuse). **Decision**: implement chaining in `quic.go` only; defer the rest to P1.2.
3. **MTU accounting**: track minimum (zero) or precompute worst-case upper bound? **Track zero.** Worst-case accounting would require tracking the source-interval value's byte length, propagating it through `calculateCPSLength`, and possibly across template intervals — a significant complication for marginal MTU safety gains. The user's MTU is already a soft constraint (`buildAndValidateCPS` shrinks tags until they fit at `cps.go:139-145`); a runtime-expanded `<d>` that exceeds budget will be discovered and handled by `amneziawg-go` at packet construction. Document the tradeoff in the docstring; add it to the "limitations" section of `obfuscation.md` if room permits.
4. **Kernel-module support**: feature gate or always emit? **Always emit, document caveat.** Feature gating requires `amnezigo` to know its deployment target, which is information it does not have (the same generator output is consumed by Linux kernel module, Linux userspace, mobile clients, desktop GUIs). Kernel-module-only deployments are a shrinking minority per the P0.1 mobile-compat decision. The CHANGELOG / docs explicitly warn kernel-only users to disable `<d>` in templates if they hit "unknown tag" errors; the workaround is to not pass `--protocol quic` (revert to `random` mode, which never emits `<d>`).

```go
// The tag carries no value and contributes 0 bytes at generation time; the
// AmneziaWG userspace receiver expands it at packet-emission time by reusing
// a value from an earlier I-packet position. Source of truth:
// amneziawg-go device/obf.go.
func TestBuildCPSTag_Data(t *testing.T) {

    expected := "<d>"
```

```go
// TestValidatePacketSizes_DTagDoesNotMaskCollisions guards against a future
// regression where <d>'s zero-byte semantic is mistakenly used to "fix" a
// collision by zeroing the affected interval. ValidatePacketSizes operates on
// pre-computed byte sizes (post-calculateCPSLength); a <d>-only interval has
// size 0, which trivially does not collide with any S-padded handshake size
// (>= wgTransportSize=32). This test pins that <d>'s zero-ness does not
// silence real collisions in OTHER intervals of the same config.
func TestValidatePacketSizes_DTagDoesNotMaskCollisions(t *testing.T) {
    // Construct a config where I3 collides with S1+148 — but I2 is "<d>"
    // (size 0). The validator must still flag the I3 collision, not skip it.
    s1, s2, s3, s4 := 32, 64, 128, 200
    padded := s1 + wgInitiationSize // 180
    // I1=10, I2=0 (the <d>-only interval), I3=collision, I4=20, I5=30.
    iPacketSizes := []int{10, 0, padded, 20, 30}
    err := ValidatePacketSizes(s1, s2, s3, s4, iPacketSizes, 1, 0) // empty junk
    if err == nil {
        t.Fatal("expected I3 collision, got nil")
    }
    var collisionErr *PacketSizeCollisionError
    if !errors.As(err, &collisionErr) {
        t.Fatalf("expected *PacketSizeCollisionError, got %T: %v", err, err)
    }
    if collisionErr.Kind != "i-packet" {
        t.Errorf("got Kind=%q, want %q", collisionErr.Kind, "i-packet")
    }
    if collisionErr.Size != padded {
        t.Errorf("got Size=%d, want %d", collisionErr.Size, padded)
    }
}
```

## 7. Risks and Edge Cases

| Risk | Assessment |
| --- | --- |
| `<d>` actually carries scope in upstream `device/obf.go` | **Mitigation**: pre-flight § 5 step 5 mandates re-reading `obf.go`. If true, halt and revise the plan to add a scope parameter to `BuildCPSTag("d", scope)`. **Likelihood**: low — roadmap initial reading says implicit-by-position, and that matches the `<d>` literal having no payload. |
| Kernel-module deployments break after this PR | **Documented, accepted.** § 5d adds a doc warning. The shrinking kernel-only user base has the workaround `--protocol random`. Mobile-compat decision (P0.1 rationale) already prioritised userspace. |
| MTU accounting under runtime size growth | `<d>` contributes 0 bytes at generation; runtime expansion can grow the on-the-wire packet. **In practice**: the source value's size (e.g., I1's `<r 8>` = 8 bytes) becomes the runtime-added bytes for I2's `<d>`. I2's total grows by ≤ source size. **Worst case for QUIC template**: I2 was ~32 bytes pre-PR (per the comment at `quic.go:38` "Length approx 32"); post-PR it's I2_base − 8 (`<random 8>` removed) + 8 (`<d>` runtime) = same 32. No regression. **Watch for**: if a future template chains a large source (e.g., `<r 40>`) into multiple `<d>` callsites, MTU could be exceeded at runtime. Document in the QUIC comment that `<d>` runtime-cost equals source-tag cost. |
| Parser strictness for stray values like `<d foo>` | **Not a parser concern today.** `parser.go` does not parse CPS strings (commented out, verified P0.1 inventory). `BuildCPSTag("d", "foo")` ignores the value and returns `<d>` (matches the `case "t"` precedent — `BuildCPSTag("t", "anything")` also returns `<t>`). Test 1a pins this. **If** parser support lands later (P1.5), strictness can be added there; for now, no parser change. |
| Interaction with `ValidatePacketSizes` | **Tested.** Step 1e adds `TestValidatePacketSizes_DTagDoesNotMaskCollisions`. The validator operates on pre-computed sizes; `<d>` adding 0 to its interval simply yields a smaller `iPacketSizes` entry, which still gets checked against all four padded sizes. Zero never collides with `≥ wgTransportSize=32`. |
| Template change in `quic.go` causes I2 to collide with a padded handshake size | **Possible but unlikely.** Step 4 explicitly re-runs `TestGenerateCPSConfig_Protocol` after the template change. If it fails, the QUIC I2 is adjusted with a small `<r N>` to escape collision. Note in PR body. |
| Random-mode silently picks `<d>` | **Mitigated by 2e.** `allTagTypes` is unchanged; `<d>` is intentionally excluded. Code comment documents the rationale to prevent a future "completionist" PR from adding it. |
| `<d>` in linter unused-warning radar | **No risk.** All paths reference the literal `"<d>"` from production code; the test references it via `BuildCPSTag("d", "")`. No unused symbol. |
| `tagTerminate` constant pattern not mirrored for `<d>` | **Accepted asymmetry.** `tagTerminate = "<t>"` exists because `<t>` has multiple callsites in the perturbation fallback (`cps.go:151`, `cps.go:317`). `<d>` has one callsite (`BuildCPSTag`); a constant would be over-engineering. Note for future cleanup if `<d>` proliferates. |
| Go-userspace version older than the one supporting `<d>` | **User responsibility.** `<d>` landed in AWG 2.0 reference; older `amneziawg-go` versions reject it. Doc § 5d warns; no in-code feature gate (we don't know the target version). |
| Markdown table in `obfuscation.md` rendering broken | **Visual-only.** Verified by adding the row with the same separator format (`| --- |`). If GitHub rendering breaks, pre-PR-review preview catches it. |
