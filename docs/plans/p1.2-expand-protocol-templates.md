## 1. Context

`amnezigo` is a Go CLI that generates AmneziaWG configuration files. Each peer's `[Interface]` section embeds five **CPS intervals** (`I1`–`I5`) — synthetic packet payloads constructed from CPS tags (`<b 0xHEX>`, `<r N>`, `<rc N>`, `<rd N>`, `<t>`). At runtime the AmneziaWG userspace receiver expands those tags into bytes and prepends them to the first five I-packets of a session.

The five intervals are protocol-shaped: the receiver places them on the wire to look like a known UDP protocol's opening handshake. Today `amnezigo` ships **four templates**: QUIC Initial, DNS Query, DTLS 1.2 ClientHello, STUN Binding Request. They live as one-file-per-protocol Go modules at the package root (`quic.go`, `dns.go`, `dtls.go`, `stun.go`) and route through `protocols.go:getTemplate(string) I1I5Template`.

**Why diversity matters.** DPI vendors enumerate. Once a "shape" is enumerable — e.g., "AmneziaWG with QUIC mimicry always emits a long-header byte at offset 0" — the shape becomes a positive fingerprint, and AmneziaWG goes from invisible to clearly identified. Adding more shapes (SIP, NTP, WebRTC TURN, MQTT-over-UDP, WireGuard-handshake) has compounding payoff:

- **Set size growth**: 4 → 9 templates almost triples the per-config decision tree the DPI must build.
- **Shape variance**: SIP's ASCII-heavy CRLF lines and NTP's tiny 48-byte fixed-format frames look nothing like QUIC/DNS/DTLS/STUN, so a vendor cannot reuse one parser to detect another.
- **Per-deployment selection**: with 9 protocols, an operator can pick the one that blends with their environment (corporate LANs prefer SIP/NTP; home users prefer QUIC/DNS).

**Why a framework, not a 5-template megapatch.** Each new template is a self-contained piece of work with its own RFC research, byte-budget arithmetic, and risk profile. Bundling all five into one PR would (a) explode review surface to ~1000+ LOC, (b) couple unrelated risks (an MTU bug in NTP could block SIP), and (c) prevent incremental merging. The plan therefore defines the **contribution framework** once, then ships **one PR per template** following that framework. This PR is the framework + the first template (SIP).

Observations:

- **No template uses `<c>`** (verified post-P0.1). Random-mode generator already excludes `<c>` (`cps.go:178`).
- **`I5` is always empty** in named templates. Random mode (`cps.go:246` `generateSimpleI`) populates `I5` with a generated CPS. The convention "`I5` empty for named templates" is documented in `docs/obfuscation.md:99` and is part of the framework contract.
- **MTU headroom**: with default MTU 1420 and S1=32, `maxISize = 1420 − 49 − 149 − 32 = 1190` (`docs/obfuscation.md:217`). Even DTLS I1 (~88 B) sits at 7% of budget. New templates can comfortably target ≤200 B per interval and still fit on MTU 1280 with S1=64 (`maxISize = 1018`).
- **Distinguishability**: the four existing templates have non-overlapping leading byte patterns (`c0` for QUIC, `??` random txn-id for DNS, `16` for DTLS, `00 01` for STUN). Any new template should pick a leading-byte signature that does not collide with any existing template *or* with a vanilla WireGuard message type-id (`01..04`). SIP's `OPTIONS ` (0x4F 0x50…) is safely ASCII; NTP's leading `LI|VN|Mode` byte is `0x1B` (NTPv3 client), distinct from all four.

`protocols.go` is the entire dispatcher; there is no YAML/JSON loader, no plugin system, no template registry. The roadmap brainstorm question "data-driven instead of hardcoded Go?" is therefore **explicitly deferred** (§ 11 Out of Scope).

## 4. Reality Check Decisions Driving the Framework

Before defining the contract, three reality-driven calls anchor the design:

- **Hardcoded Go, not data-driven** — every template is a `.go` file at the package root, exporting one `func XxxTemplate() I1I5Template`. Rationale: the CPS tag set is small and well-typed; YAML adds parsing, validation, schema-evolution, error reporting, and runtime cost for zero current benefit. Defer until ≥3 community-contributed templates exist outside core.
- **`I5` stays empty for named templates** — preserves the existing convention; random-mode behavior is the only path that fills `I5`. New templates set `I5: []TagSpec{}` (or omit, since the zero value is nil — but the convention is the explicit empty slice for readability; see `quic.go:73`).
- **No new tag types** — templates use only the five existing CPS tags (`bytes`, `random`, `random_chars`, `random_digits`, `timestamp`). Authors who feel they need a new tag should propose it in P1.1's scope, not bundle it into a template PR.

## 5. Template Contribution Framework

This is the **contract** every protocol template must satisfy. The framework PR codifies this section into `AGENTS.md` (see § 7 Step 5). All subsequent template PRs reference this contract and tick its checklist in their PR description.

### 5.0 Pre-implementation RFC verification (MANDATORY gate)

Before writing **any** template that mimics a real protocol, the contributor MUST verify the wire-format byte signatures (magic prefixes, method codes, version bytes, message-type bytes) against the **current** RFC via web search. Cached or training-data knowledge of "the leading byte of NTPv3 client mode is `0x1B`" is unacceptable for byte-level mimicry — wrong bytes mean wrong fingerprint and the template fails its purpose.

The verification produces two artefacts:

1. **A top-of-file comment** in `<protocol>.go` citing the RFC number, section, and the specific byte signature being mimicked. Example for SIP:

   ```go
   // SIPTemplate returns an I1I5Template mimicking a SIP OPTIONS request.
   //
   // Wire-format reference: RFC 3261 § 7.1 (Request-Line grammar) and § 27.4
   //   (registered method tokens — "OPTIONS" is one of the six core methods).
   //   Branch magic cookie "z9hG4bK" mandated by RFC 3261 § 8.1.1.7.
   //   Verified against RFC 3261 (current as of 2026-04-29).
   ```

2. **An entry in the PR description** stating which RFCs were checked, what byte signatures were verified, and the verification date. Reviewers reject PRs missing either artefact.

Per-protocol verification targets that follow-up template PRs MUST hit:

| Template | RFC(s) | Byte signatures to verify |
| --- | --- | --- |
| SIP (this PR) | RFC 3261 § 7.1, § 27.4, § 8.1.1.7 | `OPTIONS ` method literal, branch cookie `z9hG4bK`, header tokens |
| NTP | RFC 5905 § 7.3 (packet header format) | leading byte `LI|VN|Mode` — verify the exact bit layout for NTPv3 client (mode 3) and NTPv4 client (mode 3) before encoding `0x1B` or `0x23` |
| WebRTC TURN | RFC 5766 § 6.1 (Allocate request), RFC 5389 § 6 (STUN message header) | message-type `0x0003` (Allocate), magic cookie `0x2112A442` at offset 4, attribute TLVs |
| MQTT-over-UDP | MQTT-SN spec v1.2 § 5.4 | length byte + message-type byte (avoid PINGREQ which clashes with DTLS `0x16`) |
| WG-handshake | WireGuard whitepaper § 5.4 (handshake message format) | message-type byte (init=`0x01`), MAC1/MAC2 layout, ephemeral pubkey offset |

Outdated cached knowledge is a P0 issue for any template — verify before code, not after.

### 5.1 Required interface

Every template lives in **one source file** at the repository root, named `<protocol>.go` where `<protocol>` is the lowercase canonical short-name (matches the `--protocol` flag value). The file exports **exactly one** function:

```go
// XxxTemplate returns an I1I5Template mimicking <full protocol name>.
// <Brief RFC reference and packet structure summary>.
func XxxTemplate() I1I5Template { ... }
```

The function:

- Takes **no parameters**. (S-prefix awareness, MTU clipping, and collision avoidance happen downstream in `cps.go` — templates stay pure.)
- Returns an `I1I5Template` value (not pointer).
- Is **deterministic in shape** but the receiver fills `<r N>`, `<rc N>`, `<rd N>` randomly at emit time. The Go function itself returns the same `[]TagSpec` graph on every call.
- Has **no I/O, no globals, no logging, no panics**. Pure data construction.

### 5.3 Tag mix rules

Hard rules — violating any one is a PR blocker:

1. **No `<c>` tag.** The counter tag is gone post-P0.1. Authors who need pseudo-monotonic-looking bytes use `<rd N>` (digits) or `<r N>` (binary) and document the choice in a comment. Per-template enforcement scans the **rendered CPS string** (output of `BuildCPS` on each interval) for the literal substring `<c>`, not the `Type` field. Rationale: `mapTagType` returns `""` for unknown types, so a `Type:"counter"` field never reaches `<c>` output today; the test guards against future regressions where someone might reintroduce a `c` mapping or hand-code a `<c>` literal anywhere downstream.
2. **`<t>` size is 4 bytes** (post-P0.2). If you cite "8-byte timestamp" in a comment, you are reading stale docs — fix the doc, not the code.
3. **`<rc>` is `[a-zA-Z]` only** (post-P0.5). Use it for ASCII-letter-shaped fields (DNS labels, SIP header values). Do not assume it produces digits or symbols. For mixed letter+digit fields, concatenate: `<rc 4><rd 2>`.
4. **At most one `<t>` per interval.** The receiver's timestamp is monotonic-ish; emitting two `<t>` in the same interval looks artificial. (Random-mode generator enforces this implicitly via tag-pool composition; named templates must self-enforce.)
5. **No new tag types.** If you feel a need for `<rc-with-digits>` or `<base64>`, that is a P1.1 conversation, not a template PR.

Soft rules — strongly recommended but not auto-enforced:

- **I1 ≥ I2 ≥ I3 ≥ I4 in byte budget.** Matches existing convention (`docs/obfuscation.md:161-163`). STUN currently has I2 > I1 (`docs/obfuscation.md:165`); that's a known exception, not a license to invert at will.
- **I5 is empty `[]TagSpec{}` literally.** Explicit zero rather than implicit nil for grep-ability.
- **At least one `<r N>` or `<rc N>` per interval.** Deterministic-looking intervals (only `<bytes>`) are fingerprintable across two captures.

### 5.4 Byte budget — distinguishability and MTU

**Lower bound (distinguishability)**: each interval should be ≥ 16 bytes after expansion. Smaller intervals collide too easily with vanilla WireGuard message constants (148, 92, 64, 32). The pairwise size-collision validator (P0.3 `ValidatePacketSizes`) catches such collisions at generation time and forces I-packet retry, but staying clear by design avoids retry pressure.

**Upper bound (MTU)**: each interval must satisfy `calculateCPSLength(I_n) < maxISize` where `maxISize = MTU − 49 − 149 − S1` (`docs/obfuscation.md:215-217`). For target environment **MTU 1280 + S1 = 64** (worst-case mobile tunnel), `maxISize = 1018 B`. Every template's I1 should fit this with ~30% headroom (so ≤ 700 B per interval is a safe ceiling). Templates exceeding this trigger the tag-trimming fallback (`cps.go:103-122`) and lose protocol fidelity at runtime.

**Distinguishability heuristic** (informal, no test enforces it but reviewers should sanity-check):

- Leading 1–4 bytes should not match the leading bytes of an existing template. QUIC starts `c0`; DNS starts with random txn-id (no fixed prefix); DTLS starts `16`; STUN starts `00 01`. New templates must avoid all four prefixes if they themselves have a fixed prefix. Random-prefix templates (DNS-style) have no constraint.
- Ideally the protocol mimicked has a public RFC and is plausibly seen on the operator's network.

### 5.5 Test surface — required tests per template

Every new template ships with a `<protocol>_test.go` file containing **at minimum** these tests. Test names are factual (no internal references, no ticket IDs — per `code-quality.md`).

| Test name | What it proves |
| --- | --- |
| `TestXxxTemplate_AllIntervalsNonEmpty_I1ToI4` | I1, I2, I3, I4 each have `len(template.I1) > 0` etc. (I5 is allowed empty.) |
| `TestXxxTemplate_I5Empty` | `len(template.I5) == 0` — locks the convention. |
| `TestXxxTemplate_NoForbiddenTags` | Iterates every `TagSpec` in I1–I5; asserts `mapTagType(spec.Type)` is one of `"b"`, `"r"`, `"rc"`, `"rd"`, `"t"`. Catches typos and post-P0.1 `<c>` regression. |
| `TestXxxTemplate_FitsMTU` | For each interval, build the CPS via `buildAndValidateCPS` (or just `buildCPSFromTemplate` + `calculateCPSLength` — depends on what is package-private at framework time) and assert byte length `< maxISize` for `(MTU=1280, S1=64)` worst case. |
| `TestXxxTemplate_ByteBudgetUnderCeiling` | Each interval ≤ 700 B (informal ceiling, § 5.4). Soft assertion documenting the choice; raise the limit only with reviewer agreement. |
| `TestXxxTemplate_AtMostOneTimestampPerInterval` | Counts `Type == "timestamp"` per interval; asserts count ≤ 1. |
| `TestXxxTemplate_NoCounterLiteral` | Builds the rendered CPS via `BuildCPS(interval)` for each of I1–I5 and asserts the result does NOT contain the substring `<c>`. Scans the rendered string (not the `Type` field) because `mapTagType` already silently drops `Type:"counter"` to `""` — a field-level check guards a non-existent attack surface. The string-level scan also catches future regressions where someone hand-codes `BuildCPSTag("c", ...)` or sneaks a literal `<c>` in via `bytes` value. |
| `TestXxxTemplate_AvoidsExistingPrefixes` (optional but recommended) | If the template emits a fixed leading byte sequence, assert it differs from every prefix in the centralized `existingTemplatePrefixes` slice (see § 5.5.1). Skip if leading bytes are random. |

In addition, the **shared** test `protocols_test.go:TestGetTemplate_NamedProtocols` gets one extra row per new template (`{"<protocol>", false}`). This is part of the framework contract: every template appears in this single switch-coverage test.

Optional but recommended **end-to-end** test in `cps_mtu_test.go` (or its successor): run `generateProtocolCPS("xxx", 1280, 64)` and assert `ValidatePacketSizes(...)` returns nil. This is the integration-level guard for P0.3 invariant; templates that produce wildly-collision-prone byte counts will fail it.

### 5.5.1 Shared test helpers (centralized prefix registry)

Per-template `AvoidsExistingPrefixes` tests must NOT inline the list of existing prefixes — that couples the test to a snapshot of the codebase and silently breaks when later templates add or change prefixes. Instead, every template's prefix test calls a shared helper that consults a single centralized slice.

The framework PR adds the following to `protocols_test.go` (since the file already exists at the repo root and shares `package amnezigo`, no new file is needed):

```go
// existingTemplatePrefixes is the canonical list of leading-byte prefixes
// already in use by shipped templates. New templates MUST NOT begin with any
// of these so that --protocol random produces distinguishable shapes.
//
// MAINTENANCE CONTRACT: every template PR that introduces a new fixed leading
// byte sequence appends it here in the same diff. Reviewers reject template
// PRs that add a new prefix without updating this slice.
var existingTemplatePrefixes = [][]byte{
    {0xC0},                 // QUIC long-header byte 0
    {0x16},                 // DTLS handshake content type
    {0x00, 0x01},           // STUN binding request message type
    []byte("OPTIONS "),     // SIP OPTIONS method literal (added in this PR)
    // future templates append here
}

// assertTemplateAvoidsExistingPrefixes builds I1's CPS via BuildCPS and asserts
// the rendered byte stream does not start with any prefix in
// existingTemplatePrefixes. Skips if the template's I1 leading bytes are
// random (no fixed prefix to check).
func assertTemplateAvoidsExistingPrefixes(t *testing.T, tmpl I1I5Template, ownPrefix []byte) {
    t.Helper()
    cps := BuildCPS(tmpl.I1)
    for _, prefix := range existingTemplatePrefixes {
        if bytes.Equal(prefix, ownPrefix) {
            continue // skip the template's own prefix
        }
        if bytes.HasPrefix([]byte(cps), prefix) {
            t.Errorf("template I1 begins with already-used prefix %x", prefix)
        }
    }
}
```

Each per-template `TestXxxTemplate_AvoidsExistingPrefixes` then becomes a one-liner that calls `assertTemplateAvoidsExistingPrefixes(t, XxxTemplate(), <ownPrefix>)`. This centralizes the framework guarantee: when a future template (e.g., NTP) adds a leading `0x1B` to the slice, every prior template's test automatically gains coverage against the new collision surface — no risk that SIP's test silently passes while the framework guarantee breaks.

## 7. Step-by-Step Implementation (TDD) — SIP as the Worked Example

This walkthrough implements **SIP OPTIONS** as the framework PR's reference template. The choice of SIP is justified in § 8.1.

### Step 0 — Decide the SIP shape

SIP runs over UDP/5060 (or 5061 for SIPS). The simplest mimicry-friendly request is `OPTIONS` — a no-op ping commonly used by VoIP gateways and rarely blocked by enterprise firewalls. Full structure of an OPTIONS request:

```text
OPTIONS sip:user@domain.example SIP/2.0\r\n
Via: SIP/2.0/UDP <local-ip>:5060;branch=z9hG4bK<random-token>\r\n
From: "Caller" <sip:alice@example.com>;tag=<random-tag>\r\n
To: <sip:user@domain.example>\r\n
Call-ID: <random-token>@<local-host>\r\n
CSeq: 1 OPTIONS\r\n
Max-Forwards: 70\r\n
User-Agent: <random-token>\r\n
Content-Length: 0\r\n
\r\n
```

ASCII-only, line-delimited, ~250–400 bytes for a realistic OPTIONS, ~150 bytes for a minimal one. Fits MTU-1280-with-S1=64 (1018 B budget) with massive headroom.

Mapping to CPS tags:

- Fixed text (`OPTIONS sip:`, `SIP/2.0\r\n`, header keywords) → `<b 0xHEX>` of the ASCII bytes.
- Variable tokens (branch values, tags, Call-IDs, User-Agent strings) → `<rc N>` for letter tokens, `<rd N>` for digit-only tokens, `<r N>` only when binary noise is needed (almost never in SIP).
- No timestamp on the wire — SIP doesn't have one in headers. Skip `<t>` entirely. (This is a deliberate distinguishability move: every other template uses `<t>`; SIP not using it makes its byte-length distribution different.)

### Step 0.1 — Verify SIP byte signatures against RFC 3261 (mandatory gate)

Per § 5.0, before writing any code: web-search and read the current RFC 3261 to confirm:

- **Method token** `OPTIONS` is one of the six core methods (§ 27.4) — bytes `0x4F 0x50 0x54 0x49 0x4F 0x4E 0x53`.
- **Request-Line grammar** (§ 7.1) is `Method SP Request-URI SP SIP-Version CRLF`. Trailing `\r\n` is `0x0D 0x0A`.
- **Branch parameter magic cookie** `z9hG4bK` is mandated by § 8.1.1.7 — bytes `0x7A 0x39 0x68 0x47 0x34 0x62 0x4B`.
- **Header field syntax** for `Via`, `From`, `To`, `Call-ID`, `CSeq`, `Max-Forwards`, `User-Agent`, `Content-Length` matches §§ 20.42, 20.20, 20.39, 20.8, 20.16, 20.22, 20.41, 20.14 respectively.

Record the verification date in the top-of-file comment of `sip.go` (see § 5.0 for the exact format). Subsequent template PRs (NTP, WebRTC TURN, MQTT, WG-handshake) repeat this step against their respective RFCs from the § 5.0 table.

// method tokens — OPTIONS is one of the six core methods), § 8.1.1.7 (branch
// parameter magic cookie "z9hG4bK"), §§ 20.8, 20.14, 20.16, 20.20, 20.22, 20.39,
// 20.41, 20.42 (header field grammars used below). Verified against the current
// RFC 3261 text on the Step 0.1 verification date recorded in PR description.
//
// SIP runs over UDP/5060. OPTIONS is a no-op ping that VoIP gateways send routinely
// and that enterprise firewalls almost always permit. The template constructs an
// ASCII, line-delimited request:

//     Via: SIP/2.0/UDP <random branch>\r\n

### Adding a New Protocol Template (P1.2 framework)

Every new protocol template MUST satisfy this contract. Reviewers reject PRs that miss any item.

**Required interface**

- File `<protocol>.go` at the repository root.
- Constructor `XxxTemplate() I1I5Template` — pure data construction, no I/O, no globals.
- Test file `<protocol>_test.go` co-located.
- Switch case key, `--protocol` flag value, and doc table row all match the file's lowercase short-name.

**Tag mix rules**

- No `<c>` tag (removed in P0.1). For pseudo-monotonic bytes use `<rd N>` or `<r N>`.
- `<t>` is 4 bytes (post-P0.2).
- `<rc>` is `[a-zA-Z]` only (post-P0.5). For mixed letter+digit fields, concatenate `<rc 4><rd 2>`.
- At most one `<t>` per interval.
- No new tag types — propose those in P1.1, not in a template PR.

**Byte budget**

- Each interval ≥ 16 B (avoid raw-WG size collisions).
- Each interval ≤ `MTU − 49 − 149 − S1` (worst case MTU=1280, S1=64 → 1018 B).
- Recommended ceiling ≤ 700 B per interval; raise only with reviewer agreement.
- I5 always empty (`[]TagSpec{}` literal) for named templates.
- I1 ≥ I2 ≥ I3 ≥ I4 in byte budget (STUN's I2 > I1 is a known exception).
- Leading bytes should not collide with any prefix in the centralized `existingTemplatePrefixes` slice in `protocols_test.go`. New templates that introduce a fixed prefix MUST append it to that slice in the same PR — otherwise existing per-template prefix tests silently miss the new collision surface.

**Required tests**

- `TestXxxTemplate_AllIntervalsNonEmpty_I1ToI4`
- `TestXxxTemplate_I5Empty`
- `TestXxxTemplate_NoForbiddenTags`
- `TestXxxTemplate_NoCounterLiteral` — scans the rendered CPS string for `<c>` substring (not the `Type` field; see § 5.3)
- `TestXxxTemplate_FitsMTU`
- `TestXxxTemplate_ByteBudgetUnderCeiling`

Expected: `I1`–`I4` are non-empty CPS strings starting with `<b 0x4f5054494f4e53>` (the `OPTIONS` literal), `I5` is empty. No `<c>` anywhere.

## 8. Candidate Analysis Table

The following five candidates appear in the roadmap. Only **SIP** ships in the framework PR; the other four are deferred to follow-up PRs in the recommended order.

### 8.1 Per-candidate analysis

| Candidate | MTU feasibility | Distinguishability | Tag mix | `<d>` opportunity | Risks | Recommended order |
| --- | --- | --- | --- | --- | --- | --- |
| **SIP** (OPTIONS) | Excellent. Full request ≈ 360 B; minimal ≈ 120 B. Fits MTU 1280 + S1=64 (1018 B) with massive headroom. | Excellent. ASCII method literal `OPTIONS `, distinct from QUIC `c0`, DTLS `16`, STUN `00 01`. No timestamp use makes byte distribution different. | `bytes`, `rc` (letters), `rd` (digits) | None now (no peer-identity field naturally embeddable). With P1.1, Call-ID could embed peer pubkey via `<d>`. | None significant. RFC 3261 is well-known; OPTIONS is a routine no-op. | **1 (this PR)** |
| **NTP** | Tight. NTP packet is **48 B fixed** (RFC 5905). Five intervals × 48 B = 240 B total; per-interval byte ceiling drops from 700 B to 50 B. Distinguishability against raw WG transport size 32 is fine (48 ≠ 32). | Good. Leading byte `0x1B` (LI=0, VN=3, Mode=3=client) or `0x23` (VN=4). No collision with existing prefixes. | `bytes` (mostly), `random` (txn/origin timestamps) | High — NTP has reference identifier (4 B) and stratum field where peer pubkey shards could land via `<d>`. | All four intervals at the same 48 B size make the I-packet sizes a tight cluster — collision-avoidance retry pressure is higher. P0.3 validator handles it but be alert in property tests. Test ceiling must be relaxed (50 B not 700 B). | **3** — second to ship after WebRTC because the byte-budget tight-cluster needs fresh attention. |
| **WebRTC TURN-Allocate** | Good. ~80–120 B per interval. | **Watch out — magic-cookie collision.** WebRTC TURN reuses STUN's magic cookie `0x2112A442` at offset 4. Naive implementation looks like STUN to a DPI vendor. Differentiate via the **TURN-specific message type** (`0x0003` Allocate, not STUN's `0x0001` Binding) AND TURN-specific attributes (`REQUESTED-TRANSPORT`, `LIFETIME`). | `bytes`, `random` | Some — TURN attributes have variable-length value fields suitable for `<d>` shards. | Need explicit reviewer attention to the prefix collision: § 5.4 distinguishability heuristic must reference both STUN's `00 01` AND require TURN's `00 03`. Otherwise the random-mode selector silently produces two near-identical shapes. | **2** — ship right after SIP since it shares STUN's framework patterns and the prefix-collision care is the only novel work. |
| **MQTT-over-UDP** | Good. MQTT-SN (the UDP variant) packets are ~14–60 B. Five intervals fit easily. | Moderate. MQTT-SN starts with a length byte then message-type byte. Need to pick types unlikely to collide (CONNECT=0x04, PUBLISH=0x0C, PINGREQ=0x16). PINGREQ leading bytes `02 16` are dangerously close to DTLS leading `16`. Pick CONNECT or PUBLISH. | `bytes`, `random`, `rc` | Some — ClientId field is ASCII-safe and length-bounded; perfect for `<rc>` filling and potentially `<d>` fragments. | Niche — MQTT-SN over plain UDP is rare in the wild; review may push back on whether this template adds real DPI evasion or just noise. Counter-argument: rarity itself is a feature when the goal is "hard to enumerate". | **5** (last) — lowest mainstream presence; ship after the others have validated the framework. |
| **WireGuard-handshake** | Good. WG handshake init is 148 B; mimicking just the first 80–120 B is enough for shape similarity. | **Highest payoff, highest care.** Anti-canary against vendors who detect AmneziaWG by looking for "not-WireGuard" but accept WireGuard-itself as a baseline. | `bytes` (type-id 1 in CPS body — see § 9 below for why this is OK), `random` (ephemeral pubkey, MAC1, MAC2) | High — peer pubkey shards naturally fit the static-key field of the handshake. | **Critical clarification**: P0.4 forbids type-ids 1–4 in **H1–H4 ranges** (`generator.go:GenerateHeaderRanges`). That constraint applies to the four uint32 ranges that AWG uses to recognise its own message-types. **It does NOT apply to CPS body content** — CPS bytes are payload, not header type-ids. A WG-handshake template emitting `0x01 0x00 0x00 0x00` as its first four bytes is **legal**: those bytes are part of the I-packet payload, not part of an H-range. The framework's "no `<c>`" rule is the only tag-level body rule. | **4** — ship after NTP. The H-range vs CPS-body clarification needs to be in `docs/obfuscation.md` before this template lands, otherwise contributors and reviewers will incorrectly flag the leading `0x01`. |

### 8.2 Why SIP first

SIP is the lowest-risk worked example for the framework PR:

- **No prefix-collision arithmetic**: SIP starts with the ASCII `OPTIONS ` literal — distinguishable at first glance from every existing template prefix.
- **No MTU squeeze**: full OPTIONS comfortably fits MTU 1280 with double-digit % headroom.
- **No clarification debt**: doesn't bump into P0.4's H-range rule, doesn't share STUN's magic cookie, doesn't have the 48-B fixed-size collision-cluster problem.
- **Excellent shape variance**: ASCII line-delimited bytes look completely unlike binary-framed QUIC/DNS/DTLS/STUN.
- **Real-world ubiquity**: enterprise networks pass SIP OPTIONS routinely; mimicking it in restrictive environments is plausible.
- **Pedagogically clear**: the byte-budget arithmetic, the `<rc>`/`<rd>` mix, the `<t>`-free choice, and the test surface all illustrate framework patterns that NTP/WebRTC/MQTT/WG-handshake will reuse.

NTP is a strong second candidate technically but a worse pedagogical first: the 48-B fixed size forces an immediate "soft rule violation" (test ceiling 700 B → 50 B) that distracts from the framework. WebRTC TURN is technically interesting but couples to STUN, requiring the reader to hold two templates in mind. WG-handshake requires the H-range vs CPS-body clarification doc note that hasn't shipped yet. MQTT is the lowest-priority candidate.

## 9. Risks and Edge Cases

| Risk | Assessment |
| --- | --- |
| New template's I-packet sizes collide with an S-padded handshake size | **Handled by P0.3.** `ValidatePacketSizes` runs as a post-condition in `GenerateConfig` (`generator.go`) and the I-packet builders perturb on collision. The framework adds no new collision surface — every template inherits the same protection. Property test `TestGenerateConfig_NoCollisionsProperty_AllProtocols` covers this if extended to include SIP. |
| Reviewer flags `<bytes 0x01...>` in WG-handshake template as P0.4 violation | **Documentation gap, not a code bug.** P0.4 forbids type-ids 1–4 in **H1–H4 ranges** — those are uint32 values the receiver consults to classify incoming packet headers. CPS body bytes are the synthetic payload of an I-packet, not a header. The two surfaces are different. Mitigation: add a short note to `docs/obfuscation.md` (in the WG-handshake follow-up PR) explicitly stating "header-range forbid ≠ CPS-body forbid". |
| WebRTC TURN template silently looks like STUN | **Distinguishability requirement.** § 5.4 mandates leading-byte distinction; the TURN PR must use the Allocate message type (`0x00 0x03`) at offset 0, not `0x00 0x01`. A test like `TestWebRTCTemplate_NotMistakenForSTUN` should assert the second byte is `0x03`. |
| NTP MTU squeeze degrades fidelity | **Acceptable.** NTP packets are 48 B fixed; relaxing the 700 B test ceiling to 50 B is documented in the NTP PR's rationale. Authentic NTP traffic IS 48 B; matching that size is the whole point. |
| Random-mode selection becomes biased | **Verify in property test.** With N templates, each should be picked with probability 1/N over many calls. Existing `TestGetTemplate_RandomIsNotDeterministic` only proves variety, not uniformity. **Optional follow-up**: add `TestGetTemplate_RandomIsUniform` (chi-squared over 10000 calls). Not required for the framework PR; nice-to-have. |
| Data-driven (YAML) templates discussion re-emerges | **Defer.** § 11 explicitly defers until ≥3 community-contributed templates exist outside core. |
| Helptext truncation | `--protocol`'s helptext is currently `"Obfuscation protocol"` (no enumeration). Switching to `"Obfuscation protocol: random, quic, dns, dtls, stun, sip"` is harmless on every supported terminal width (≤80 chars). For longer lists in the future, fall back to `"Obfuscation protocol (see docs/cli-reference.md)"`. |
| `AGENTS.md` framework block goes stale | **Acceptable risk.** `AGENTS.md` is the canonical source of conventions in this repo. Framework changes are rare and concentrated; if they happen, the block updates as part of the same PR that changes the framework. |
| Tests in `<protocol>_test.go` reference internal helpers (`buildCPSFromTemplate`, `calculateCPSLength`) that may move | **Acceptable.** Tests live in `package amnezigo` (verified: `protocols_test.go:1`), so they share the internal namespace. If a future refactor renames `buildCPSFromTemplate`, all `<protocol>_test.go` files break together — caught by CI on the refactor PR, fixed in the same diff. |

## 11. Out of Scope (Explicit)

The following are **deliberately excluded** from this plan and from the framework PR:

- **Implementing all five candidates in this PR.** Framework + SIP only. NTP, WebRTC, MQTT, WG-handshake each get their own PR per § 12.
- **Data-driven (YAML/JSON) templates.** Defer until ≥3 community-contributed templates exist outside core. The complexity cost (parser, validator, schema evolution, error reporting, test surface duplication) is not justified for the current 4–9 template scale.
- **Per-preset protocol overrides.** "User wants to ship a custom template per peer profile" is P1.5 territory (presets). The framework defines the contract; presets define the selection policy.
- **Protocol-specific MTU heuristics.** `calculateMaxISize` is template-agnostic. If a future template (e.g., NTP) wants to hard-cap at 48 B regardless of MTU, that lives in the template's own builder, not in core.
- **`--protocol` allow-list validation in the CLI.** P1.3 (`amnezigo validate`) territory. Today unknown values fall through to random — documented behavior, intentionally preserved.
- **Adding a new tag type** to support a template's needs. New tags are P1.1's scope; templates use only the existing five.
- **Refactoring `protocols.go` into a registry pattern** (`map[string]func() I1I5Template`). Not justified at 5–9 templates; revisit at 15+.
- **Renaming or relocating existing templates.** `quic.go`, `dns.go`, `dtls.go`, `stun.go` stay where they are.
- **Updating `docs/roadmap.md` to mark P1.2 done.** Roadmap updates happen in a sweep PR after all five template PRs land, not piecemeal.
- **Performance benchmarking.** Template construction is O(1) data allocation; benchmarks add noise without signal.

If any of these come up during implementation, **do not include them**. Note them for the next PR or an issue.
