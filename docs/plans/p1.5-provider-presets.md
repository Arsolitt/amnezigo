## 1. Context

`amnezigo` is a Go CLI that generates AmneziaWG configuration files. AmneziaWG's obfuscation parameters (S1-S4 padding, H1-H4 type-id ranges, Jc/Jmin/Jmax junk parameters) must be tuned to defeat the local DPI environment. Today these tuning ranges are hardcoded in `generator.go:10-23`:

```go
const (
    s4RangeMax         = 33
    sPrefixRangeMax    = 65
    jcRangeMax         = 11
    junkMinValue       = 64
    junkRangeSize      = 961 // 1024 - 64 + 1
    headerMinValue     = uint32(5)
    headerMaxValue     = uint32(2147483647)
    headerMinRange     = uint32(10000000)
    headerMaxAttempts  = 1000
    sMaxAttempts       = 1000
    junkMaxAttempts    = 1000
    iPacketMaxAttempts = 1000
)
```

These are reasonable defaults but not universal. Empirical reports from the field show that some networks (Iranian DPI, MTS Russia mobile, GFW) classify packets that fall inside the default ranges. A user in such a network has no way today to influence the generator's behavior short of patching the source.

**Why a preset system instead of more flags?**

- Flag explosion: exposing `--s1-min`, `--s1-max`, `--s2-min`, `--s2-max`, ..., `--jc-min`, `--jc-max`, ..., `--h-range-size-min`, `--h-range-size-max` would push `init` past 25 flags. Cognitive load and shell-history fragility increase non-linearly.
- Empirical knowledge is *bundled*: a ru-mts profile is not 8 independent ranges — it's a tested *combination*. Splitting them into flags forces every user to re-learn that combination.
- Versioning: as AWG evolves (new constants in `amneziawg-go`), preset files can be re-validated against the new version without rewriting CLI code.
- Community contribution: a YAML file in a `presets/` directory is a low-barrier PR target. Anyone with a working network can contribute a preset; flag-set changes require Go knowledge and full code review.

**Roadmap decision 2** explicitly defers the *contribution process documentation* (CONTRIBUTING.md, issue templates, maintainer review SOP) until first community PRs land. This plan ships only the *infrastructure*: schema, loader, CLI flag, embed, and one canonical example preset. The community process is designed reactively, when contributors show up.

## 2. Goal

Ship preset infrastructure + one example preset, end-to-end:

- **Schema**: a versioned YAML schema for preset files with parameter ranges (jc/jmin/jmax/s1/s2/s3/s4 + h_range_size), provider metadata (name, description, version, maintained_by, tested_on, protocols, notes).
- **Loader**: a new `internal/presets/` package that reads embedded YAML files at compile time via `go:embed` and validates them against the schema.
- **Generator refactor**: introduce a `GeneratorOptions` struct holding all overrideable ranges. The current package-level constants become struct defaults. All public generator entry points (`GenerateSPrefixes`, `GenerateSPrefixesWithS1`, `GenerateJunkParamsWithForbidden`, `GenerateHeaderRanges`, `GenerateConfig`, `GenerateServerConfig`) gain optional `WithOptions(GeneratorOptions)` variants. Existing zero-arg signatures continue to work using built-in defaults.
- **CLI wiring**: `amnezigo init --preset NAME` loads the named preset, builds `GeneratorOptions`, threads it through `GenerateServerConfig`. Without `--preset`, behavior is byte-identical to today.
- **Example preset**: `presets/example.yaml` covering all schema fields, with comments. `docs/presets.md` explains the schema and contribution flow.
- **CI integration**: a one-liner that iterates over `presets/*.yaml`, generates a config per preset, and runs `amnezigo validate` against each. Gated on P1.3.

**Out of scope for THIS plan** (deferred):

- `--preset-file path.yaml` runtime escape hatch — defer to a follow-up; embedded presets are sufficient for the v1 contract.
- Preset overrides for CPS templates (per-protocol I-packet template overrides) — that is **P3.5** territory.
- `update-presets` remote registry / preset versioning across binary releases — **P3.5**.
- Curated provider presets beyond `presets/example.yaml` (ru-mts, ir-irancell, etc.) — community contributions in PR 5+, not part of this plan.
- CONTRIBUTING.md document for preset submissions — roadmap decision 2: design when first PRs land.
- Snapshot testing of preset → generated config (locking an `expected_output.conf` per preset) — P3.5 territory; locks the implementation prematurely.
- `amnezigo presets list` / `amnezigo presets show NAME` subcommands — useful but not required for the v1 contract; track in P3.5 backlog.

## 4. Schema Design

This is the public contract for community contributions. Get it right.

### 4.1 Top-level YAML structure

```yaml
# Required fields.
name: string                # filename without .yaml; must match [a-z0-9-]+
description: string         # one-line summary, max 200 chars
version: integer            # schema version; v1 is the only supported value today
maintained_by: string       # GitHub username, free-form

# Optional metadata.
tested_on:                  # list of "tool: version" pairs; informational only
  - awg-go: "v0.2.17"
  - amneziawg-tools: "v1.0.20260223"

# Required: the parameter ranges.
params:
  jc:           { min: int, max: int }    # all eight pairs are required
  jmin:         { min: int, max: int }
  jmax:         { min: int, max: int }
  s1:           { min: int, max: int }
  s2:           { min: int, max: int }
  s3:           { min: int, max: int }
  s4:           { min: int, max: int }
  h_range_size: { min: int, max: int }    # span of each H range, NOT min/max headers

# Optional.
protocols: [string]         # subset of {random, quic, dns, dtls, stun}; informational
notes: string               # free-form, multi-line OK
```

### 4.2 Field-level validation rules

| Field | Type | Required | Constraints | Failure mode |
| --- | --- | --- | --- | --- |
| `name` | string | yes | `^[a-z0-9-]+$`, 1..32 chars; must equal filename basename | hard error at load time |
| `description` | string | yes | 1..200 chars | hard error |
| `version` | int | yes | currently must equal `1`; future schema versions bump this | hard error if unknown major |
| `maintained_by` | string | yes | 1..64 chars, free-form (typically GitHub username) | hard error if empty |
| `tested_on` | list | no | each entry is a single-key map; key `[a-z0-9-]+`, value any string | warning if absent; hard error on malformed |
| `params.jc.min` | int | yes | `0 <= min < max`, `min, max <= 128` | hard error |
| `params.jc.max` | int | yes | same as above | hard error |
| `params.jmin.{min,max}` | int | yes | `64 <= min, max <= 1024`; `min < max` (cap matches the historical default `junkMinValue + junkRangeSize - 1 = 64 + 961 - 1 = 1024` from `generator.go:14-15`; pushing higher risks MTU fragmentation) | hard error if floor/ceiling violated |
| `params.jmax.{min,max}` | int | yes | `64 <= min, max <= 1024`; `min < max` (same cap as `jmin`; cited from `generator.go:14-15`) | hard error |
| `params.s1.{min,max}` | int | yes | `0 <= min, max <= 256`; `min < max` | hard error |
| `params.s2.{min,max}` | int | yes | same | hard error |
| `params.s3.{min,max}` | int | yes | same | hard error |
| `params.s4.{min,max}` | int | yes | `0 <= min, max <= 64`; `min < max` (S4 caps at AWG transport-padding ceiling) | hard error |
| `params.h_range_size.{min,max}` | int | yes | `1 <= min < max <= 1_000_000_000` (4 ranges must fit in `[5..2^31-1]` window: `4 * max + 3 * gap < 2^31`; gap defaults to `headerMinRange=10_000_000`) | hard error if window exhausted |
| `protocols` | list | no | each entry in `{random, quic, dns, dtls, stun}` (set at PR 4 time; canonical list lives in `protocols.go`) | warning on unknown name; hard error on type mismatch |
| `notes` | string | no | no length cap; multi-line OK | n/a |

**Rationale for caps**: the upper bounds (`jc.max <= 128`, `s1.max <= 256`, `s4.max <= 64`, etc.) are deliberate guard rails. They prevent a misguided preset from generating absurdly large junk packets that would break MTU. They are **not** AWG protocol limits — those would be `2^31-1` everywhere — they are *sanity* limits to flag preset typos at load time. Document this explicitly so contributors don't try to push them.

### 4.3 Forward-compat: how older binaries handle newer schema fields

- **Unknown top-level fields** (e.g. a future `v2` adds `tags: [...]`): emit a warning to stderr at load time, ignore the field, proceed. This keeps newer presets *partially* usable on older binaries — a user pinning to an older `amnezigo` release won't be locked out of a preset that adds optional metadata.
- **Unknown `params.<name>.<sub>` keys** (e.g. future adds `params.jc.distribution: "uniform"`): same treatment — warn, ignore, proceed.
- **`version: 2` on a v1-only binary**: hard error. Major version bumps are reserved for *breaking* schema changes; an older binary cannot safely guess at the new semantics. Error message must point to the upgrade path: "preset requires schema v2; install amnezigo >= X.Y.Z or pick a v1 preset".
- **YAML library setting**: `yaml.v3` decoder defaults to *strict* mode if we set `dec.KnownFields(true)`. We intentionally do NOT set strict mode for top-level / params — we want the warn-and-proceed behavior. We DO set strict mode for the `params.*.{min,max}` inner objects, because typos like `mim`/`max` should fail fast.

### 4.4 Versioning policy

- `version: 1` is the only supported value at time of this plan.
- Bump to `version: 2` when removing a required field, renaming a field, or changing a field's units (e.g. switching `h_range_size` from absolute size to a percentage).
- Adding a new *optional* field does NOT require a version bump — older binaries warn-and-ignore.
- Adding a new *required* field is a v2 change.
- The `version` integer is NOT semver. It's a single monotonic counter; minor / patch evolution within v1 is unversioned and subject to the warn-on-unknown-field rule.

### 4.5 Default values

Built-in defaults (matching today's `generator.go` constants exactly):

```yaml
name: default                # synthetic; not on disk
description: Built-in default ranges (matches generator.go constants)
version: 1
maintained_by: amnezigo-core
params:
  jc:           { min: 0,  max: 11 }    # jcRangeMax = 11
  jmin:         { min: 64, max: 1024 }  # junkMinValue=64, junkRangeSize=961 → 64 + 961 - 1 = 1024
  jmax:         { min: 64, max: 1024 }  # same
  s1:           { min: 0,  max: 65 }    # sPrefixRangeMax = 65
  s2:           { min: 0,  max: 65 }    # same
  s3:           { min: 0,  max: 65 }    # same
  s4:           { min: 0,  max: 33 }    # s4RangeMax = 33
  h_range_size: { min: 10_000_000, max: 2_117_483_642 }  # headerMinRange = 10_000_000; max = headerMaxValue - headerMinValue - 3*headerMinRange = 2_147_483_647 - 5 - 30_000_000 = 2_117_483_642 (constants from generator.go:16-18)
protocols: [random, quic, dns, dtls, stun]
```

This default object lives in `internal/presets/defaults.go` as a Go struct literal — NOT as an embedded YAML file. Reasons:

- Avoids the loader bootstrapping its own dependency: defaults are needed before any YAML is parsed.
- Tests can assert defaults by value comparison without touching the filesystem.
- Compile-time guarantee that defaults parse correctly.

The default object IS exposed via a public function `presets.Default() *Preset` for the loader to fall back on when `--preset` is unset.

## 5. Architecture Decisions

### 5.1 YAML library: `gopkg.in/yaml.v3`

**Decision**: add `gopkg.in/yaml.v3 v3.0.1` to `go.mod` (PR 2).

**Alternatives considered**:

| Library | Pros | Cons |
| --- | --- | --- |
| `gopkg.in/yaml.v3` | Pure Go, well-maintained, ~250KB, widely-used standard. `yaml.v3` supports `Decode().KnownFields(true)` for strict-mode validation. Permissive license (MIT + Apache 2.0). | Single transitive dep. |
| `sigs.k8s.io/yaml` | JSON-compatible (good if we later want JSON presets), used by Kubernetes. | Wraps yaml.v2 + json under the hood; pulls more transitive deps; designed for JSON-equivalent YAML, which we don't need. |
| `github.com/goccy/go-yaml` | Faster, native source-line errors. | Younger project, smaller user base, slightly larger surface area. |
| Hand-rolled INI-like parser | Zero new deps. | Reinventing YAML is a maintenance trap; multi-line `notes` and nested `params.*.{min,max}` make it non-trivial. Loses error-quality gains. |

**Choice rationale**: `yaml.v3` is the lightest pure-Go option that gives us strict-mode validation for nested objects and warn-and-continue for top-level. Its single transitive dep (none beyond stdlib) is acceptable. Supply-chain audit: maintained by Canonical's go-yaml team, tagged 2022-05-27, no known CVEs as of plan-write time. Verify version pin in PR 2 and document in the commit body.

### 5.2 Application order: defaults → preset → CLI flags (CLI wins)

```text
       built-in defaults
              │
              ▼
       preset overlay (only fields that the preset explicitly sets)
              │
              ▼
       CLI flag overrides (any --flag the user supplies)
              │
              ▼
       final GeneratorOptions → generator
```

**Rationale**: this is the standard config-precedence pattern (e.g. cobra, viper, kubectl). User intent flows from least-specific (defaults) to most-specific (CLI). A preset is an opinion; a CLI flag is a directive.

**Concrete corollaries**:

- `--preset ru-mts --mtu 1420` → preset's params win for S/J/H, but CLI's `--mtu` overrides whatever the preset suggests for MTU. (Note: `mtu` is NOT in the preset schema today; presets influence obfuscation, not transport. If a contributor wants to suggest an MTU, they put it in `notes`.)
- `--preset ru-mts` with no CLI overrides → preset wins fully.
- No `--preset` at all → built-in defaults, byte-identical to today's behavior.

### 5.3 GeneratorOptions struct

Introduce in PR 1:

```go
// IntRange is a closed-open range [Min, Max). Min must be < Max.
// Validation is responsibility of the caller (preset loader runs schema checks
// at YAML-parse time; CLI assembly path uses defaults that are valid by
// construction).
type IntRange struct {
    Min, Max int
}

// GeneratorOptions controls every range the generator draws from.
//
// Each field is a pointer: a nil pointer means "use the default for this
// range"; a non-nil pointer means "the caller has explicitly overridden
// this range and the value MUST already be valid (Min < Max, both within
// the schema-imposed sanity bounds documented in §4.2)".
//
// Pointers — not zero-value sentinels — because a legitimate range like
// IntRange{Min: 0, Max: 65} (default S1) cannot be distinguished from
// "unset" with a value-typed sentinel. The preset loader rejects malformed
// ranges at YAML-parse time, so by the time a *IntRange reaches the
// generator it is guaranteed valid.
type GeneratorOptions struct {
    Jc         *IntRange
    Jmin       *IntRange
    Jmax       *IntRange
    S1         *IntRange
    S2         *IntRange
    S3         *IntRange
    S4         *IntRange
    HRangeSize *IntRange
}

// DefaultGeneratorOptions returns the historical hardcoded ranges as
// non-nil pointer fields. Useful as a base for callers that want to override
// only specific ranges.
func DefaultGeneratorOptions() GeneratorOptions {
    return GeneratorOptions{
        Jc:         &IntRange{Min: 0, Max: jcRangeMax},
        Jmin:       &IntRange{Min: junkMinValue, Max: junkMinValue + junkRangeSize},
        Jmax:       &IntRange{Min: junkMinValue, Max: junkMinValue + junkRangeSize},
        S1:         &IntRange{Min: 0, Max: sPrefixRangeMax},
        S2:         &IntRange{Min: 0, Max: sPrefixRangeMax},
        S3:         &IntRange{Min: 0, Max: sPrefixRangeMax},
        S4:         &IntRange{Min: 0, Max: s4RangeMax},
        HRangeSize: &IntRange{Min: int(headerMinRange), Max: int(headerMaxValue - headerMinValue - headerMinRange*3)},
    }
}
```

**Why pointer fields**: a nil pointer is structurally distinct from any valid range. `IntRange{Min: 0, Max: 0}` (which a value-typed sentinel `IsZero()` would treat as "unset") is **not** a legal range — `Max == 0` always fails schema validation at preset-load time, never reaches the generator. There is no `IsZero()` fallback method. Detection of "unset" is purely `field == nil`.

**Why `IntRange` over `min, max int`**: groups fields semantically; preset YAML maps `1:1` (`{min: 0, max: 11}`); validation lives on the type. Eight pairs of separate int fields would be 16 GeneratorOptions fields.

**Why a struct (not a `func(*GeneratorOptions)` builder)**: simpler. The functional-options pattern is for constructors with many optional knobs; we have a clean dataclass.

**Why not a single `RangeMap map[string]IntRange`**: type safety. Compile-time field names beat string keys, especially for refactor.

### 5.4 Package layout: loader in `internal/presets/`, embed FS at repo root

```text
<repo root>/
├── presets_embed.go           # NEW: declares `//go:embed presets/*.yaml`; package amnezigo
├── presets/                   # community contribution surface
│   └── example.yaml
└── internal/presets/          # loader / schema / validation; package presets
    ├── defaults.go            # built-in defaults as Go literal
    ├── load.go                # Load(name) → *Preset; reads via amnezigo.EmbeddedPresets()
    ├── load_test.go
    ├── preset.go              # Preset struct, schema validation, ToGeneratorOptions()
    └── preset_test.go
```

**Decision**: schema / loader / validation live in `internal/presets/` (binary-private). The `//go:embed` directive itself lives at the repo root in package `amnezigo` because Go's embed forbids `..` in patterns (see § 5.5). The package boundary is preserved — only the embed FS crosses it, exposed via the public helper `amnezigo.EmbeddedPresets() embed.FS`.

**Reasons**:

- Encapsulation: `Preset` struct, schema validation, and YAML wiring belong together in `internal/presets/`; they stay binary-private. The only public surface is a thin loader API.
- Embed location is forced: `//go:embed` patterns must be relative to the source file's directory and must not contain `..`. The directive must live in a file co-located with (or above) `presets/`. Since `presets/` is at the repo root, the embed file goes in package `amnezigo` at the repo root.
- Future-proof: when P3.5 adds remote registry, the loader package boundary makes it easy to swap the source of truth (embedded → HTTP-fetched cache). The repo-root embed file is a one-line helper that can stay or be replaced.
- Consistent with existing structure: `internal/cli/` already establishes the "binary-private impl detail" convention; `internal/presets/` follows it for everything except the unavoidable embed directive.
- Public surface: `amnezigo.EmbeddedPresets() embed.FS` (read-only filesystem accessor) plus `presets.Load(name string) (*Preset, error)` plus `Preset.ToGeneratorOptions() amnezigo.GeneratorOptions`.

### 5.5 Embed source-of-truth: co-located embed file at repo root

The community contract demands that `presets/` lives at repo root (community PRs target `presets/ru-mts.yaml`, not `internal/presets/presets-data/ru-mts.yaml`).

**Verified constraint**: Go's `embed` directive **forbids** `..` in patterns. Empirical test with Go 1.26.1 produces `pattern ../...: invalid pattern syntax`. The Go spec states: "Patterns must not match files outside the package's module... Patterns may not contain '.' or '..' or empty path elements". This is a hard rule, not version-dependent — the directive `//go:embed all:../../presets/*.yaml` from a file in `internal/presets/` will not compile.

**Decision (verified)**: co-locate the embed directive with the `presets/` directory by placing it at the repo root in package `amnezigo`. A NEW file `presets_embed.go` is added at the repo root with this content:

```go
package amnezigo

import "embed"

//go:embed presets/*.yaml
var embeddedPresets embed.FS

// EmbeddedPresets returns the embedded preset filesystem.
// Used by internal/presets to load preset YAML at startup.
func EmbeddedPresets() embed.FS {
    return embeddedPresets
}
```

The `internal/presets` package consumes the embedded filesystem via `amnezigo.EmbeddedPresets()`; it does NOT own the embed directive itself. This keeps the loader / schema / validation logic encapsulated in `internal/presets/` while satisfying the embed-pattern constraint.

**Trade-off acknowledged**: the embed directive sits in the public package `amnezigo`, slightly increasing the public surface (one exported function and an unexported `embed.FS` variable). This is unavoidable given Go's embed constraints. The alternative (Option B, mirror directory under `internal/presets/presets-data/` with a build-time copy) was rejected — risk of drift, CI must enforce equivalence, build script ceremony on every commit.

### 5.6 Preset name → filename mapping

- A preset named `ru-mts` lives at `presets/ru-mts.yaml`.
- Mapping is mechanical: `name + ".yaml"` for the basename, joined to the `presets/` root.
- The loader validates that the YAML's `name:` field equals the filename basename. Mismatch is a hard error: "preset file ru-mts.yaml declares name 'mts-ru'; filename and name field must match".
- Naming convention `[a-z0-9-]+`: enforced both at filename level (filesystem-traversal safety; no `../`) and at name-field level. CONTRIBUTING.md (P3.5 / community-driven) will recommend `<region>-<isp>` format.

### 5.7 Where validation happens

- **At load time** (`internal/presets/load.go`): schema validation of YAML structure (required fields, types, ranges, name/filename match).
- **At preset-build time** (`Preset.ToGeneratorOptions()`): conversion to `GeneratorOptions`; produces a `GeneratorOptions` that is always valid by construction.
- **At generation time** (existing `ValidatePacketSizes` from P0.3): the AWG 2.0 size invariant. Preset-supplied ranges *cannot* opt out of this invariant; if a malicious preset chose `s1: {min: 0, max: 0}`, the existing `pairsDistinct` retry loop still runs.
- **At CI time** (PR 4, gated on P1.3): for each `presets/*.yaml`, run `amnezigo init --preset ${name} --ipaddr 1.2.3.4 --config /tmp/${name}.conf` then `amnezigo validate /tmp/${name}.conf`. End-to-end smoke test per preset. (`init`'s output flag is `--config`, verified at `internal/cli/init.go:76`.)

### 5.8 What is NOT in `GeneratorOptions`

Explicit non-goals to keep the contract tight:

- No CPS overrides (`I1..I5` template substitution) — P3.5.
- No protocol selection (`--protocol` is a CLI flag; presets only *recommend* via `protocols:` field, informational).
- No MTU override (`--mtu` is a CLI flag; not an obfuscation parameter).
- No DNS / endpoint / interface overrides — those are infrastructure, not obfuscation.
- No retry budgets (`headerMaxAttempts`, etc.) — those are tuning, not policy.
- No `headerMinValue` / `headerMaxValue` floors — those are AWG protocol invariants.

```go
// IntRange is a closed-open range used for bounded random draws.
// Validation is the caller's responsibility; the generator assumes Min < Max
// and both fields within their schema-imposed sanity bounds (§4.2).
type IntRange struct {
    Min, Max int
}

// GeneratorOptions controls every numeric range the generator draws from.
//
// Each field is a pointer: a nil pointer means "use the default for this
// range"; a non-nil pointer means the caller has explicitly set it.
//
// Pointer fields — not value-typed sentinels — because legitimate ranges like
// IntRange{Min: 0, Max: 65} (the default S1 range) cannot be distinguished
// from "unset" via a value-typed IsZero() check. The preset loader rejects
// malformed ranges (e.g., Max == 0) at YAML-parse time, so any non-nil
// *IntRange that reaches the generator is guaranteed valid by construction.
type GeneratorOptions struct {
    Jc         *IntRange
    Jmin       *IntRange
    Jmax       *IntRange
    S1         *IntRange
    S2         *IntRange
    S3         *IntRange
    S4         *IntRange
    HRangeSize *IntRange
}

// DefaultGeneratorOptions returns the historical hardcoded ranges as
// non-nil pointer fields. Useful as a base for callers that want to
// override only specific ranges.
func DefaultGeneratorOptions() GeneratorOptions {
    return GeneratorOptions{
        Jc:         &IntRange{Min: 0, Max: jcRangeMax},
        Jmin:       &IntRange{Min: junkMinValue, Max: junkMinValue + junkRangeSize},
        Jmax:       &IntRange{Min: junkMinValue, Max: junkMinValue + junkRangeSize},
        S1:         &IntRange{Min: 0, Max: sPrefixRangeMax},
        S2:         &IntRange{Min: 0, Max: sPrefixRangeMax},
        S3:         &IntRange{Min: 0, Max: sPrefixRangeMax},
        S4:         &IntRange{Min: 0, Max: s4RangeMax},
        HRangeSize: &IntRange{Min: int(headerMinRange), Max: int(headerMaxValue - headerMinValue - headerMinRange*3)},
    }
}

// resolveOptions returns a fully-populated GeneratorOptions: any nil pointer
// in the input is replaced with the corresponding default. Non-nil pointers
// pass through unchanged. The result has every field non-nil and safe to
// dereference directly.
func resolveOptions(opts GeneratorOptions) GeneratorOptions {
    d := DefaultGeneratorOptions()
    if opts.Jc == nil         { opts.Jc         = d.Jc }
    if opts.Jmin == nil       { opts.Jmin       = d.Jmin }
    if opts.Jmax == nil       { opts.Jmax       = d.Jmax }
    if opts.S1 == nil         { opts.S1         = d.S1 }
    if opts.S2 == nil         { opts.S2         = d.S2 }
    if opts.S3 == nil         { opts.S3         = d.S3 }
    if opts.S4 == nil         { opts.S4         = d.S4 }
    if opts.HRangeSize == nil { opts.HRangeSize = d.HRangeSize }
    return opts
}
```

```go
// option ranges such that the four AWG-padded handshake sizes are pairwise

func GenerateSPrefixesWithOptions(opts GeneratorOptions) SPrefixes {
    opts = resolveOptions(opts)
    for range sMaxAttempts {
        s := SPrefixes{
            S1: drawIntInRange(*opts.S1),
            S2: drawIntInRange(*opts.S2),
            S3: drawIntInRange(*opts.S3),
```

```go
    // Cannot compare random outputs directly. Compare statistical bounds.
    const iterations = 10000
    sums := struct{ s1, s2, s3, s4, jc, jmin, jmax int }{}
    for range iterations {
        cfg := GenerateConfig("random", 1280, 32, 5)
        sums.s1 += cfg.S1; sums.s2 += cfg.S2; sums.s3 += cfg.S3; sums.s4 += cfg.S4
        sums.jc += cfg.Jc; sums.jmin += cfg.Jmin; sums.jmax += cfg.Jmax
    }
    // Expected mean for S1 (uniform [0, 65)) is ~32. Allow ±15% slack.
    avgS1 := float64(sums.s1) / iterations
    if avgS1 < 27.2 || avgS1 > 36.8 {
        t.Errorf("S1 mean %.2f outside expected band [27.2, 36.8]; refactor changed distribution", avgS1)
    }
    // Repeat for s2, s3, s4 (s4 mean ~16 for [0, 33)), jc (~5), jmin/jmax (~544).
    // ... (similar ±15% bounds)
}
```

**Why ±15% bands instead of ±5%**: with 10 000 iterations from a discrete uniform `[0, 65)`, the sample mean has standard error `≈ 65 / sqrt(12 * 10000) ≈ 0.19`, so the true mean ~32 is within ±0.6 about 99 % of the time — but that's the analytical bound for the *underlying* RNG, not for what CI sees in practice. CI environments add scheduler noise; ±5 % bands fail by chance perhaps once per ~50 runs (reported flake rate from comparable Go CIs). ±15 % bands shift the false-positive rate well below 1 / 100 000, while still catching any genuine refactor-induced regression > ~15 % drift in the distribution. Trade-off acknowledged: this is a smoke-test for "did the refactor change call sites" — it cannot detect subtle semantic regressions; it only catches gross changes (replacement of `crypto/rand`, off-by-one in range arithmetic, etc.). That's acceptable: the refactor only rewires call sites and the existing functional tests (e.g., `pairsDistinct`, `ValidatePacketSizes`) cover correctness more precisely.

Alternative not taken: chi-squared goodness-of-fit at `p > 0.01`. Tighter detection but adds `gonum/stat` dependency and is harder to read in a 30-line test. Widened bands are sufficient for this transient guard.

```go
// Package presets loads and validates AmneziaWG provider preset YAML files.
package presets

import (
    "errors"
    "fmt"
    "regexp"

    "github.com/Arsolitt/amnezigo"
)

// schemaVersion is the only supported preset schema version.
const schemaVersion = 1

// nameRe matches valid preset names.
var nameRe = regexp.MustCompile(`^[a-z0-9-]+$`)

// IntRange mirrors amnezigo.IntRange but is decoded from YAML.
// Both fields are required at the schema level: the loader rejects entries
// where a sub-key (`min` or `max`) is missing or where the constraints from
// §4.2 are violated. This is enforced at YAML-parse time, not deferred to
// runtime fallback.
type IntRange struct {
    Min int `yaml:"min"`
    Max int `yaml:"max"`
}

// Params holds the eight overrideable range pairs. Each field is a pointer:
// `nil` means "the preset does not override this range; use the built-in
// default", and a non-nil value means "the preset has explicitly set this
// range". `omitempty` on the YAML tag preserves nil-ness for absent keys.
//
// Pointers — not value-typed sentinels — because a legitimate range like
// IntRange{Min: 0, Max: 65} cannot be distinguished from an unset field
// using IsZero()-style checks. Pointer-vs-nil is the structural answer.
type Params struct {
    Jc         *IntRange `yaml:"jc,omitempty"`
    Jmin       *IntRange `yaml:"jmin,omitempty"`
    Jmax       *IntRange `yaml:"jmax,omitempty"`
    S1         *IntRange `yaml:"s1,omitempty"`
    S2         *IntRange `yaml:"s2,omitempty"`
    S3         *IntRange `yaml:"s3,omitempty"`
    S4         *IntRange `yaml:"s4,omitempty"`
    HRangeSize *IntRange `yaml:"h_range_size,omitempty"`
}

// Preset is the in-memory form of a preset YAML file.
type Preset struct {
    Name          string            `yaml:"name"`
    Description   string            `yaml:"description"`
    Version       int               `yaml:"version"`
    MaintainedBy  string            `yaml:"maintained_by"`
    TestedOn      []map[string]string `yaml:"tested_on,omitempty"`
    Params        Params            `yaml:"params"`
    Protocols     []string          `yaml:"protocols,omitempty"`
    Notes         string            `yaml:"notes,omitempty"`
}

// ToGeneratorOptions converts a validated Preset to amnezigo.GeneratorOptions.
// Nil pointer fields in p.Params translate to nil pointer fields in the
// returned GeneratorOptions, signaling "use the default" to the generator.
// Non-nil fields are copied into freshly-allocated amnezigo.IntRange values.
func (p *Preset) ToGeneratorOptions() amnezigo.GeneratorOptions {
    var opts amnezigo.GeneratorOptions
    if p.Params.Jc != nil {
        opts.Jc = &amnezigo.IntRange{Min: p.Params.Jc.Min, Max: p.Params.Jc.Max}
    }
    if p.Params.Jmin != nil {
        opts.Jmin = &amnezigo.IntRange{Min: p.Params.Jmin.Min, Max: p.Params.Jmin.Max}
    }
    if p.Params.Jmax != nil {
        opts.Jmax = &amnezigo.IntRange{Min: p.Params.Jmax.Min, Max: p.Params.Jmax.Max}
    }
    if p.Params.S1 != nil {
        opts.S1 = &amnezigo.IntRange{Min: p.Params.S1.Min, Max: p.Params.S1.Max}
    }
    if p.Params.S2 != nil {
        opts.S2 = &amnezigo.IntRange{Min: p.Params.S2.Min, Max: p.Params.S2.Max}
    }
    if p.Params.S3 != nil {
        opts.S3 = &amnezigo.IntRange{Min: p.Params.S3.Min, Max: p.Params.S3.Max}
    }
    if p.Params.S4 != nil {
        opts.S4 = &amnezigo.IntRange{Min: p.Params.S4.Min, Max: p.Params.S4.Max}
    }
    if p.Params.HRangeSize != nil {
        opts.HRangeSize = &amnezigo.IntRange{Min: p.Params.HRangeSize.Min, Max: p.Params.HRangeSize.Max}
    }
    return opts
}

// Validate checks the preset against the schema rules in §4.2 of the plan.
// All sub-range validation runs at YAML-parse time (no deferred fallback);
// any malformed range fails Load() before reaching the generator.
func (p *Preset) Validate() error {
    if !nameRe.MatchString(p.Name) {
        return fmt.Errorf("preset.name %q must match %s", p.Name, nameRe.String())
    }
    if p.Version != schemaVersion {
        return fmt.Errorf("preset.version %d unsupported; this binary supports v%d", p.Version, schemaVersion)
    }
    if p.Description == "" || len(p.Description) > 200 {
        return errors.New("preset.description must be 1..200 chars")
    }
    if p.MaintainedBy == "" {
        return errors.New("preset.maintained_by is required")
    }
    return p.Params.Validate()
}

// Validate checks every non-nil Params pair against its schema bounds.
// A nil pointer means "preset does not override this range" and is allowed
// (the generator uses the default). A non-nil pointer with malformed
// content (e.g., max == 0, min >= max, out-of-bounds) is a hard error.
func (pp *Params) Validate() error {
    pairs := []struct {
        name       string
        r          *IntRange
        minF, maxF int
    }{
        {"jc", pp.Jc, 0, 128},
        {"jmin", pp.Jmin, 64, 1024},
        {"jmax", pp.Jmax, 64, 1024},
        {"s1", pp.S1, 0, 256},
        {"s2", pp.S2, 0, 256},
        {"s3", pp.S3, 0, 256},
        {"s4", pp.S4, 0, 64},
        {"h_range_size", pp.HRangeSize, 1, 1_000_000_000},
    }
    for _, p := range pairs {
        if p.r == nil {
            continue // absent override; default applies
        }
        if p.r.Max <= 0 {
            return fmt.Errorf("params.%s.max must be > 0 (got %d)", p.name, p.r.Max)
        }
        if p.r.Min < p.minF || p.r.Max > p.maxF || p.r.Min >= p.r.Max {
            return fmt.Errorf("params.%s out of bounds: got {min:%d, max:%d}, allowed [%d, %d) with min<max", p.name, p.r.Min, p.r.Max, p.minF, p.maxF)
        }
    }
    return nil
}
```

The embed directive must live in package `amnezigo` at the repo root because Go's `embed` patterns forbid `..` (verified empirically with Go 1.26.1: `pattern ../...: invalid pattern syntax`). The file is small and has no other purpose:

```go
// presets_embed.go (at repo root, in package amnezigo)
package amnezigo

import "embed"

//go:embed presets/*.yaml
var embeddedPresets embed.FS

// EmbeddedPresets returns the embedded preset filesystem. The returned FS
// is read-only and contains every YAML file shipped under presets/ at
// build time. The internal/presets package consumes this to load presets
// by name without depending on the runtime filesystem.
func EmbeddedPresets() embed.FS {
    return embeddedPresets
}
```

The `internal/presets` package no longer owns an embed directive. It calls `amnezigo.EmbeddedPresets()` at load time. This preserves the `internal/presets` boundary (loading / schema / validation logic stays binary-private) while satisfying the embed-pattern constraint that forbids `..`.

**Step 2.5 — Write `internal/presets/load.go`**

```go
package presets

import (
    "errors"
    "fmt"
    "os"
    "path"

    "gopkg.in/yaml.v3"

    "github.com/Arsolitt/amnezigo"
)

// ErrPresetNotFound is returned when no preset file matches the requested name.
var ErrPresetNotFound = errors.New("preset not found")

// Load reads a preset by name from the embedded filesystem.
//
// Returns ErrPresetNotFound if no presets/<name>.yaml file is embedded.
// Returns a wrapped validation error if the YAML parses but fails schema checks.
func Load(name string) (*Preset, error) {
    if !nameRe.MatchString(name) {
        return nil, fmt.Errorf("invalid preset name %q (must match %s)", name, nameRe.String())
    }
    filename := name + ".yaml"
    fs := amnezigo.EmbeddedPresets()
    data, err := fs.ReadFile(path.Join("presets", filename))
    if err != nil {
        if errors.Is(err, os.ErrNotExist) {
            return nil, fmt.Errorf("%w: %s", ErrPresetNotFound, name)
        }
        return nil, fmt.Errorf("read embedded preset %s: %w", filename, err)
    }
    var p Preset
    dec := yaml.NewDecoder(bytesReader(data))
    // Top-level extras are tolerated for forward compat (§4.3).
    dec.KnownFields(false)
    if err := dec.Decode(&p); err != nil {
        return nil, fmt.Errorf("parse preset %s: %w", filename, err)
    }
    if p.Name != name {
        return nil, fmt.Errorf("preset file %s declares name %q; filename and name must match", filename, p.Name)
    }
    if err := p.Validate(); err != nil {
        return nil, fmt.Errorf("validate preset %s: %w", filename, err)
    }
    return &p, nil
}

// List returns the names of all embedded presets, sorted alphabetically.
func List() ([]string, error) { /* ... */ }
```

```yaml
# presets/example.yaml
#
# Canonical example preset for amnezigo. Demonstrates every schema field with
# explanatory comments. Copy this file as a starting template when contributing
# a provider-specific preset (e.g., ru-mts.yaml, ir-irancell.yaml, cn-cmcc.yaml).
#
# After editing:
#   1. Run `amnezigo init --preset <yourname> --ipaddr 10.8.0.1/24` locally.
#   2. Run `amnezigo validate <generated-config>` (gated on roadmap P1.3).
#   3. Open a PR. CI will run validate against your preset automatically.
#
# Schema reference: docs/presets.md (or docs/plans/p1.5-provider-presets.md §4).

# === Required: identity ===

# `name` MUST equal the filename without .yaml. Format: [a-z0-9-]+, max 32 chars.
name: example

# `description` is a one-line summary, max 200 chars. Shown in `amnezigo presets list`.
description: Canonical example preset documenting every supported field

# `version` is the schema version. v1 is the only supported value today.
# Bump to 2 when adding/removing required fields (breaking change).
version: 1

# `maintained_by` is your GitHub username. Free-form; informational only.
maintained_by: amnezigo-core

# === Optional: provenance ===

# `tested_on` lists the AWG tooling versions where this preset was confirmed to
# work. Informational; the binary does not enforce these. Use any tool name.
tested_on:
  - awg-go: v0.2.17
  - amneziawg-tools: v1.0.20260223
  - amnezigo: v0.0.0+latest

# === Required: parameter ranges ===

# Each pair MUST satisfy min < max. Bounds are listed in plan §4.2.
# The generator draws uniformly from [min, max) for each parameter.
params:

  # `jc` controls the junk-packet count emitted with each handshake.
  # Valid: 0 <= min < max <= 128. Default range: [0, 11).
  jc: { min: 0, max: 11 }

  # `jmin` and `jmax` define the junk-packet size range. The generator picks
  # a single Jmin and Jmax per server, drawn from these meta-ranges.
  # Valid: 64 <= min, max <= 1024; min < max. Cap matches the historical
  # default (junkMinValue + junkRangeSize - 1 = 64 + 961 - 1 = 1024 from
  # generator.go); pushing higher risks MTU fragmentation.
  jmin: { min: 64,  max: 1024 }
  jmax: { min: 64,  max: 1024 }

  # `s1`-`s4` control the AWG handshake-padding sizes (S1=init, S2=response,
  # S3=cookie-reply, S4=transport). Valid: 0 <= min, max <= 256 (s4 caps at 64).
  # The default ranges produce padded handshake sizes that fit comfortably below
  # standard MTU and avoid pairwise collision (enforced by the generator at runtime).
  s1: { min: 0, max: 65 }
  s2: { min: 0, max: 65 }
  s3: { min: 0, max: 65 }
  s4: { min: 0, max: 33 }

  # `h_range_size` controls the SPAN of each H1-H4 type-id range. The four
  # ranges are placed within the AWG header window [5, 2^31-1] and validated
  # to be pairwise non-overlapping.
  # Valid: 1 <= min < max <= 1_000_000_000 (the upper bound enforces enough
  # window to fit four ranges with headerMinRange-equivalent gaps).
  h_range_size:
    min: 10000000
    max: 2117483642  # = headerMaxValue - headerMinValue - 3*headerMinRange = 2147483647 - 5 - 30000000

# === Optional: protocol recommendations ===

# `protocols` lists which I-packet protocol templates are recommended for this
# preset. Informational; the user still passes `--protocol` at the CLI. If you
# include this field, every entry must be in {random, quic, dns, dtls, stun}.
protocols:
  - random
  - quic
  - dns
  - dtls
  - stun

# === Optional: free-form notes ===

# `notes` is multi-line free-form text. Document anything users should know:
# tested networks, known caveats, alternative parameter combinations.
notes: |
  This preset matches the built-in default ranges exactly. It exists as a
  template for contributors and as a smoke-test fixture for CI.

  When creating a region/ISP-specific preset, adjust `params.*` to match
  empirically observed working values, document the test environment in
  `tested_on` and `notes`, and open a PR.
```

## 9. CI Integration

**Goal**: every preset in `presets/*.yaml` is automatically validated end-to-end on every push and PR.

**Pseudocode (shipped in PR 4 as `.github/workflows/presets.yml`)**:

```bash
#!/usr/bin/env bash
set -euo pipefail
go build -o ./build/amnezigo ./cmd/amnezigo/
for f in presets/*.yaml; do
  name=$(basename "${f%.yaml}")
  tmp=$(mktemp -d)
  cfg="${tmp}/awg.conf"
  echo "::group::preset ${name}"
  ./build/amnezigo init --preset "${name}" --ipaddr 10.8.0.1/24 --config "${cfg}"
  ./build/amnezigo validate "${cfg}"  # gated on P1.3
  echo "::endgroup::"
done
```

**Gating**:

- Until P1.3 lands, the `validate` step is replaced with a placeholder check (e.g., `grep -q '\[Interface\]' "${cfg}"` to verify a config was actually written).
- When P1.3 merges, the placeholder is upgraded to the real `amnezigo validate`. PR 4 can ship with the placeholder and a follow-up trivial PR can swap the line.

**Failure handling**:

- Any preset that fails `init` or `validate` fails the CI job — community PRs cannot land.
- Failure messages should include the preset name and the error from `init`/`validate` directly.

**Performance**:

- One preset takes ~1s to build + ~10ms to init + ~10ms to validate. With 50 presets (long-term ceiling), the loop runs in ~1.5s. Negligible.

**Per-PR scope**:

- The CI loop runs for *every* preset on every PR — even PRs that don't touch presets. This is intentional: a generator refactor that breaks one preset must be caught.

## 10. Risks and Edge Cases

| Risk | Mitigation |
| --- | --- |
| **Preset name collision** — two community PRs add `ru-mts` simultaneously | Naming convention enforced at PR time by maintainer review. CI fails on duplicate filenames (filesystem doesn't allow). Recommend in `docs/presets.md` that contributors search existing presets before submitting. |
| **Preset references future schema fields** (e.g., `params.distribution: "gaussian"` in v2 on a v1 binary) | YAML decoder runs in non-strict mode at top level (§ 4.3 / § 5.1); unknown fields warn-and-ignore. `version: 2` on a v1 binary errors hard with upgrade guidance. |
| **Generator constants are referenced from existing tests directly** (e.g., test asserts `s1 < sPrefixRangeMax`) | PR 1 keeps constants exported (or rather, package-private) and the new `GeneratorOptions` defaults derive from them numerically. Tests still pass. Statistical band test (Step 1.3) catches accidental regression. |
| **`go:embed` doesn't include subdirectory contents in TinyGo / non-standard builds** | `amnezigo` is built with stdlib `go build` only. Document in `docs/presets.md`: "preset embedding requires standard Go toolchain; TinyGo builds drop preset support." |
| **Preset overrides interact with existing CLI flags** (`--mtu`, `--dns`, etc.) | CLI always wins (§ 5.2). The schema deliberately excludes infrastructure fields (MTU, DNS) from presets — they go in `notes` if a contributor wants to suggest values. |
| **YAML library transitively pulls test deps** | `gopkg.in/yaml.v3 v3.0.1` has zero transitive runtime deps and only stdlib in tests. Verified at plan-write time; verify again during PR 2 review. |
| **Preset's `s1` range incompatible with user's `--s1` (if such flag is added)** | No `--s1` CLI flag exists today and none is added by this plan. If a future PR adds it, the precedence rule (§ 5.2) makes CLI win — preset's `s1` range becomes informational. |
| **AWG protocol updates change valid range bounds** (e.g., S4 ceiling moves from 32 to 64) | The `tested_on` field captures the AWG version a preset was tested against. When updating, bump preset `version`. Older binaries warn. |
| **`pairsDistinct` retry budget exhausted under a tight preset** (e.g., `s1: {min: 5, max: 6}` forces only one S1 value, and that value collides with all generated S2/S3/S4) | The existing `sMaxAttempts = 1000` retry loop panics on exhaustion. Document in `docs/presets.md` that overly-tight ranges may cause generation failure; recommend min span of ≥ 5 per parameter. Add a smoke test in PR 2 that loads a synthetic tight preset and asserts the panic message is informative. |
| **Preset YAML edited at runtime to inject malicious values** | Embedded presets are compiled into the binary; runtime modification requires recompilation. The (deferred) `--preset-file path.yaml` escape hatch would change this; gate that PR with explicit security review. |
| **Schema validation false-positives** (an honest preset is rejected by an over-tight `pp.maxF` cap in `Params.Validate`) | Caps are sanity guards (§ 4.2), not protocol limits. If a real-world preset legitimately needs `s1.max = 300`, the right response is to relax the cap in code, not to bypass validation. Document the caps' rationale in `docs/presets.md`. |
| **CI loop slow with many presets** | Per § 9 perf analysis, ≤2s for 50 presets. Far from a CI bottleneck. Re-evaluate if presets exceed 200 (unlikely). |
| **`presets/.keep` fixture left in tree after PR 4** | Delete in PR 4. Verify with `git status` before merging PR 4. |

## 12. Out of Scope (Explicit)

The following are **deliberately excluded** from this plan:

- **`--preset-file path.yaml` runtime escape hatch.** Embedded presets are sufficient for v1. Adding a runtime-file path opens a security review (path traversal, untrusted YAML); deferred.
- **Preset overrides for CPS templates** (per-protocol I-packet template substitution) — this is **P3.5** territory.
- **Remote preset registry** (`update-presets` subcommand fetching `https://github.com/.../presets-registry.json`) — **P3.5**.
- **CONTRIBUTING.md guidelines for community submission** — roadmap decision 2: design when first PRs land. PR 4's `docs/presets.md` provides a minimal stub.
- **Provider-specific empirical presets** (`ru-mts.yaml`, `ir-irancell.yaml`, `cn-cmcc.yaml`, etc.) — separate community PRs (PR 5+), not this plan.
- **Snapshot testing** (locking `expected_output.conf` per preset) — over-constrains the implementation; defer to P3.5 if needed at all.
- **`amnezigo presets list` / `amnezigo presets show NAME` subcommands** — useful but out of scope for v1; track in P3.5 backlog.
- **Versioning the preset library independently of the binary** — out of scope; preset versioning is per-file via `version: 1`.
- **Refactoring `internal/cli/init.go`'s endpoint detection** — orthogonal; stays as-is.
- **MTU / DNS / endpoint fields in preset schema** — those are infrastructure, not obfuscation. Deliberately omitted.
- **GeneratorOptions for retry budgets** (`headerMaxAttempts`, `sMaxAttempts`, etc.) — those are tuning, not policy; not exposed.
- **Removing the package-level constants `s4RangeMax`, `sPrefixRangeMax`, etc.** — kept as the source of truth that `DefaultGeneratorOptions()` reads; deletion would break the equivalence proof.
- **Migration of `cps.go`'s package-private constants to GeneratorOptions** — CPS ranges are P3.5 territory.

If any of these come up during implementation, **do not include them**. Note them for follow-up.
