## 1. Context

`amnezigo` is a Go CLI that generates AmneziaWG configuration files. Every packet on the wire (handshake init, response, cookie reply, transport, junk, I-packet) has a length that the receiver uses for classification. AmneziaWG's S-padding adds `S1..S4` bytes to the four WireGuard message types so that obfuscated packets do not match vanilla sizes — but that only works if the four padded sizes (and the per-peer I-packet sizes, and the junk range) are pairwise distinct.

[PR #103 in `amneziawg-go`](https://github.com/amnezia-vpn/amneziawg-go/pull/103) (commit `0361c54d`, merged 2025-12-01) fixed a bug where transport packets matching handshake-padded sizes were misclassified as init/response/cookie. Until that fix landed, this was a security/correctness issue. The generator must enforce that no collision is ever produced — a config that triggers the legacy bug is unsafe even on patched servers, because mixed-version deployments still occur.

WireGuard message constants from `device/noise-protocol.go` of `amneziawg-go`:

```text
MessageInitiationSize  = 148
MessageResponseSize    =  92
MessageCookieReplySize =  64
MessageTransportSize   =  32   // empty transport / keepalive
```

After AWG padding, the four padded sizes must all differ pairwise:

- `S1 + 148`
- `S2 +  92`
- `S3 +  64`
- `S4 +  32`

Plus three transitive constraints:

1. Each `len(I_i)` (per-peer, `i ∈ {1..5}`) must not equal any of the four padded sizes.
2. The junk range `[Jmin..Jmax]` must not include any of the four padded sizes.
3. The junk range `[Jmin..Jmax]` should additionally not include the un-padded WG constants (`148, 92, 64, 32`), because some receivers may probe before unpacking.

## 2. Goal

Make `amnezigo` provably emit configs that satisfy AWG 2.0's size-classification invariant:

- All six pairwise S-padding collisions are checked at generation time.
- I-packet generator retries on collision against any of the four padded sizes.
- Junk range generator rejects ranges that include any of the four padded sizes (and the un-padded constants).
- A reusable `ValidatePacketSizes` helper exists for P1.3's `validate` CLI command.
- A property test asserts that **1000 random generations produce zero collisions**.

## 3. Key Finding from Inventory

Verified actual code state (paths, line numbers) against the roadmap's claims. **Three discrepancies found**, listed in section 12 (References — Roadmap deltas). The most consequential:

- The roadmap says: "`generator.go` — `GenerateSPrefixes` already checks `S1+56 != S2`". Confirmed: the check lives at `generator.go:38`, with the loop wrapping `s1`/`s2` generation at `generator.go:31-41`. **However**, this check is **incomplete** — `S1+56 != S2` is the *Init/Response* pairing only (148-92=56), so it covers exactly one of the six pairwise collisions. The generator silently accepts the other five. There is **no** validation involving `S3, S4`, no validation of I-packet sizes against handshake padding, and no validation of the junk range against either padded or un-padded WG sizes.
- The roadmap says: "`cps.go` — when generating I-packets, reject sizes that collide". Confirmed: I-packets are produced by two paths — `buildAndValidateCPS` (`cps.go:103`) for protocol templates and `generateSimpleI` (`cps.go:246`) for `--protocol random`. **Neither path performs collision checks today**. Both have a fallback to `tagTerminate` (`<t>`, 4 bytes after P0.2) — that fallback itself can collide with `S4+32` if `S4 == -28`, which is impossible (`S4 ≥ 0`), but **with `S4=0` the fallback length 4 is fine; with `S4=0` the padded transport size is 32**, a different number. We must still verify the fallback at runtime, not by static reasoning about magic numbers.
- The roadmap implies the new helper signature `ValidatePacketSizes(s1, s2, s3, s4 int, iPacketSizes []int, jmin, jmax int) error` lives in a new file. Confirmed: no such helper exists. Best home is a new file `validation.go`, not an addition to `generator.go`, because P1.3's `validate` CLI command will import it independently of the generator.

### Step 2 — Code change (Green)

#### 2a. Create `validation.go`

```go
package amnezigo

import (
	"errors"
	"fmt"
	"sort"
)

// WireGuard message size constants from amneziawg-go device/noise-protocol.go.
// These are the on-the-wire sizes BEFORE AWG S-padding is applied.
const (
	wgInitiationSize  = 148
	wgResponseSize    = 92
	wgCookieReplySize = 64
	wgTransportSize   = 32
)

// PacketSizeCollisionError describes a single size-classification collision in a config.
// It is returned by ValidatePacketSizes when any of the AWG 2.0 invariants is violated.
type PacketSizeCollisionError struct {
	// Kind is one of: "s-pair", "i-packet", "junk-range".
	Kind string
	// Pair names the colliding entities, e.g. "S1+148 vs S2+92" or "I3 vs S4+32"
	// or "[Jmin..Jmax] contains 148".
	Pair string
	// Size is the colliding numeric size in bytes (or the boundary value for junk ranges).
	Size int
}

func (e *PacketSizeCollisionError) Error() string {
	return fmt.Sprintf("packet size collision (%s): %s at %d bytes", e.Kind, e.Pair, e.Size)
}

// errEmptyJunkRange is returned when jmin > jmax. ValidatePacketSizes treats
// this as a structural error, not a collision.
var errEmptyJunkRange = errors.New("junk range is empty (jmin > jmax)")

// paddedSizes returns the four AWG-padded packet sizes.
// Order: init, response, cookie, transport.
func paddedSizes(s1, s2, s3, s4 int) [4]int {
	return [4]int{
		s1 + wgInitiationSize,
		s2 + wgResponseSize,
		s3 + wgCookieReplySize,
		s4 + wgTransportSize,
	}
}

// ValidatePacketSizes enforces the AWG 2.0 size-classification invariant:
//   1. The four S-padded handshake sizes are pairwise distinct.
//   2. No I-packet length equals any of the four padded sizes.
//   3. The junk range [jmin..jmax] does not include any of the four padded sizes
//      and does not include any of the four raw WireGuard message sizes.
//
// It returns nil if all invariants hold, or a *PacketSizeCollisionError describing
// the first violation found. Order of checks: S-pairs → I-packets → junk range.
//
// Designed to be reused by P1.3's `amnezigo validate` CLI command.
func ValidatePacketSizes(s1, s2, s3, s4 int, iPacketSizes []int, jmin, jmax int) error {
	if jmin > jmax {
		return errEmptyJunkRange
	}
	padded := paddedSizes(s1, s2, s3, s4)
	pairLabels := [4]string{"S1+148", "S2+92", "S3+64", "S4+32"}

	// 1. Six pairwise S-padding checks.
	for i := range 4 {
		for j := i + 1; j < 4; j++ {
			if padded[i] == padded[j] {
				return &PacketSizeCollisionError{
					Kind: "s-pair",
					Pair: fmt.Sprintf("%s vs %s", pairLabels[i], pairLabels[j]),
					Size: padded[i],
				}
			}
		}
	}

	// 2. I-packet vs padded-size checks.
	for idx, sz := range iPacketSizes {
		for i, p := range padded {
			if sz == p {
				return &PacketSizeCollisionError{
					Kind: "i-packet",
					Pair: fmt.Sprintf("I%d vs %s", idx+1, pairLabels[i]),
					Size: sz,
				}
			}
		}
	}

	// 3. Junk range vs padded sizes and raw WG constants.
	forbidden := []int{
		padded[0], padded[1], padded[2], padded[3],
		wgInitiationSize, wgResponseSize, wgCookieReplySize, wgTransportSize,
	}
	sort.Ints(forbidden)
	for _, f := range forbidden {
		if f >= jmin && f <= jmax {
			return &PacketSizeCollisionError{
				Kind: "junk-range",
				Pair: fmt.Sprintf("[Jmin..Jmax] contains %d", f),
				Size: f,
			}
		}
	}

	return nil
}
```

Extend `buildAndValidateCPS` and `generateSimpleI` to take a "forbidden sizes" set (the four padded sizes) and treat collision as a retry trigger.

```go
// buildAndValidateCPS builds a CPS from template tags and validates it fits
// within maxSize AND its built byte-length is not in forbiddenSizes.
// On forbidden collision, attempts to perturb the template (append <rd 1>) up
// to a small budget before falling back to tagTerminate.
func buildAndValidateCPS(tags []TagSpec, maxSize int, forbiddenSizes [4]int) string {
	cps := buildCPSFromTemplate(tags)
	if cpsAcceptable(cps, maxSize, forbiddenSizes) {
		return cps
	}
	// Try shrinking first (existing behavior preserved).
	for len(tags) > 0 {
		tags = tags[:len(tags)-1]
		cps = buildCPSFromTemplate(tags)
		if cpsAcceptable(cps, maxSize, forbiddenSizes) {
			return cps
		}
	}
	// Try padding-perturbation: append <rd 1>, <rd 2>, ... up to <rd 8>.
	// Each iteration changes built length by +1 byte, so at most 8 attempts cover
	// any 4-element forbidden set (pigeonhole: 8 distinct lengths, ≤4 forbidden).
	for n := 1; n <= 8; n++ {
		perturbed := tagTerminate + fmt.Sprintf("<rd %d>", n)
		if cpsAcceptable(perturbed, maxSize, forbiddenSizes) {
			return perturbed
		}
	}
	// Last resort: tagTerminate alone (4 bytes after P0.2). If even this collides,
	// the surrounding generator will retry with different S-prefixes.
	return tagTerminate
}

// cpsAcceptable returns true iff cps fits maxSize AND its length is not forbidden.
func cpsAcceptable(cps string, maxSize int, forbidden [4]int) bool {
	n := calculateCPSLength(cps)
	if n >= maxSize {
		return false
	}
	for _, f := range forbidden {
		if n == f {
			return false
		}
	}
	return true
}

// generateSimpleI now takes a forbidden-sizes set.
func generateSimpleI(maxSize int, forbiddenSizes [4]int) string {
	for range iPacketMaxAttempts {
		tags := generateRandomTags()
		cps := tagsToCPS(tags)
		if cpsAcceptable(cps, maxSize, forbiddenSizes) {
			return cps
		}
	}
	// Fallback ladder identical to buildAndValidateCPS perturbation.
	for n := 1; n <= 8; n++ {
		perturbed := tagTerminate + fmt.Sprintf("<rd %d>", n)
		if cpsAcceptable(perturbed, maxSize, forbiddenSizes) {
			return perturbed
		}
	}
	return tagTerminate
}
```

## 7. Design Decisions

This section documents the non-obvious decisions and their rationale. Every brainstorm question from `docs/roadmap.md` § P0.3 has an explicit answer here.

### 7.1 Where the validator lives — `validation.go`

**Decision**: new file `validation.go` in the package root, not an addition to `generator.go`.

**Rationale**:

- P1.3 (`amnezigo validate <config>` CLI command) will import the validator independently of the generator. Co-locating it with `generator.go` would force `validate` to depend on randomness primitives it doesn't need.
- `validation.go` is the natural home for the AWG 2.0 invariants as a whole. P0.4 (H1-H4 type-id forbidding) and P0.5 (`<rc>` charset) will likely contribute to the same file as their validation flavors.
- The package layout in `AGENTS.md` already lists separate files per responsibility (`generator.go`, `cps.go`, `helpers.go`, etc.). `validation.go` fits the pattern.

### 7.2 Error type — typed `*PacketSizeCollisionError`

**Decision**: typed error with `Kind`, `Pair`, `Size` fields, not a sentinel.

**Rationale**:

- The `validate` CLI in P1.3 will print human-readable diagnostics. Sentinels lose the offending values; structured fields preserve them.
- `errors.Is` is unnecessary here — collision errors are not "expected" in the way `os.ErrNotExist` is; callers want to inspect, not branch on identity.
- The `errEmptyJunkRange` sentinel is kept separately because it represents a structural error in the input, not a generated collision. Exposing it as a sentinel allows `validate` to distinguish "config is malformed" from "config is unsafe".

### 7.3 Retry budget — 1000 attempts each (mirrors `headerMaxAttempts`)

**Decision**: `sMaxAttempts = junkMaxAttempts = iPacketMaxAttempts = 1000`. Three separate constants, not one shared.

**Rationale**:

- The roadmap brainstorm question: "What's the retry budget before the generator gives up? Current code has `maxAttempts = 1000` for H ranges; same here?" — answer: yes, but use distinct constants.
- Distinct constants because the search spaces differ. S-prefix space is `(65×65×65×33) ≈ 9M` combinations with only 6 pair constraints — the first attempt almost always succeeds. Junk range space is `(961×961) ≈ 923K` combinations with up to 8 forbidden integers in `[64..1024]` — chance of overlap per attempt ≈ `8/961` ≈ 0.83%; 1000 attempts is overkill but cheap. I-packet space depends on `<r N>` random byte counts and tag composition — 1000 attempts is justified because the random-mode generator already retries up to 100× on MTU alone, and we are adding a second filter that's roughly the same density.
- **Going below 1000** risks panic on hot loops; **going above 1000** wastes CPU on edge cases (e.g. `S1=0` with junk range `[64..150]`). 1000 is the sweet spot consistent with `headerMaxAttempts`.

### 7.4 ±1 buffer around WG constants — NO

**Decision**: do NOT widen the constraint to also avoid `±1` of the WG constants.

**Rationale**:

- Roadmap brainstorm question: "Should we widen the constraint to also avoid sizes within ±1 of WG constants?"
- The actual collision condition is exact equality. Padding by 1 byte changes the size by 1 byte, which is detectable as a different size — so `S1+148+1 = 149` is *not* a collision.
- Widening to ±1 over-constrains the search: junk range `[64..1024]` has 961 valid integers; subtracting 8 forbidden + 16 buffered (±1 around 8 numbers, with overlap) ≈ 22 lost integers, still ≈ 97.7% legal — fine for junk. But for `S` prefixes (`[0..64]`, only 65 values), each `±1` doubles the constraint density and risks the 1000-retry budget on adversarial S1 inputs.
- If AWG ever shifts a message size by 1 byte (extremely unlikely — these are protocol constants from the Noise framework), the right fix is to update `wgInitiationSize` and friends, not to pre-emptively buffer.

### 7.5 Junk range — exclude both raw WG constants AND padded sizes

**Decision**: forbid all 8 sizes (`148, 92, 64, 32, S1+148, S2+92, S3+64, S4+32`).

**Rationale**:

- Roadmap brainstorm question: "For the junk range, is excluding 148/92/64/32 sufficient, or should we exclude the padded sizes too (which depend on S1-S4)?"
- Junk packets are received by the same classifier as handshakes. After the receiver strips S-padding (or before, if the implementation probes raw lengths), a junk packet whose size matches `S1+148` will be misclassified as a handshake init.
- Some receivers may probe before unpacking — that's why raw constants 148/92/64/32 must also be excluded. (If `Jmin..Jmax = [60..200]`, a junk of length 148 looks like a vanilla WG initiation to a probing classifier.)
- Implementation cost: 8 integers to check per range, O(1) per attempt. Cheap.

### 7.6 Randomness for the property test — `crypto/rand`, not `math/rand`

**Decision**: use the existing `crypto/rand` path; do not introduce `math/rand` with a deterministic seed.

**Rationale**:

- The codebase commits to `crypto/rand` for all generation. Introducing `math/rand` for tests creates a discrepancy: the test would prove the math/rand version is collision-free, not the crypto/rand version that ships.
- Determinism comes for free with 1000 iterations: if any run fails, the failure is reproducible by the same loop on the next CI run (all seeds covered statistically).
- Trade-off accepted: a flaky test would be hard to debug without a seed. Mitigation: `t.Errorf` includes the offending S/J/I values so a single failure is fully diagnostic without needing to replay.

### 7.7 I-packet retry strategy — perturb with `<rd N>`

**Decision**: on collision, append `<rd N>` for `N ∈ [1..8]` until non-colliding.

**Rationale**:

- Roadmap implies: "retry with a tweaked CPS template (e.g. add/remove an `<rd 1>` or extend `<r N>`)".
- `<rd N>` (random digits) is the cheapest perturbation: 1 byte per increment, doesn't conflict with template semantics (digits are inert in QUIC/DNS/DTLS/STUN templates' meaningful regions).
- Pigeonhole: with 4 forbidden lengths, 8 distinct perturbation lengths guarantee at least 4 valid options. Budget is finite and small.
- Could use `<r 1>` (random binary) instead — same result. Sticking with `<rd>` for human-readability of the produced CPS during debugging.

### 7.8 Duplicate I-packet sizes — allowed

**Decision**: `iPacketSizes` may contain duplicates (e.g. `I1` and `I2` happen to land at 167 bytes each). `ValidatePacketSizes` does not flag this.

**Rationale**:

- AWG receiver classifies incoming packets by length against handshake/transport sizes. Multiple I-packets sharing a length is not a classification error — both are I-packets, both behave the same.
- Disallowing duplicates would force the I-packet generator into a 5-way joint search, increasing retry rate without security benefit.
- Documented explicitly in `TestValidatePacketSizes_DuplicateIPacketSizes_OK`.

### 7.9 User-supplied S1 — validator uses output S1, not random S1

**Decision**: `GenerateConfig` validates against the S1 actually written to the config (the user-supplied one), and the inner S2/S3/S4 generation must adapt to it via `GenerateSPrefixesWithS1`.

**Rationale**:

- `GenerateConfig(protocol, mtu, s1, jc)` takes user `s1` and writes it to the output, ignoring the random `s.S1` from `GenerateSPrefixes()`. The current code at `generator.go:101` already does this; the bug it inherits is that `s.S2, s.S3, s.S4` were generated against the random `s.S1`, so the only existing pair check (`s.S1+56 != s.S2`) used the wrong S1.
- New helper `GenerateSPrefixesWithS1(fixedS1 int) SPrefixes` accepts the fixed S1 and validates the six-pair constraint against it.
- `GenerateSPrefixes()` (arity 0) is preserved for `GenerateServerConfig` and tests; it just calls `GenerateSPrefixesWithS1(rand)` internally for symmetry.

## 8. Risks and Edge Cases

| Risk | Assessment |
| --- | --- |
| Property test flakes on rare retry-budget exhaustion | **Low.** With 1000 retries per primitive and ≈1% collision rate per attempt, the probability of exhausting the budget is `(0.01)^1000` — astronomically unlikely. If it does happen, the panic is loud and reproducible from logs. |
| `GenerateJunkParams` signature change breaks downstream | **Mitigated.** Only `generator.go:93,177` call it. Both updated in this PR. Public API note in commit body. Verified via `grep` in step 6.2c. |
| User-supplied `s1` makes six-pair impossible (e.g. forces collision regardless of S2/S3/S4 choice) | **Bounded.** Since `S1 ∈ [0..64]` user-input and `S2,S3 ∈ [0..64], S4 ∈ [0..32]`, every user S1 has at least 50%-density of legal (S2,S3,S4) triples by counting. 1000 retries trivially finds one. |
| `cpsAcceptable` uses `<` for `maxSize` (strict) — boundary case `n == maxSize-1` falls through forbidden check incorrectly | **N/A.** `maxSize` is the MTU upper bound (`>=` triggers reject); forbidden is exact-equality. They're orthogonal checks; both must pass. |
| Existing test `TestGenerateSPrefixes` (line 28) becomes redundant with new six-pair test | **Keep both.** The single-pair check is a regression guard for the original bug; cheap to keep, valuable signal if the new logic regresses. |
| `<c>` removal (P0.1) hasn't landed; current generator still emits `<c>` whose 8-byte accounting may push I-sizes into `S1+148` collision range with templates relying on it | **Acceptable.** This PR sits *after* P0.2 in the chain and *before* P0.1 in roadmap order. The collision validator catches such collisions at runtime; the I-packet retry will perturb away from them. After P0.1 lands, the forbidden set is unchanged but the I-packet length distribution shifts — the property test catches any regression. |
| Performance: `ValidatePacketSizes` called on every `GenerateConfig` as a post-condition adds overhead | **Negligible.** O(8 + 5*4 + 8) = ~36 integer comparisons per call. Generation is dominated by `crypto/rand` and string formatting; validation is <0.1% of that budget. |
| Race condition in property test if `GenerateConfig` shares state | **Verified no shared state.** All RNG state lives in `crypto/rand` package globals, which are goroutine-safe. Test is single-threaded; even if extended to `t.Parallel()`, no shared mutable state. |
| Property test slow in CI | **Measured budget.** 1000 iterations × ~5ms per `GenerateConfig` ≈ 5s. Acceptable for CI; if it exceeds 30s, reduce to 500 iterations and document. |
