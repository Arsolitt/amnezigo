## 1. Context

`amnezigo` is a Go CLI at `/home/arsolitt/projects/amnezigo` that generates AmneziaWG v2.0 configuration files. The generator side is now well-validated (P0.1-P0.5 all merged on `dev`). The **read** side, however, has a glaring asymmetry: `ParseServerConfig` (`parser.go:20`) silently `continue`s on every unknown key (`parser.go:46`), so a config containing leftover `<c>` tags, malformed S-prefixes, junk-range collisions, or H1-H4 ranges that overlap WG message type-ids will load without complaint — and only fail at runtime on the wire, possibly in production.

Reasons to ship `validate` now (in priority order):

1. **Migration sanity check**. Users coming from other AWG generators (the original Habr-article shell scripts, AmneziaVPN GUI exports, hand-rolled INI files) need a single source of truth: "is this config compatible with the AWG 2.0 invariants amnezigo enforces today?"
2. **Legacy `<c>` detection**. P0.1 removed `<c>` from the generator. Existing client configs in user directories may still contain `<c>` (shell scripts kept generating them; users archived old templates). Today, `parser.go:46` silently swallows the unknown line; the user discovers the problem only when iOS / Android refuses the import. `validate` surfaces it as a hard error referencing the P0.1 rationale.
3. **Pre-flight before deployment**. Before pushing a config to a server or distributing it to clients, run `amnezigo validate server.conf` in CI / Makefile / pre-commit. Non-zero exit fails the pipeline.
4. **Community education**. The output explains *why* each rule exists (size collision attacks, vanilla-WG masquerade, mobile compatibility), not just *what* failed. Turning silent generator constants into visible diagnostics is a documentation multiplier.
5. **Foundation for P1.4 (`analyze`)**. The `Finding` struct, severity model, output formatting, and strict-parser lift cleanly into `analyze` — that command extends the same primitives with traffic-shape inference rather than rewriting them.

Current state: `parser.go` parses INI, `validation.go` exports `ValidatePacketSizes` (P0.3) and houses unexported `validateHeaderRange` (P0.4). Generator orchestrates these via `GenerateConfig` (`generator.go:174-216`). No CLI surface for read-side validation; no `Finding` model; no JSON output convention.

## 2. Goal

Ship `amnezigo validate <config>` — a new Cobra subcommand that reads a config file, parses it (server format primarily; client format scope decided in § 4.3), runs every applicable validation rule, and prints findings with explicit severity. Exit code is `0` when there are no errors, `1` when at least one error is present. Warnings and info findings never fail the exit code unless `--strict` (warnings-fatal) is passed.

The command is **read-only**. No mutation, no auto-fix, no rewriting on disk.

**In scope (this PR):**

- New CLI subcommand `validate`, registered alongside `init`, `add`, `edit`, `list`, `export`, `remove`.
- Strict-parser opt-in: `ParseOptions{Strict bool}` accepted by a new `ParseServerConfigWithOptions(io.Reader, ParseOptions)`. `ParseServerConfig` keeps current arity (back-compat).
- New types `Finding`, `Severity`, `Location` in `validation.go`.
- New orchestrator `ValidateServerConfig(*ServerConfig) []Finding` in `validation.go` that runs every check.
- Output formats: human-readable (default) and `--output json`.
- `--strict` flag (warnings-fatal) and `--quiet` flag (only print findings, suppress summary).
- All P0.1-P0.5 invariants surfaced as findings.
- Integration test: generate fresh config → `validate` → 0 findings; mutate config (inject `<c>`, swap H1 to overlap WG type-ids, force S1+S2 collision) → expect specific errors.
- Documentation update: `docs/cli-reference.md` and a top-level `README.md` mention.

**Explicit out-of-scope (deferred):**

- `--fix` auto-fix mode. Even for the trivial `<c>` → `<r 1>` substitution, "fix" semantics need a follow-up design (do we preserve the user's tag intent? Re-run pairwise size collision avoidance? Sign-and-write semantics?). Not in this PR. Tracked as P1.3-followup; revisit only if a user needs it.
- WireGuard interop (does the IP route? does the key auth?) — out of scope; that's `wg-quick` territory and `analyze` may probe later.
- Cryptographic validation (private key matches public key, preshared key well-formed). Decoders already error on malformed Base64; we surface decoder errors verbatim but do not re-derive keys.
- New CLI commands beyond `validate` (`analyze` is P1.4).
- A `Validate(io.Reader) []Finding` library convenience wrapper that bundles parse+validate. Defer to P1.4 once the second consumer makes the abstraction worthwhile.

## 4. Architecture Decisions

### 4.1 Strict-mode parser — `ParseOptions` struct, not a separate function

**Decision**: introduce `type ParseOptions struct{ Strict bool }` and a new entry point `func ParseServerConfigWithOptions(r io.Reader, opts ParseOptions) (ServerConfig, []ParseWarning, error)`. `ParseServerConfig(r io.Reader)` keeps its current signature and internally calls `ParseServerConfigWithOptions(r, ParseOptions{})`.

**Rationale**:

- Adding a positional `bool` parameter to `ParseServerConfig` is a back-compat break (all callers — `manager.go`, `parser_test.go`, downstream — would need to update). Avoidable.
- A bool struct lets us add future options (`AllowI1I5 bool`, `MaxConfigBytes int`) without recompiling callers each time.
- Return tuple grows by one slot: `[]ParseWarning`. In `Strict=false` (current behavior), it's always empty. In `Strict=true`, it carries unknown-key sightings, raw `<c>` mentions, malformed numeric values, etc. Distinct from the H-range error: H-range malformations remain `error` because they're structurally fatal — you cannot validate a config whose `H2` line failed to parse into a `HeaderRange`. The findings model handles semantic violations, not structural ones.
- Alternative considered: `ParseServerConfigStrict(io.Reader)` as a sibling. Rejected: bool flips the parser's behavior in a small number of code paths (lines 46, 96-148, 199-209); duplicating the parser into a strict copy would diverge under maintenance. One implementation, options-driven.

**Consequence**: the parser collects warnings but H-range errors still abort parse. `validate` runs the parser in strict mode and converts a returned `error` into a single `Finding{Severity: Error, ...}` plus zero or more warnings recovered before the abort.

### 4.2 Findings model — `Finding`, `Severity`, `Location` in `validation.go`

**Decision**: define the model in `validation.go` (not a new package). Three types:

```go
// These types are the shared findings model. P1.4 (`analyze` command) reuses them; do not break wire-compatibility without coordinating.

type Severity string

const (
    SeverityError   Severity = "error"
    SeverityWarning Severity = "warning"
    SeverityInfo    Severity = "info"
)

type Location struct {
    File string `json:"file,omitempty"` // populated by the CLI from the user's argument
    Line int    `json:"line,omitempty"` // 0 = unknown / whole-config check
    Key  string `json:"key,omitempty"`  // INI key name when applicable, "" otherwise
}

type Finding struct {
    Severity Severity `json:"severity"`
    Code     string   `json:"code"`               // stable machine-readable identifier, e.g. "PSC001"
    Location Location `json:"location,omitempty"` // omitempty: empty location should not produce "location":{"file":"","line":0,"key":""}
    Message  string   `json:"message"`            // human-readable, single-line, no terminal newline
    Detail   string   `json:"detail,omitempty"`   // optional multi-line elaboration (rule rationale, fix hint)
}
```

**JSON-tag contract**: lowercase keys (`severity`, `code`, `location`, `message`, `detail`, `file`, `line`, `key`); `omitempty` on `Location`, `Detail`, and every `Location` sub-field. This is the canonical wire format; P1.4 reuses these types unchanged. Tags are surfaced at Step A, not retrofit later.

**Rationale**:

- Co-location with `ValidatePacketSizes` keeps related machinery in one file. P1.4's `analyze` will likely import the same module — a separate `internal/validate/` package would force `validation.go` to reach across the boundary anyway.
- A new package would also force exports today that we may want to keep package-private. The single-file approach defers the boundary decision to the moment when it matters (when `analyze` actually exists and we can see what it consumes).
- `Code` is critical for tooling: CI scripts grep `PSC001` reliably; humans rely on `Message`. The convention is `<RULE-FAMILY><NUMBER>`, e.g. `PSC` (packet size collision), `HDR` (header range), `CPS` (CPS tag), `FLD` (required field), `KEY` (unknown key).
- `Location.Line` is best-effort: the bufio.Scanner counts lines; we expose it to the user. For checks that span the whole config (e.g. pairwise S-collision), `Line=0` indicates "config-wide".
- `Detail` (multi-line) lets us attach the "why" of each rule. Crucial for community education (goal § 1.4).

**Out-of-scope**: structured "fix suggestions" attached to findings. Add only if `--fix` returns to scope.

### 4.3 Severity levels — error / warning / info; `--strict` makes warnings fatal

**Decision**:

- **Error**: violation that prevents the config from working. Examples: pairwise S-prefix size collision, H1 overlaps WG type-id, junk range contains 148, missing `PrivateKey`. Exit code 1 (or non-zero if more nuance ever needed).
- **Warning**: violation that does not prevent connectivity but signals risk or deprecation. Examples: unknown INI key, raw `<c>` in any value, S4 outside the `[0..32]` recommended range (still legal, just unusual).
- **Info**: noteworthy but neither broken nor deprecated. Examples: `MTU=1280` is the default (not actually printed; reserved for future use).
- `--strict` flag promotes warnings to errors for the purpose of exit code computation. Findings still report as `warning` in their `Severity` field; only the exit-code formula changes.

**Rationale**:

- Three levels match the tooling industry standard (clippy, golangci-lint, eslint). Mapping is unambiguous.
- `--strict` (warnings-fatal) is the CI-friendly mode. Default mode is permissive so first-time users don't see a red wall.
- Exit code is binary (0 / 1) for v1. A future `--max-severity warning` flag (per roadmap brainstorm) is `--strict` in different syntax — defer the alternate spelling.

### 4.4 Output formats — `text` (default), `json`

**Decision**: support `--output text` (default) and `--output json`. Reject anything else with a clear error message. No `yaml`, no `sarif` in v1 (tracked as future work; sarif would be the natural fit for GitHub-Actions integration in P1.5).

**Text format example** (clean):

```text
amnezigo validate /tmp/awg0.conf
✓ /tmp/awg0.conf: 0 errors, 0 warnings, 0 info
```

**Text format example** (with findings):

```text
amnezigo validate /tmp/server.conf

[ERROR PSC001] /tmp/server.conf: pairwise S-prefix size collision
  S1+148 vs S2+92 at 148 bytes
  rationale: AWG 2.0 receivers classify packets by length; identical
  padded sizes cause init/response misclassification.

[WARNING KEY001] /tmp/server.conf:42 (key=Foobar): unknown INI key
  the parser ignores unknown keys; if this was a typo, fix it.

✗ /tmp/server.conf: 1 error, 1 warning, 0 info
exit 1
```

**JSON format**: a single object with `file`, `summary{errors, warnings, info}`, and `findings[]`. Each finding mirrors the `Finding` struct. Example in § 9.

**Rationale**:

- Text-default keeps interactive use ergonomic; JSON-opt-in keeps tooling-integration robust.
- Single-line `Message` + multi-line `Detail` works in both: text indents `Detail` two spaces; JSON puts `Detail` in its own field.
- We do NOT print findings to stderr — stdout for both formats. Stderr is reserved for runtime errors (file not found, permission denied, malformed flags). This keeps `amnezigo validate ... --output json | jq` working without redirect dance.

### 4.5 Where validation orchestration lives — extend `validation.go`, not a new package

**Decision**: add `ValidateServerConfig(cfg *ServerConfig) []Finding` to `validation.go`. Defer the package extraction until P1.4 demands it.

**Rationale**:

- Current `validation.go` is 147 lines. After this PR it grows to ~300 lines — still well within idiomatic Go-package-file size.
- A new `internal/validate/` package would require exporting `paddedSizes`, `validateHeaderRange` (already promoted to `ValidateHeaderRange` by this PR), `wgInitiationSize`, etc. Crossing the package boundary forces decisions we don't have evidence to make yet.
- If `analyze` (P1.4) needs to compose validation primitives differently (e.g. only run a subset, skip the I-packet check), that's the right time to refactor into a package. Premature now.

### 4.6 CLI library — Cobra (no change)

**Decision**: use `github.com/spf13/cobra`. Already in use throughout `internal/cli/`.

**Rationale**: matches `init.go`, `add.go`, `edit.go`, `list.go`, `export.go`, `remove.go`. No reason to introduce a second framework. `viper` is not in use; do not add it.

### 4.7 Client config support — deferred

**Decision**: v1 of `validate` accepts server configs only. `ParseClientConfig` does not exist (parser comment at `parser.go:149-158` documents the gap). Adding it is a separate effort ~equivalent in size to this PR.

**Rationale**:

- The server format is what `init` writes and what users hand to `add`/`edit`/`export`. It's the format with installed-base.
- Client configs are produced by `export` and consumed by client devices, not by `amnezigo` itself. Round-trip parsing of client configs is desirable but not blocking.
- Surfacing `<c>` in a client config is the most-requested check. We approximate this in v1 by scanning the **raw file bytes** for `<c>` regex during strict-mode parse — independent of whether the parser understands the field. This catches the migration-from-other-generators case without requiring full client-format support.
- A CLI-level `--client` flag is reserved for v2. v1 errors with "client config validation not yet supported; pass a server config" if the file looks like a client config (heuristic: contains `[Peer]\nPublicKey` AND `Endpoint` line — server configs don't have `Endpoint` per peer).

**Tracking**: open follow-up "P1.3-followup: client config support" in the roadmap brainstorm sweep.

### 4.8 Error code stability — semver-style `<FAMILY><NNN>`

**Decision**: each rule has a stable `Code`. Changing a `Code` is a breaking change for downstream tooling. `Message` and `Detail` are free to evolve.

**Initial allocations**:

- `PSC001` — pairwise S-prefix size collision
- `PSC002` — junk range contains a forbidden size (padded or raw WG)
- `PSC003` — I-packet length collides with a padded size (deferred-trigger; unreachable until client-format support)
- `HDR001` — H1-H4 range overlaps WG message type-ids (1..4)
- `HDR002` — H1-H4 range structurally invalid (`Max < Min`)
- `CPS001` — raw `<c>` (counter) tag detected — kernel-only, breaks mobile clients
- `FLD001` — required field missing
- `FLD002` — field has invalid value (out of range, wrong type)
- `KEY001` — unknown INI key
- `JNK001` — junk range structurally invalid (`Jmin > Jmax`)

> **JNK001 double-coverage note**: `JNK001` is intentionally emitted from two paths — directly from `validateJunkRange` (when `Jmin > Jmax` on the parsed config) and indirectly via `findingsFromValidationError` translating `ErrEmptyJunkRange` returned by `ValidatePacketSizes`. Both produce the same code. The orchestrator is responsible for not surfacing duplicates; if both fire on the same config a deduplication step (by `Code`+`Location`) collapses them. This is acceptable: same root cause, same code, single user-visible finding.

Add new codes monotonically (`PSC004`, `HDR003`, ...). Do not reuse retired codes; mark them in a comment so future readers see the gap.

**Then implement** in `validation.go` — append the types listed in § 4.2 plus a small `(Finding) OneLine() string` helper:

```go
// OneLine returns the canonical single-line representation of a finding,
// suitable for CLI text output and log lines. Format:
//   "[<SEVERITY> <CODE>] <file>:<line> (key=<key>): <message>"
// Line and key segments are omitted when empty.
func (f Finding) OneLine() string {
    var locParts []string
    if f.Location.File != "" {
        locParts = append(locParts, f.Location.File)
    }

// ValidateServerConfig runs every validation rule against the parsed config
// and returns all findings. The slice is empty when the config is clean.
// Severity ranking, ordering, and code allocation are documented in
// docs/plans/p1.3-validate-command.md § 4.8.
func ValidateServerConfig(cfg *ServerConfig) []Finding {
    var findings []Finding

    findings = append(findings, validateRequiredFields(cfg)...)
    findings = append(findings, validateSPrefixes(cfg)...)
    findings = append(findings, validateJunkRange(cfg)...)
    findings = append(findings, validateHeaderRanges(cfg)...)

    return findings
}

func validateRequiredFields(cfg *ServerConfig) []Finding {
    var out []Finding
    add := func(key string) {
        out = append(out, Finding{
            Severity: SeverityError,
            Code:     "FLD001",
            Location: Location{Key: key},
            Message:  fmt.Sprintf("required field %q is missing", key),
            Detail:   "server configs require PrivateKey, Address, and ListenPort to function.",
        })
    }
    if cfg.Interface.PrivateKey == "" {
        add("PrivateKey")
    }
    if cfg.Interface.Address == "" {
        add("Address")
    }
    if cfg.Interface.ListenPort == 0 {
        add("ListenPort")
    }
    return out
}

func validateSPrefixes(cfg *ServerConfig) []Finding {
    o := cfg.Obfuscation
    err := ValidatePacketSizes(o.S1, o.S2, o.S3, o.S4, nil, o.Jmin, o.Jmax)
    return findingsFromValidationError(err)
}

func validateJunkRange(cfg *ServerConfig) []Finding {
    o := cfg.Obfuscation
    if o.Jmin > o.Jmax {
        return []Finding{{
            Severity: SeverityError,
            Code:     "JNK001",
            Message:  fmt.Sprintf("junk range Jmin (%d) > Jmax (%d)", o.Jmin, o.Jmax),
        }}
    }
    return nil
}

func validateHeaderRanges(cfg *ServerConfig) []Finding {
    var out []Finding
    ranges := [4]HeaderRange{
        cfg.Obfuscation.H1, cfg.Obfuscation.H2,
        cfg.Obfuscation.H3, cfg.Obfuscation.H4,
    }
    for i, r := range ranges {
        if err := ValidateHeaderRange(r); err != nil {
            code := "HDR001"
            if r.Max < r.Min {
                code = "HDR002"
            }
            out = append(out, Finding{
                Severity: SeverityError,
                Code:     code,
                Location: Location{Key: fmt.Sprintf("H%d", i+1)},
                Message:  err.Error(),
                Detail:   "H1-H4 ranges must avoid WG message type-ids 1..4.",
            })
        }
    }
    return out
}

func findingsFromValidationError(err error) []Finding {
    if err == nil {
        return nil
    }
    var psc *PacketSizeCollisionError
    if errors.As(err, &psc) {
        code := map[string]string{
            "s-pair":     "PSC001",
            "i-packet":   "PSC003",
            "junk-range": "PSC002",
        }[psc.Kind]
        return []Finding{{
            Severity: SeverityError,
            Code:     code,
            Message:  err.Error(),
        }}
    }
    if errors.Is(err, ErrEmptyJunkRange) {
        return []Finding{{
            Severity: SeverityError,
            Code:     "JNK001",
            Message:  err.Error(),
        }}
    }
    return []Finding{{
        Severity: SeverityError,
        Code:     "PSC000",
        Message:  err.Error(),
    }}
}

import (

    "github.com/Arsolitt/amnezigo"

var (
    validateOutputFormat string
    validateStrict       bool
    validateQuiet        bool
)

// exitFn is the OS exit function; tests override it via t.Cleanup
// to capture the exit code without aborting the test binary.
// See § 9 risk row for rationale.
var exitFn = os.Exit

// NewValidateCommand returns a fresh `validate` subcommand. Reads a server
// config, runs every validation rule, prints findings in the requested
// format, and exits non-zero on errors (or warnings under --strict).
func NewValidateCommand() *cobra.Command {
    cmd := &cobra.Command{
        Use:   "validate <config>",
        Short: "Validate an AmneziaWG server config against AWG 2.0 invariants",
        Long: `Validate runs every check the generator enforces (size collisions,
header ranges, required fields, deprecated tags) against an existing config.

Exit code:
  0 — no errors (warnings/info may still be printed)
  1 — at least one error (or any warning when --strict is set)

Examples:
  amnezigo validate /etc/amnezia/awg0.conf
  amnezigo validate awg0.conf --output json
  amnezigo validate awg0.conf --strict --quiet
`,
        Args: cobra.ExactArgs(1),
        RunE: runValidate,
    }
    cmd.Flags().StringVar(&validateOutputFormat, "output", "text", "Output format: text|json")
    cmd.Flags().BoolVar(&validateStrict, "strict", false, "Treat warnings as errors for exit code")
    cmd.Flags().BoolVar(&validateQuiet, "quiet", false, "Suppress summary line; print findings only")
    return cmd
}

func runValidate(cmd *cobra.Command, args []string) error {
    path := args[0]
    f, err := os.Open(path) //nolint:gosec // user-supplied path; intentional
    if err != nil {
        return fmt.Errorf("open %q: %w", path, err)
    }
    defer f.Close()

    cfg, warnings, parseErr := amnezigo.ParseServerConfigWithOptions(
        f, amnezigo.ParseOptions{Strict: true})

    var findings []amnezigo.Finding

    // Convert pre-parse warnings (unknown keys, raw <c>) into findings.
    for _, w := range warnings {
        findings = append(findings, amnezigo.Finding{
            Severity: amnezigo.SeverityWarning,
            Code:     w.Code,
            Location: amnezigo.Location{File: path, Line: w.Line, Key: w.Key},
            Message:  w.Message,
        })
    }

    // If parse aborted (structural error), emit one fatal finding and stop.
    if parseErr != nil {
        findings = append(findings, amnezigo.Finding{
            Severity: amnezigo.SeverityError,
            Code:     "PSE001",
            Location: amnezigo.Location{File: path},
            Message:  parseErr.Error(),
        })
    } else {
        // Run the full validation orchestrator.
        for _, ff := range amnezigo.ValidateServerConfig(&cfg) {
            ff.Location.File = path
            findings = append(findings, ff)
        }
    }

    return emitFindings(cmd.OutOrStdout(), path, findings)
}

func emitFindings(w io.Writer, path string, findings []amnezigo.Finding) error {
    summary := summarize(findings)
    switch validateOutputFormat {
    case "text":
        printText(w, path, findings, summary)
    case "json":
        if err := printJSON(w, path, findings, summary); err != nil {
            return err
        }
    default:
        return fmt.Errorf("unknown --output format %q (want: text|json)", validateOutputFormat)
    }

    failed := summary.Errors > 0
    if validateStrict && summary.Warnings > 0 {
        failed = true
    }
    if failed {
        // Cobra translates a returned error into exit-1; we want exit-1 but
        // not a Cobra error message after our own output. Call exitFn (a
        // package var so tests can override) instead of os.Exit directly.
        exitFn(1)
        return nil // belt-and-suspenders: when exitFn is a test recorder, it returns
    }
    return nil
}

type summary struct {
    Errors   int `json:"errors"`
    Warnings int `json:"warnings"`
    Info     int `json:"info"`
}

func summarize(findings []amnezigo.Finding) summary {
    var s summary
    for _, f := range findings {
        switch f.Severity {
        case amnezigo.SeverityError:
            s.Errors++
        case amnezigo.SeverityWarning:
            s.Warnings++
        case amnezigo.SeverityInfo:
            s.Info++
        }
    }
    return s
}

func printText(w io.Writer, path string, findings []amnezigo.Finding, s summary) {
    for _, f := range findings {
        fmt.Fprintln(w, f.OneLine())
        if f.Detail != "" {
            for _, l := range strings.Split(f.Detail, "\n") {
                fmt.Fprintln(w, "  "+l)
            }
        }
    }
    if !validateQuiet {
        marker := "✓"
        if s.Errors > 0 {
            marker = "✗"
        }
        fmt.Fprintf(w, "%s %s: %d errors, %d warnings, %d info\n",
            marker, path, s.Errors, s.Warnings, s.Info)
    }
}

func printJSON(w io.Writer, path string, findings []amnezigo.Finding, s summary) error {
    type doc struct {
        File     string              `json:"file"`
        Summary  summary             `json:"summary"`
        Findings []amnezigo.Finding  `json:"findings"`
    }
    if findings == nil {
        findings = []amnezigo.Finding{} // never null in JSON
    }
    enc := json.NewEncoder(w)
    enc.SetIndent("", "  ")
    return enc.Encode(doc{File: path, Summary: s, Findings: findings})
}
```

## 7. Validation Checks Catalog

Every check this PR exposes, mapped to source-of-truth and rule code.

| Code | Severity | Rule | Source primitive | Trigger | Detail |
| --- | --- | --- | --- | --- | --- |
| `PSC001` | error | Pairwise S-prefix size collision | `validation.go:76` (`ValidatePacketSizes`) → `Kind="s-pair"` | `paddedSizes(S1..S4)` not all distinct | Delivered in P0.3. Fail-fast: first colliding pair is reported. |
| `PSC002` | error | Junk range contains forbidden size (padded or raw WG) | `ValidatePacketSizes` → `Kind="junk-range"` | `[Jmin..Jmax]` contains any of `{S1+148, S2+92, S3+64, S4+32, 148, 92, 64, 32}` | Delivered in P0.3. |
| `PSC003` | error | I-packet length collides with padded size | `ValidatePacketSizes` → `Kind="i-packet"` | Any `len(I_i) == Sj+Mj` | Delivered in P0.3. Inactive in v1 (server config has no I-packets); reserved for client-config support. Contract pinned by `TestValidateServerConfig_PSC003UnreachableWithNilIPackets` (Step D) — that test fails loudly if a future change makes PSC003 reachable through the server path without an explicit decision. |
| `HDR001` | error | H1-H4 range overlaps WG message type-id (1..4) | `validation.go:138` (`ValidateHeaderRange`) | `r.Min <= 4 && r.Max >= 1` | Delivered in P0.4. Promoted to public by Step B of this plan. |
| `HDR002` | error | H1-H4 range structurally invalid | `ValidateHeaderRange` | `r.Max < r.Min` | Delivered in P0.4. Same primitive, different finding code. |
| `CPS001` | warning | Raw `<c>` (legacy counter) tag detected | strict-parser regex scan (`parser.go`, this PR) | Any line contains `<c>` | Documents migration intent. Generator never emits `<c>` after P0.1; this rule catches hand-edited / migrated configs. |
| `FLD001` | error | Required field missing | `validateRequiredFields` (this PR) | `PrivateKey`, `Address`, or `ListenPort` empty | New in this PR. |
| `FLD002` | error | Field has invalid value | reserved (this PR ships placeholder; first user is P1.4) | n/a in v1 | Reserved code; not currently emitted. |
| `KEY001` | warning | Unknown INI key | strict-parser default arm (this PR) | Key not in known set | Documents typos and forward-incompatible imports. |
| `JNK001` | error | Junk range structurally invalid | `validateJunkRange` (this PR) | `Jmin > Jmax` | Pre-empts the `ErrEmptyJunkRange` sentinel from `ValidatePacketSizes`. |
| `PSE001` | error | Parse aborted (structural) | `runValidate` wrapper (this PR) | `ParseServerConfigWithOptions` returned error | Catch-all for malformed H1-H4 numeric values, malformed sections, scanner errors. |
| `RC001` | n/a | `<rc>` charset compliance | not surfaced in v1 | n/a | Generator-side check delivered in P0.5. Only enforceable on client configs (which carry CPS strings). Deferred with client-config support. |

**Coverage matrix vs. P0.x roadmap deliverables**:

- P0.1 (`<c>` removal) → `CPS001` warning catches legacy.
- P0.2 (`<t>` size 4) → not surfaced. The parser doesn't read CPS strings in v1; this is enforced only at generation time. Acceptable: server-side configs do not carry CPS.
- P0.3 (size collision invariants) → `PSC001/002/003` errors.
- P0.4 (H1-H4 type-id forbidding) → `HDR001/002` errors.
- P0.5 (`<rc>` charset) → not surfaced in v1; deferred.

## 9. Risks and Edge Cases

| Risk | Likelihood | Impact | Mitigation |
| --- | --- | --- | --- |
| Client-config format diverges from server format; users pass a client config and `validate` misreports | Medium | User confusion; false-clean exits | v1 heuristic: if file contains `[Peer]` and `Endpoint=` line (client-only), exit 1 with message "client config validation not yet supported in v1; pass a server config or wait for P1.3-followup". Implement in `runValidate` before opening `ParseServerConfigWithOptions`. |
| User-edited custom fields (e.g. comments with embedded INI-like `Key=value`) trigger spurious `KEY001` warnings | Low | Noise in output | The strict-parser already skips comment lines (`parser.go:28`). User-supplied keys inside `PostUp = ip ... Foo=bar` are values not keys, so they don't trip the unknown-key path. Verified by re-reading `parser.go:43-49`. |
| Partial / truncated config file | Medium | Parse error or false-clean | `bufio.Scanner` returns `scanner.Err()` at end; current code already handles via `parser.go:178`. `PSE001` finding wraps it. |
| Very large CPS strings (multi-KB) in some future client-config support | Low | Performance | Out of scope v1. When client support lands, regex `<c>` scan is O(n) on file bytes; even a 1MB config validates in <10ms. Document as benchmark TODO for P1.4. |
| Round-trip property regression: a generator change produces configs that fail `validate` | Medium | Breaks the contract from § 4.8 | `TestValidateServerConfig_RoundTripGenerated_AllProtocols` in Step D runs 50 generations and asserts zero errors. Include in default `go test ./...`. If a future change breaks it, the test fails immediately. |
| Exit-code semantics conflict with Cobra's "non-nil error → exit 1" default | Low | Double error message | We bypass Cobra's exit by calling `os.Exit(1)` directly after our own output. The `RunE: runValidate` returns nil after `os.Exit` (unreachable). Drawback: `os.Exit` skips deferred cleanup. Acceptable for a read-only command — `f.Close()` failure on a read fd is benign. |
| Symlink config files | Low | None | `os.Open` follows symlinks. Document, but no special handling. |
| Permission-denied on config file | Common | Misleading error | `runValidate` returns `fmt.Errorf("open %q: %w", path, err)` which Cobra surfaces. User sees `Error: open /etc/awg0.conf: permission denied`. Clear enough. |
| `--output yaml` requested by user, currently unsupported | Common | Bad UX | `runValidate` fails with `unknown --output format "yaml" (want: text|json)`. Track P1.3-followup if demand emerges. |
| Findings printed to stdout interleave with summary on slow terminals | Low | Cosmetic | `bytes.Buffer` flushed via single `fmt.Fprintln` per finding line; no interleave because there's no concurrency. |
| `os.Exit(1)` inside `runValidate` would make the function untestable | Resolved at Step E | n/a | Resolved by introducing the package var `exitFn = os.Exit` from the start of Step E. Tests override `exitFn` via `t.Cleanup` to capture the exit code without aborting the test binary. See Step E `TestValidateCommand_NonZeroExitOnErrors` and `TestValidateCommand_StrictPromotesWarnings` for the canonical pattern. |
| Non-deterministic unknown-key ordering between map iteration and slice append | Low | Test flakes | `parser.go` uses a `switch`-based key-lookup, not a map — order is deterministic per source line. Verified. |

## 11. Out of Scope (Explicit)

The following are **deliberately excluded** from this PR:

- **`--fix` auto-fix mode**. Even the trivial `<c>` → `<r 1>` substitution is non-trivial when paired with collision avoidance: removing 8 bytes of payload from a CPS may push the resulting I-packet length into a forbidden size, requiring a re-run of the generator's perturbation logic. Combining "validate" and "rewrite-on-disk" doubles the surface and the test matrix. Tracked as P1.3-followup; only revisit if user demand exists.
- **Cryptographic key validation**. We do not parse Base64 to verify a private key matches its claimed public key. The parser already errors on malformed Base64 fields via decoder errors; that's enough for v1.
- **WireGuard interop / on-the-wire validation**. `validate` does not open a UDP socket, does not probe AWG message exchange, does not inspect `wg show` output. That's `analyze` (P1.4) and beyond.
- **Client-config format support**. See § 4.7 — deferred to P1.3-followup. The roadmap's "(or document client format as out of scope for v1)" lets us defer cleanly.
- **`--max-severity warning` flag spelling**. We expose `--strict` instead; both spellings denote the same thing. Add the alias in P1.5 if CI authors prefer it.
- **YAML or SARIF output formats**. `text` and `json` only. SARIF is the right format for GitHub-Actions integration in P1.5; defer to that PR.
- **Strict-mode flag at the parser-library level**. The strict toggle exists as `ParseOptions.Strict`; we do NOT add a CLI flag to disable strict mode for `validate`. Validation is the entire point — strict is always on. CLI-level strictness on `init`/`add`/`edit` is a separate question handled if and when raised.
- **Config file diff against generator defaults**. "What does this config differ from a freshly-generated one?" is a useful question but not what `validate` answers. Reserved for `analyze` or a hypothetical `diff` subcommand.
- **`info`-severity findings**. The `SeverityInfo` constant exists; no rule emits it in v1. Reserved for P1.4.
- **Subcommand for batch validation** (`amnezigo validate /etc/amnezia/*.conf`). v1 takes exactly one path argument. Shell globbing in user-space works fine; if multi-arg becomes a frequent ask, defer to a follow-up.
- **Refactoring `validation.go` into a package** (per § 4.5).

If any of these come up during implementation, **do not include them**. Note for the next PR.
