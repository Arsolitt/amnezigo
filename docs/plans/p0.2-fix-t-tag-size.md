## 1. Context

`amnezigo` is a Go CLI that generates AmneziaWG configuration files. CPS (custom packet specifier) templates use tags like `<b 0xHEX>`, `<r N>`, `<t>` to describe synthetic packets used for I-packets and junk traffic.

The current generator treats `<t>` (timestamp) as **8 bytes** in size accounting. The reference implementation `amneziawg-go/device/obf_timestamp.go` writes the timestamp as:

```go
binary.BigEndian.PutUint32(buf, uint32(time.Now().Unix()))
```

That is **4 bytes** (`uint32`), not 8. The discrepancy causes:

- Wrong `maxISize` ceiling — generator may produce I-packets shorter than expected, which can collide with handshake sizes and break AWG 2.0 size-collision validation.
- Tests asserting wrong byte counts (e.g. `cps_test.go:154` expects 20 bytes for `<b 0xdeadbeef><c><t>` = 4 + 8 + 8; the `<t>` part should contribute 4).

## 2. Goal

Align `amnezigo`'s CPS size accounting with `amneziawg-go` reference:

- `<t>` contributes **4 bytes** everywhere it is counted.
- `<c>` (counter, kernel-only) keeps its **8 bytes** — its full removal happens in P0.1, not here.
- Add a sanity test that locks the accounting against the actual built CPS bytes.

## 3. Key Finding from Inventory

The constant `bitsPerByte = 8` in `cps.go:18` is **misnamed**. It is used as the per-tag byte size for `<t>` and `<c>`, **not** as a bits-per-byte conversion. The regex `<[tc]>` in `cps.go:168-171` lumps both tags into one match group and multiplies the count by this single constant.

To give `<t>` size 4 while keeping `<c>` at 8 (until P0.1 removes it), we **must split** the regex into two and introduce two named constants.

### Step 3 — Sanity test (Lock-in)

**Reality check during planning revealed there is no on-the-wire CPS-to-bytes builder in `amnezigo`.** `BuildCPSTag` (`cps.go:42-62`) returns the literal template string (e.g. `"<t>"`, `"<r 8>"`, `"<c>"`), NOT actual on-the-wire bytes. The roadmap brainstorm question "compare against actual byte length of a built CPS string" assumes a builder primitive that does not exist. We reframe the sanity test as a **hand-computed expectations table** for `calculateCPSLength`.

**Naming**: `TestCalculateCPSLengthMatchesAccountedSize`.

**Approach**: Table-driven test with hand-derived expected sizes from the AWG 2.0 spec:

- `<t>` → 4 (timestamp, `uint32`)
- `<c>` → 8 (counter, `uint64`)
- `<b 0xHEX>` → `len(HEX) / 2`
- `<r N>` / `<rc N>` / `<rd N>` → `N`

**Test cases** (covers acceptance criteria spot-check):

| Template | Expected length | Derivation |
| --- | --- | --- |
| `<t>` | 4 | timestamp = 4 |
| `<b 0xff>` | 1 | hex `ff` / 2 = 1 |
| `<b 0xff><t>` | 5 | 1 + 4 — **acceptance criterion** |
| `<r 10><t>` | 14 | 10 + 4 |
| `<b 0xdeadbeef><t>` | 8 | 4 + 4 |
| `<rc 5><rd 3><t>` | 12 | 5 + 3 + 4 |

**Do not include `<c>`-only cases** in this new test — `<c>` is being removed in P0.1; cross-validating it now is wasted work.

**Test sketch** (table-driven):

```go
func TestCalculateCPSLengthMatchesAccountedSize(t *testing.T) {
    tests := []struct {
        name     string
        template string
        want     int
    }{
        {"timestamp_only", "<t>", 4},
        {"single_byte", "<b 0xff>", 1},
        {"byte_plus_timestamp", "<b 0xff><t>", 5}, // acceptance criterion
        {"random_plus_timestamp", "<r 10><t>", 14},
        {"long_bytes_plus_timestamp", "<b 0xdeadbeef><t>", 8},
        {"random_charset_digit_plus_timestamp", "<rc 5><rd 3><t>", 12},
    }

    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            got := calculateCPSLength(tt.template)
            if got != tt.want {
                t.Errorf("calculateCPSLength(%q) = %d, want %d", tt.template, got, tt.want)
            }
        })
    }
}
```

**Forward-looking note**: If a CPS-to-bytes builder is added in a future PR (e.g. as part of P1.x), extend this test to also cross-validate `len(builder(template)) == calculateCPSLength(template)`. For now, the hand-computed table is the source of truth.

Verify:

```bash

go test ./...
```

## 7. Constants Naming Decision

**Decision**: introduce `cpsTimestampSize = 4` and `cpsCounterSize = 8` rather than keep a single `bitsPerByte` magic value.

**Rationale:**

- `bitsPerByte = 8` is **semantically wrong** — used as a byte count, not a bit width. Misleading future readers.
- Forward-compat: if AWG ever changes `<t>` to `uint64` again, one constant edit suffices.
- Symmetric with other named constants likely already present in the const block (e.g. `handshakeSize`, `reserve`).
- `cpsCounterSize` keeps `<c>` accounting honest until P0.1 removes it — avoids re-introducing a magic literal `8` in the splitter loop.

## 8. Risks and Edge Cases

| Risk | Assessment |
| --- | --- |
| Other MTU assertions break after fix | **Low.** All MTU comparisons use `<` (not `<=`). After fix, `<t>` is smaller, so I-packets have *more* headroom. Tests stay green; no false positives. |
| `cps_mtu_test.go:31 exact_fit` passes vacuously | **Mitigated.** Plan explicitly updates `maxSize: 29 → 25` to preserve "exact fit" intent. |
| Protocol templates exceed MTU after fix | **Low.** `quic.go`, `dtls.go`, `dns.go`, `stun.go` produce CPS strings <200 bytes; `maxI ≈ 1050` at default MTU 1280. Huge headroom. |
| `<c>` accounting accidentally broken | **Mitigated.** `cpsCounterSize = 8` keeps it explicit until P0.1. |
| Hidden dependency on `bitsPerByte` outside CPS | **Verify during inventory.** If `bitsPerByte` is referenced anywhere else (unlikely from naming), reconsider rename vs. introduce-new-constants strategy. |
