# P0.4 ‚Äî Forbid H1-H4 Ranges Containing Standard WG Type-IDs

> Atomic execution plan for roadmap item **P0.4** (see `docs/roadmap.md`).
>
> **Status**: Ready for implementation
> **Estimated diff**: ~80 LOC across 3 files (+ 1 new test file or test additions)
> **Depends on**: nothing ‚Äî independent of all other P0 items (different code path)
> **Blocks**: P1.3 (`validate` CLI command will reuse the validator helper introduced here)

---

## 1. Context

`amnezigo` generates AmneziaWG configurations. The `H1`-`H4` parameters are **integer ranges** (`Min-Max`, `uint32`) that an AWG-aware peer accepts as substitutes for the four standard WireGuard message type-ids (1 = Initiation, 2 = Response, 3 = Cookie Reply, 4 = Transport).

If any generated `Hk` range **includes** any of `1, 2, 3, 4`, then a vanilla WireGuard packet ‚Äî for example a `type = 1` initiation ‚Äî is accepted by the AWG-aware peer **as if** it were a magic-header packet. That defeats the obfuscation goal: the AWG and vanilla-WG networks are supposed to be **inert to each other**. If they are not, an attacker can probe with vanilla WG, observe a response, and conclude "this endpoint speaks both" ‚Äî a fall-back attack that fingerprints AWG.

The constraint must be enforced both:

- when **generating** ranges (server `init`), and
- when **loading** existing configs (so misconfigured imports are caught ‚Äî this rule lives in P0.4 even though the dedicated `validate` CLI subcommand is P1.3).

## 2. Goal

Guarantee that **no `Hk` range ever contains any of `1, 2, 3, 4`**, anywhere in the project's lifecycle:

- Generator post-condition: every produced `[4]HeaderRange` satisfies the rule. On violation, retry up to `headerMaxAttempts`.
- Loader post-condition: a parsed config that violates the rule produces an `error` from `ParseServerConfig`.
- One shared helper does the check, used by both sites.
- New tests cover boundaries (`0`, `1`, `2`, `3`, `4`, `5`) plus a 10 000-iteration property test.

## 3. Key Findings from Inventory

Inventory was performed against `dev` HEAD (commit `fd9537e`). Findings that change the shape of the plan:

1. **`headerMinValue = uint32(5)`** (`generator.go:15`). The current generator never emits a range starting below `5`, so the `1-4` collision is **already structurally impossible** for *newly generated* ranges. The post-condition is therefore **a defence-in-depth check** for the generator (cheap, catches future bugs if the constant is ever lowered) and **the only line of defence** for parsed configs.
2. **Existing test already asserts `r.Min < 5`** (`generator_test.go:228-230`). It catches the `Min` half of the constraint indirectly via the constant `5`. The new helper makes the rule explicit and re-usable.
3. **`parseHeaderRange`** (`parser.go:184-195`) accepts any `uint32` with no validation ‚Äî silently returns a zero `HeaderRange{}` on a malformed value. A hostile or legacy config can have `H1 = 0-10`, `H1 = 1-1`, etc. Loader-side validation does not exist anywhere yet.
4. **Project error idiom** is `errors.New("‚Ä¶")` (see `helpers.go:78`); no custom error types so far. Stay with that style.
5. **Only `ParseServerConfig` exists** (`parser.go:19`); there is no `ParseClientConfig`. The loader-side check therefore lives inside `ParseServerConfig` for now. (P1.3 will add the broader `validate` command and likely a client parser.)
6. **Forbidden type-ids = exactly `{1, 2, 3, 4}`.** WireGuard `device/noise-protocol.go` defines `MessageInitiationType = 1`, `MessageResponseType = 2`, `MessageCookieReplyType = 3`, `MessageTransportType = 4`. There is no `MessageType = 0` (zero is reserved/unused, but **not** a real packet type ‚Äî vanilla peers do not emit it, so excluding it from H ranges is unnecessary for the security argument). No type-ids exist beyond `4`. **Conclusion**: forbid set is exactly `{1, 2, 3, 4}` ‚Äî no buffer, no expansion. Documented in the helper.
7. **`Hk` ranges are inclusive on both ends.** Confirmed by `parseHeaderRange` (no `+/-1`) and by writer (`writer.go:41-44`, format `Min-Max`). Helper must treat both ends as inclusive.

```go
// validateHeaderRange returns a non-nil error if the range includes any of
// the standard WireGuard message type-ids (1..4). H1-H4 ranges that include
// these values would accept vanilla WireGuard packets, which breaks the
// obfuscation guarantee that AWG and vanilla-WG networks are inert to each
// other.
//
// The check is inclusive on both ends because parser/writer use inclusive
// "Min-Max" notation.
func validateHeaderRange(r HeaderRange) error {
    if r.Max < r.Min {
        return fmt.Errorf("invalid header range: Max (%d) < Min (%d)", r.Max, r.Min)
    }
    if r.Min <= wgMessageTypeMax && r.Max >= wgMessageTypeMin {
        return fmt.Errorf("header range [%d-%d] contains forbidden WG type-id(s) in [%d..%d]",
            r.Min, r.Max, wgMessageTypeMin, wgMessageTypeMax)
    }
    return nil
}
```

> **Implementation note**: `fmt` is not currently imported in `generator.go` (verify). If unimported, add `"fmt"` to the import block. Alternative: use `errors.New` + `strconv` for a stable, alloc-cheap error string ‚Äî but `fmt.Errorf` is idiomatic and the helper runs O(1) per call. Prefer `fmt.Errorf`.

## 7. Decisions

### 7.1 Constraint shape: forbid *any* overlap with `{1, 2, 3, 4}` ‚Äî no buffer

**Decision**: a range `[Min, Max]` is invalid iff `Min <= 4 && Max >= 1`. This catches:

- Single-value ranges that equal a forbidden id: `1-1`, `2-2`, `3-3`, `4-4`.
- Ranges that span forbidden ids: `0-5`, `1-4`, `4-10`, `0-100`.
- Mixed boundary cases: `4-4`, `4-5`.

It does **not** forbid:

- Pure-zero range `0-0` (no real WG type-id at 0; range can't matter for the security argument). Caught only if separately rejected as "empty range" ‚Äî explicitly out of scope here.
- A buffered exclusion like "0-9 entirely". Buffers over-constrain without security justification; an attacker probing with vanilla WG sends type values exactly in `{1, 2, 3, 4}` ‚Äî values outside that set are not legitimate vanilla WG packets, so accepting `5` or `6` does not enable a fall-back attack. **Reject buffer expansion** per the brainstorm question's recommended answer.

### 7.2 Forbidden set: exactly `{1, 2, 3, 4}`

**Decision**: confirm via `amneziawg-go/device/noise-protocol.go` (referenced in roadmap). The four message-type constants are `MessageInitiationType = 1`, `MessageResponseType = 2`, `MessageCookieReplyType = 3`, `MessageTransportType = 4`. **No other type-ids exist** in the public reference. **Do not forbid `0`** (not a real packet type; only an unused sentinel ‚Äî vanilla peers do not emit it).

If a future AWG version introduces a `MessageTypeKeepalive = 5` (etc.), the constants `wgMessageTypeMin`/`wgMessageTypeMax` are the single edit point. Document this in the helper godoc.

### 7.3 Helper signature: `validateHeaderRange(r HeaderRange) error`

**Decision**:

- Accept a value, not a pointer (cheap copy of two `uint32` values; no mutation needed; idiomatic Go).
- Return `error`, not `bool`. Errors carry context (range bounds, which type-ids collide). The loader path needs to bubble the message; a `bool` would force the loader to re-format. Errors also play well with `errors.Is`/`fmt.Errorf("%w", ‚Ä¶)` if P1.3 wants to layer.
- **Unexported** (`validateHeaderRange`, lowercase). Promotes to `ValidateHeaderRange` only when P1.3's `validate` command needs it from outside the package ‚Äî defer that decision.

### 7.4 Helper location: `generator.go` (now), migrate to `validation.go` later

**Decision**: place the helper at the bottom of `generator.go` for this PR. P1.3 will create `validation.go` to consolidate generator rules + parser rules; at that point this helper moves with no behaviour change. Avoid a one-helper-per-file proliferation today.

### 7.5 Pairwise non-overlap stays as-is

The roadmap mentions "plus pairwise non-overlap of H1..H4 (already enforced ‚Äî verify)". Verified: the existing loop in `generator.go:152-159` enforces strict non-overlap between sorted ranges (uses `>=` so even adjacent ranges that share a single boundary value are rejected ‚Äî over-strict by 1, but that's preexisting behaviour and intentional). **No change needed in this PR**; document the verification in the PR description.

### 7.6 Loader fixture audit

The CLI test fixtures use `H1 = 100` (single value, not a range). `parseHeaderRange("100")` returns `{0, 0}` today. After this PR, that zero range passes `validateHeaderRange` (no overlap with 1..4 by the inclusive bounds check `Min <= 4 && Max >= 1`: `0 <= 4 && 0 >= 1` ‚Üí `false`, so `0-0` is not rejected by the WG-id rule; the optional `Max < Min` check also passes). **Net effect**: no fixture breakage from the WG-id rule alone.

But `validateHeaderRange` adds the `Max < Min` guard. For `{0, 0}`, `Max < Min` is `0 < 0` ‚Üí false, so still passes. Safe.

**Net audit outcome**: existing fixtures continue to parse without error after this PR. Confirm during implementation by running `go test ./internal/cli/...` after Step 2d and reporting any failures.
