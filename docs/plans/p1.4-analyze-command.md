## 1. Context

`amnezigo` is a Go CLI at `/home/arsolitt/projects/amnezigo` that generates AmneziaWG v2.0 server and client configs. After Phase 0 fixes, the generator is provably correct (collision-free padded handshakes, lawful header ranges, accurate CPS length accounting). Phase 1 shifts focus from "is it correct?" to "is it useful?" — and the first useful thing a user wants is *visibility*.

Two complementary tools sit at this layer:

| Tool | Question it answers | Exit semantics |
| --- | --- | --- |
| `validate <config>` (P1.3) | "Is this config correct under AWG 2.0 invariants?" | Non-zero on any violation |
| `analyze <config>` (P1.4 — this plan) | "What does this config look like on the wire?" | Always zero — informational |

Today, neither tool exists as a CLI surface. The library functions that *implement* the analysis (`paddedSizes`, `calculateCPSLength`, `calculateMaxISize`) are all present and correctly wired into the generator post-Phase 0, but a user holding a `awg0.conf` has no way to ask "what packet sizes will this produce?" without manually computing `S1+148`, decoding the `<r N>` tags inside `Jc`/`I_i`, and cross-referencing source. The only existing inspection surface is `amnezigo list`, which dumps peer names — nothing about obfuscation parameters.

The motivating use cases are concrete:

1. **DPI tuning** — a user testing against a specific censor wants to see whether their junk range (e.g. `[64..1024]`) overlaps any DPI-classified probe size. They need to *see* the range, the I-packet sizes, and the padded handshake totals in one place.
2. **Fingerprint risk inspection** — a user choosing between QUIC, DNS, DTLS, STUN templates wants to compare what the resulting on-the-wire byte distribution looks like before deploying to clients. Re-running `init` until the numbers "look random" is the current workaround.
3. **Pre-deployment review** — an operator handing a config to a third party wants a one-page summary of "here is what this peer's traffic shape looks like" without exposing the private key.
4. **Debugging classification mismatches** — when a peer connects but transport packets are misclassified, the first diagnostic step is "what's the actual padded transport size, and does it overlap anything?"

`validate` answers (4) by failing loudly on collisions; `analyze` answers (1)-(3) by *describing* the wire profile without judgement.

## 2. Goal

Ship `amnezigo analyze <config>` as a new Cobra subcommand under `internal/cli/analyze.go`, backed by an `analysis.go` orchestrator in the package root. Concrete deliverables:

- New CLI subcommand:
  - `amnezigo analyze --config awg0.conf` — default snapshot mode, human-readable text output covering padded handshakes, junk range, per-peer I-packet sizes, on-the-wire ordering, and a "fingerprint risk" findings section.
  - `amnezigo analyze --config awg0.conf --peer NAME` — narrow output to one peer's I-packets.
  - `amnezigo analyze --config awg0.conf --output json` — machine-parsable form for tooling.
  - `amnezigo analyze --config awg0.conf --samples N` — distribution mode: regenerate I-packets `N` times and report mean/median/min/max per slot.
- Public `Analyze(cfg ServerConfig, peerName string, opts AnalyzeOptions) (*AnalysisReport, error)` function in `analysis.go`, importable by future callers (P1.5 `--compare`, automated test harnesses).
- `AnalysisReport` struct with `Handshakes`, `Junk`, `Peers`, `Ordering`, `Risks` sections — designed to serialize cleanly to both text and JSON.
- Heuristic risk scanner producing structured findings (severity, code, message) without altering exit code.
- Snapshot integration test asserting that a fixed config produces stable output (with one source of nondeterminism stubbed).
- Unit tests for each formatter, the heuristics, and the distribution math.

Out of scope for **this PR** (deferred or excluded outright — see § 12):

- `--compare <preset>` mode against a reference profile (depends on P1.5 preset library).
- Graphical plotting beyond ASCII-text bar charts.
- Live packet-capture verification (`tcpdump`, `pcap`) — analysis is purely arithmetic from the config.
- Machine-learning fingerprint scoring.
- Modifications to `validate` (P1.3) — `analyze` *reuses* its parsing infrastructure but does not extend it.

## 3. Goal — Explicit Scope Boundary

This subcommand is purely *descriptive*. It MUST NOT:

- Mutate the config file.
- Regenerate any field of the on-disk config (sample mode regenerates I-packets *in-memory only* for distribution analysis; the file is never touched).
- Exit non-zero on any finding (collisions, risks, low entropy).
- Prompt the user for input.
- Perform network I/O (no remote endpoint resolution, no DNS lookup; if the config lacks an endpoint, the report omits that field rather than fetching it).

Violating any of these turns the command from "describe" into "act", which is the responsibility of `init`, `add`, `edit`, or `validate` — separate concerns.

## 5. Architecture Decisions

Each decision documents the alternatives weighed and the chosen path.

### 5.0 Reuse `Finding`/`Severity`/`Location` from P1.3 — DO NOT redefine

**Decision**: Reuse the `Finding`, `Severity`, and `Location` types from P1.3 (`validation.go`). Do NOT redefine them. Analysis observations are `Finding` instances with code namespace `RISK###` (e.g., `RISK001` for junk-range overlap with WG sizes).

**Canonical types** (declared in P1.3 § 4.2, package `amnezigo`):

```go
type Severity string

const (
    SeverityError   Severity = "error"
    SeverityWarning Severity = "warning"
    SeverityInfo    Severity = "info"
)

type Location struct {
    File string `json:"file,omitempty"`
    Line int    `json:"line,omitempty"`
    Key  string `json:"key,omitempty"`
}

type Finding struct {
    Severity Severity `json:"severity"`
    Code     string   `json:"code"`
    Location Location `json:"location,omitempty"`
    Message  string   `json:"message"`
    Detail   string   `json:"detail,omitempty"`
}
```

**Rationale**:

- Same package (`amnezigo`), same name, two shapes won't compile. Avoiding redefinition is mandatory, not stylistic.
- P1.3 owns these types as part of its public surface; P1.4 is a *consumer*.
- `analyze`'s heuristics emit `Finding` with `Code: "RISK001"` ... `RISK009`, allowing future tooling to filter findings by namespace (`validate` uses `VAL###`, `analyze` uses `RISK###`).

**Severity downgrade in `analyze`**: `analyze` never exits non-zero. Where `validate` would emit `SeverityError`, `analyze` emits `SeverityWarning` or `SeverityInfo`. The `Finding` struct itself is unchanged — only the severity *value* differs by callsite.

**If P1.3 is absent**: P1.4 inlines the exact canonical declarations above (see § 4.4). No drift.

### 5.1 Placement — `analysis.go` at the package root, NOT `internal/analyze/`

**Decision**: new file `analysis.go` in the package root (alongside `validation.go`, `generator.go`, `cps.go`).

**Alternatives considered**:

- `internal/analyze/analyze.go` — distinct subpackage, shorter file.
- Inside `validation.go` — co-locate with the related invariants.

**Rationale**:

- Project convention (`validation.go`, `generator.go`, `cps.go`, `parser.go`, `writer.go`, `manager.go`) is one-file-per-responsibility *at the root*. `internal/cli/` is the only subpackage today, scoped strictly to Cobra commands. Introducing `internal/analyze/` for ~250 LOC violates the established layout.
- Keeping `analysis.go` at root means it can use unexported helpers (`paddedSizes`, `calculateCPSLength`, `calculateMaxISize`, `wgInitiationSize`) without forcing the package to widen its export surface. Widening for a single internal consumer is an anti-pattern.
- `validation.go` is already focused on AWG 2.0 invariants (pass/fail). Mixing in descriptive analysis dilutes that focus and would push the file past 400 LOC; analysis is naturally a peer file, not an addition.

**Implication**: `Analyze`, `AnalysisReport`, and all formatters live in `package amnezigo`. The CLI wrapper in `internal/cli/analyze.go` imports them by name.

### 5.2 Output format — text by default, JSON via `--output json`

**Decision**: two formats. Text is the default (human reading); JSON is opt-in (tooling/scripting).

**Alternatives considered**:

- YAML output as a third format.
- `--format` flag instead of `--output` (matches `kubectl --output yaml` precedent).

**Rationale**:

- YAML adds a dependency (`gopkg.in/yaml.v3` is not currently in `go.mod`). JSON is in the standard library. Two formats covers >95% of use cases.
- `--output` matches `kubectl`/`gh`/`git` precedent — users will reach for it without thinking. `--format` is shorter but less idiomatic.

**Validation**: reject `--output` values not in `{text, json}` with an explicit error listing supported values.

### 5.3 Per-peer selection — `--peer NAME` flag, default to all peers

**Decision**: `--peer NAME` filters output to one peer's I-packets and risk findings. Default (no flag) emits a section per peer, in declaration order from the config file.

**Alternatives considered**:

- Default to first peer only (terser; matches roadmap draft "default = first peer in config").
- Require `--peer` always (no default).

**Rationale**:

- The roadmap says "default = first peer in config", but inspection of `parser.go:185-194` shows that header-range validation is server-wide, not per-peer. Most of the report is server-level (handshake sizes, junk range, ordering, H-ranges). Only the I-packet section is per-peer. Showing one peer by default hides the multi-peer reality.
- "All peers" matches `list` (which shows every peer). Consistency with the existing CLI shape.
- Users can always narrow with `--peer NAME` if a config has 50 peers and the output is overwhelming.

**Edge case**: a config with zero peers. The output omits the per-peer section and emits a single info-level finding `"no peers configured; I-packet analysis unavailable"`. Server-level sections (handshakes, junk, H-ranges, ordering) still render.

### 5.4 I-packets — generate fresh, not parse from disk

**Decision**: `analyze` regenerates I-packets via `GenerateCPS(protocol, mtu, s1, 0)` rather than reading them from the input config.

**Justification (forced by parser limitation)**:

- `parser.go:149-159` shows I1-I5 are *deliberately* commented-out in `ParseServerConfig` because they live only in client-side configs, not server configs. The server config (which is the canonical input to `init`/`add`/`list`/`export`/`analyze`) has no I-packets to extract.
- Therefore: `analyze <server.conf>` must invoke the generator with the same parameters `export` would use (`MTU`, `S1`, protocol). This makes "snapshot mode" a *re-derivation* of what `export` would produce, not a read-back of stored bytes.
- Add a `--protocol PROTO` flag (default `random`, matching `export`'s default at `internal/cli/export.go:38`) so the user can ask "what would I-packets look like if I exported this peer with QUIC/DNS/DTLS/STUN?"

**Implication for snapshot tests**: I-packet generation uses an injectable `io.Reader` (defaults to `crypto/rand.Reader`). The `analyze` command exposes a `--seed N` flag that creates a deterministic `math/rand`-backed `io.Reader` for snapshot-test reproducibility. **Do NOT mutate the package-global `crypto/rand.Reader`** — that approach is racy under parallel tests and pollutes other tests in the same binary. **§ 7 Step F documents the seeded-reader approach.**

**Fresh-sample semantic — must be surfaced to users**: the I-packet bytes shown in any `analyze` output are a freshly generated SAMPLE, not the bytes that will be transmitted to a specific peer at runtime. Each `amnezigo export` produces new I-packets for that peer; `analyze` shows what one such generation would look like, given the obfuscation parameters in the config. This must be documented in `cobra.Command.Long` (§ 7 Step E) and as a footer line in every output sample (§ 8). Without this, users will assume `analyze` reads bytes from disk.

### 5.5 Snapshot vs distribution — snapshot is default, `--samples N` is opt-in

**Decision**: default mode runs the generator once and reports concrete numbers (a single instantiation). `--samples N` regenerates `N` times and reports mean/median/min/max per I-packet slot.

**Alternatives considered**:

- Default to `--samples 100` for "more honest" reporting.
- Make distribution mode the only mode.

**Rationale**:

- Snapshot output is reproducible *given a seed* — testable as a fixed string. Distribution output is statistical — fundamentally noisier in tests.
- Most users run `analyze` once to "see what it looks like". They want a number, not a distribution. Power users debugging entropy claims want `--samples 100`.
- `N=1` snapshot = `--samples 1` mathematically. The two modes share the iteration loop; snapshot is just the `N=1` special case formatted differently.

**Validation**: `--samples` must be `≥ 1`. Cap at `1000` to bound runtime. Reject negative or zero with explicit error.

### 5.6 Fingerprint risk findings — informational, never error

**Decision**: heuristics emit structured findings with `Severity ∈ {INFO, WARN}`. No `ERROR` level. Process exit is always `0`.

**Rationale**:

- Anything `ERROR`-grade belongs in `validate` (P1.3), where it surfaces as non-zero exit. `analyze` describes; it does not gate.
- A `WARN`-level finding ("junk range overlaps within ±10 of WG init size — DPI-prone") is a *recommendation*, not a violation. The user may still legitimately ship the config — e.g. testing against a censor that ignores ±10.
- Mixing pass/fail and advice is confusing. `validate` exits non-zero if a hard invariant fails; `analyze` exits zero with text-tagged findings the user can `grep WARN` against if they want to script.

**Severity definitions used in tables below**:

- `INFO` — observation worth surfacing but not necessarily a problem.
- `WARN` — likely fingerprint-detectable shape; recommend regeneration or manual override.

### 5.7 Output struct shape — section-per-concern, no flat fields

**Decision**: `AnalysisReport` groups by concern (`Handshakes`, `Junk`, `Peers`, `Ordering`, `Risks`). Each section is a struct; the top-level type is a struct of structs.

**Rationale**:

- JSON output stays predictable for tooling (each section has a stable schema; users can `jq '.handshakes.init'` reliably).
- Adding a new section in the future (P1.5 `Comparison`, P2.x `EntropyScore`) is a struct field, not a flat-field rename — backward compatible.

**Schema** (proposed):

```go
type AnalysisReport struct {
    Config      ConfigInfo        `json:"config"`
    Handshakes  HandshakeProfile  `json:"handshakes"`
    Junk        JunkProfile       `json:"junk"`
    Headers     HeaderProfile     `json:"headers"`
    Peers       []PeerProfile     `json:"peers"`
    Ordering    OrderingDescription `json:"ordering"`
    Risks       []Finding         `json:"risks"` // Finding type from validation.go (P1.3); see § 5.0
}
```

`HeaderProfile` is one field per header range, four separate field declarations:

```go
type HeaderProfile struct {
    H1 HeaderRangeView `json:"h1"`
    H2 HeaderRangeView `json:"h2"`
    H3 HeaderRangeView `json:"h3"`
    H4 HeaderRangeView `json:"h4"`
}
```

### 5.8 Reproducibility — `--seed N` flag with injected `io.Reader`

**Decision**: `analyze` accepts a `--seed N` integer flag. When non-zero, the analysis context constructs a deterministic `io.Reader` backed by `math/rand` (seeded from `N`) and injects it into the I-packet generator. When zero (default), the generator uses `crypto/rand.Reader`.

**Rationale**:

- Snapshot tests need reproducibility; mutating the package-global `crypto/rand.Reader` is racy under `t.Parallel()` and pollutes neighbouring tests in the same binary. Injection avoids both problems.
- `--seed` is opt-in; production `analyze` runs unchanged use `crypto/rand`.
- `math/rand` is acceptable here because the seeded reader is used ONLY for `analyze`'s descriptive output (no key material is ever derived from it). Production `init`/`add`/`export` continue to use `crypto/rand` exclusively.
- `--samples N` distribution mode benefits from the same injection: with `--seed`, distribution output is reproducible across runs; without it, distribution remains statistical/noisy by design.

**Wiring** (see § 7 Step A and Step E):

- `AnalyzeOptions` gains `Rand io.Reader` (default: `crypto/rand.Reader`).
- The CLI handler parses `--seed N` and, when non-zero, constructs a `math/rand`-backed `io.Reader` adapter.
- The generator and `peerSnapshot` / `peerDistrib` accept the reader as a parameter.

**Documentation**: `--seed` flag help text and `cobra.Command.Long` both note the seed is for reproducibility, not security. Footer note in every output sample reminds users that `--seed` values appearing in CI logs are NOT secrets, but a seed combined with the obfuscation config does deterministically reveal one possible I-packet shape — users should not paste real configs to public forums alongside the seed.

### Step A — Design `AnalysisReport` struct + tests for serialisation (TDD red)

Create `analysis.go` with the type definitions only (no method bodies yet). Create `analysis_test.go` asserting JSON serialisation is stable for a hand-built `AnalysisReport`.

**Type reuse note**: this file uses `Finding`, `Severity`, and `Location` from `validation.go` (per P1.3 § 4.2). Do NOT redefine them here. Analysis introduces no new model types beyond `AnalysisReport` and its sub-structs. If P1.3 has not landed yet, see § 4.4 for inlining the canonical declarations.

**`analysis.go` skeleton:**

```go
package amnezigo

import (
    cryptorand "crypto/rand"
    "io"
)

// AnalyzeOptions configures Analyze. Zero value is valid: snapshot mode,
// random protocol, all peers, no samples, crypto/rand for I-packet bytes.
type AnalyzeOptions struct {
    Protocol string    // CPS protocol; "" defaults to "random".
    PeerName string    // if non-empty, only this peer's I-packets are analyzed.
    Samples  int       // 0 or 1 = snapshot; >1 = distribution mode.
    Rand     io.Reader // RNG source for I-packet generation; nil = crypto/rand.Reader (production default).
}

// resolveRand returns opts.Rand or crypto/rand.Reader if nil. Used by Analyze
// before passing into peerSnapshot / peerDistrib. The CLI handler constructs
// a math/rand-backed io.Reader when --seed is non-zero (see § 7 Step E).
func resolveRand(r io.Reader) io.Reader {
    if r == nil {
        return cryptorand.Reader
    }
    return r
}

// AnalysisReport is the on-the-wire profile of a server config. All fields are
// populated by Analyze; consumers should not construct this directly.
type AnalysisReport struct {
    Config     ConfigInfo          `json:"config"`
    Handshakes HandshakeProfile    `json:"handshakes"`
    Junk       JunkProfile         `json:"junk"`
    Headers    HeaderProfile       `json:"headers"`
    Peers      []PeerProfile       `json:"peers"`
    Ordering   OrderingDescription `json:"ordering"`
    Risks      []Finding           `json:"risks"`
}

type ConfigInfo struct {
```

```go
func Analyze(cfg ServerConfig, opts AnalyzeOptions) (*AnalysisReport, error) {
    if opts.Protocol == "" {
        opts.Protocol = "random"
    }
    if opts.Samples < 0 {
        return nil, fmt.Errorf("samples must be >= 0, got %d", opts.Samples)
    }
    if opts.Samples > 1000 {
        return nil, fmt.Errorf("samples must be <= 1000, got %d", opts.Samples)
    }

    obf := cfg.Obfuscation
    padded := paddedSizes(obf.S1, obf.S2, obf.S3, obf.S4)

    report := &AnalysisReport{
        Config: ConfigInfo{
            MTU:        cfg.Interface.MTU,
            ListenPort: cfg.Interface.ListenPort,
            PeerCount:  len(cfg.Peers),
            Protocol:   opts.Protocol,
        },
        Handshakes: HandshakeProfile{
            Init:      buildPaddedPacket(obf.S1, wgInitiationSize, "S1+148"),
            Response:  buildPaddedPacket(obf.S2, wgResponseSize, "S2+92"),
            Cookie:    buildPaddedPacket(obf.S3, wgCookieReplySize, "S3+64"),
            Transport: buildPaddedPacket(obf.S4, wgTransportSize, "S4+32"),
        },
        Junk: JunkProfile{
            Count: obf.Jc, Min: obf.Jmin, Max: obf.Jmax, Width: obf.Jmax - obf.Jmin,
        },
        Headers: HeaderProfile{
            H1: viewHeader(obf.H1), H2: viewHeader(obf.H2),
            H3: viewHeader(obf.H3), H4: viewHeader(obf.H4),
        },
        Ordering: OrderingDescription{
            Description: "Per handshake: I1→I2→I3→I4→I5, then Jc junk packets, then Handshake Init.",
            Sequence:    []string{"I1", "I2", "I3", "I4", "I5", "junk×Jc", "handshake_init"},
        },
    }

    peerProfiles, err := analyzePeers(cfg, opts, padded)
    if err != nil {
        return nil, err
    }
    report.Peers = peerProfiles
    report.Risks = scanRisks(cfg, padded, peerProfiles)
    return report, nil
}
```

Helpers (`buildPaddedPacket`, `viewHeader`, `analyzePeers`, `scanRisks`) live in the same file.

**`analyzePeers` core loop** (snapshot or distribution depending on `opts.Samples`):

```go
func analyzePeers(cfg ServerConfig, opts AnalyzeOptions, _ [4]int) ([]PeerProfile, error) {
    peers := cfg.Peers
    if opts.PeerName != "" {
        peers = filterByName(peers, opts.PeerName)
        if len(peers) == 0 {
            return nil, fmt.Errorf("peer %q not found in config", opts.PeerName)
        }
    }
    out := make([]PeerProfile, 0, len(peers))
    for _, p := range peers {
        prof := PeerProfile{Name: p.Name, AllowedIP: p.AllowedIPs}
        if opts.Samples <= 1 {
            prof.Snapshot = peerSnapshot(cfg, opts.Protocol)
        } else {
            prof.Distrib = peerDistrib(cfg, opts.Protocol, opts.Samples)
        }
        out = append(out, prof)
    }
    return out, nil
}
```

**`peerSnapshot`** calls `GenerateCPS(opts.Protocol, cfg.Interface.MTU, cfg.Obfuscation.S1, 0)` once and computes lengths via `calculateCPSLength`.

**`peerDistrib`** calls `GenerateCPS` `opts.Samples` times, builds five `[]int` slices (one per slot), computes `Stats` per slot via a small `statsOf` helper.

```go
func runAnalyze(_ *cobra.Command, _ []string) error {
    if analyzeOutput != "text" && analyzeOutput != "json" {
        return fmt.Errorf("invalid --output %q: supported values are text, json", analyzeOutput)
    }
    if analyzeSamples < 1 {
        return fmt.Errorf("--samples must be >= 1, got %d", analyzeSamples)
    }

    mgr := amnezigo.NewManager(cfgFile)
    cfg, err := mgr.Load()
    if err != nil {
        return fmt.Errorf("failed to load server config: %w", err)
    }

    var rng io.Reader // nil = crypto/rand.Reader inside Analyze (default).
    if analyzeSeed != 0 {
        // math/rand-backed reader: deterministic, NOT cryptographically secure.
        // Used only for analyze's descriptive output. Never used by init/add/export.
        rng = newSeededReader(analyzeSeed)

        Protocol: analyzeProtocol,
        PeerName: analyzePeer,
        Samples:  analyzeSamples,
        Rand:     rng,
    })
    if err != nil {
        return err
    }

    switch analyzeOutput {
    case "json":
        return report.FormatJSON(os.Stdout)
    default:
        return report.FormatText(os.Stdout)
    }
}
```

### Step F — Snapshot integration test

The hardest part: making "regenerate I-packets" deterministic for testing.

**Approach**: pass an explicit seeded `io.Reader` via `AnalyzeOptions.Rand`. Do NOT mutate the package-global `crypto/rand.Reader` — that pattern is racy under `t.Parallel()` and pollutes other tests in the same binary.

**Seeded-reader helper** (in `internal/cli/analyze_test.go` or a small `testutil` shared file):

```go
type mathRandReader struct {
    r *mathrand.Rand
}

func (mr *mathRandReader) Read(p []byte) (int, error) {
    for i := range p {
        p[i] = byte(mr.r.Intn(256))
    }
    return len(p), nil
}

func newSeededReader(seed int64) io.Reader {
    return &mathRandReader{r: mathrand.New(mathrand.NewSource(seed))}
}
```

**Test fixture**: `testdata/analyze_snapshot/awg0.conf` — a hand-edited config with known fixed S1-S4, Jc, Jmin, Jmax, H1-H4. Created by running `amnezigo init` once and committing the result.

**Test** (parallel-safe):

```go
func TestAnalyze_SnapshotIntegration(t *testing.T) {
    t.Parallel() // Safe: no global state mutation.
    if testing.Short() { t.Skip("integration test") }

    cfg, err := amnezigo.LoadServerConfig("testdata/analyze_snapshot/awg0.conf")
    if err != nil { t.Fatal(err) }

    report, err := amnezigo.Analyze(cfg, amnezigo.AnalyzeOptions{
        Protocol: "quic",
        Rand:     newSeededReader(42), // Deterministic; stable across runs/processes.
    })
    if err != nil { t.Fatal(err) }

    var buf bytes.Buffer
    if err := report.FormatText(&buf); err != nil { t.Fatal(err) }

    golden, err := os.ReadFile("testdata/analyze_snapshot/expected.txt")
    if err != nil { t.Fatal(err) }
    if !bytes.Equal(buf.Bytes(), golden) {
        t.Errorf("snapshot mismatch\n--- got ---\n%s\n--- want ---\n%s", buf.String(), golden)
    }
}
```

**Why this is safer than reassigning `crypto/rand.Reader`**:

- No package-global mutation → safe under `t.Parallel()` and across test binaries.
- No interaction with other tests in the same binary that legitimately use `crypto/rand`.
- The `Rand` field is a documented part of `AnalyzeOptions`; tests use the same public API as production.
- The seeded reader produces an unbounded byte stream, so over-provisioning concerns disappear.

Update protocol: re-run with `-update` flag, regenerate `expected.txt`. Standard Go golden-file pattern.

> **Note on the risks section in distribution mode**: heuristics that depend on a specific I-packet length (e.g. "I3 too close to padded transport") become statistical claims in `--samples N` mode. Decision: in distribution mode, the risk scanner runs against the *first* sample only, with a footer line "Risks computed from sample 1; variance shown above." Implementing a full "X% of samples violate heuristic Y" is over-scoping and would push PR2 past 800 LOC. Defer that to a future enhancement; document the limitation.

## 9. Fingerprint Risk Heuristics

Each heuristic is a pure function `(*AnalysisReport) → []Finding`. Independent; order of registration determines order of output. Each heuristic emits `Finding` instances with `Severity` from the canonical `Severity` enum (P1.3) and `Code` in the `RISK###` namespace (assigned table below).

**`RISK###` code allocation** (stable contract — never re-number):

| `Code` | Identifier | Severity | Trigger | Message template |
| --- | --- | --- | --- | --- |
| `RISK001` | `junk-near-wg-size` | `SeverityWarning` | Junk range `[Jmin..Jmax]` falls within ±10 bytes of any padded handshake size *or* any raw WG size (148/92/64/32) | `Jmin=%d is within ±10 of %s=%d. Recommendation: regenerate junk range or move Jmin up by ≥ %d.` |
| `RISK002` | `i-packet-cluster` | `SeverityWarning` | The five I-packet sizes fit in a window ≤ 20 bytes wide (low entropy on the wire — looks like a fixed-size fingerprint) | `I-packet sizes %v span only %d bytes; consider regenerating with --protocol random or a wider template.` |
| `RISK003` | `s4-low-padding` | `SeverityInfo` | `S4 < 8` (very small transport padding leaves transport packets near vanilla WG size) | `S4=%d is low-entropy transport padding; consider S4 ∈ [20..32].` |
| `RISK004` | `s-pair-tight` | `SeverityInfo` | Any two padded sizes differ by ≤ 4 bytes (legal but distinguishable by adversary statistics) | `Padded sizes %s=%d and %s=%d differ by only %d bytes — fingerprintable cluster.` |
| `RISK005` | `wg-shape-padding` | `SeverityInfo` | A padded size lands within ±4 of *another* WG raw size (e.g. `S1+148=240` is close to a vanilla MTU-related shape) | `Padded %s=%d is within ±4 of WG %s=%d.` |
| `RISK006` | `junk-width-zero` | `SeverityWarning` | `Jmax - Jmin < 32` (very narrow range — small entropy) | `Junk range width is %d bytes; AWG recommends ≥ 32 for shape variability.` |
| `RISK007` | `headers-narrow` | `SeverityInfo` | Any H_i range width < 1_000_000 (legal post-Phase-0 but small relative to default) | `H%d range width %d is below the default minimum 10000000; verify intentional.` |
| `RISK008` | `no-peers` | `SeverityInfo` | `len(cfg.Peers) == 0` | `No peers configured; I-packet analysis unavailable.` |
| `RISK009` | `vanilla-wg-config` | `SeverityWarning` | `S1==S2==S3==S4==0 && Jc==0 && Jmin==Jmax` (config has no obfuscation parameters set — looks like vanilla WireGuard) | `Config has no obfuscation parameters; this is vanilla WireGuard, not AWG 2.0.` |

The `Identifier` column above is the human-friendly slug (used in test names, log greps, and documentation); the wire-stable contract is the `Code` value (`RISK001`...`RISK009`). The slug appears in `Finding.Detail` for context but consumers should filter on `Code`.

**Heuristic registration** — a slice of function values, looped in `scanRisks`:

```go
type heuristic func(cfg ServerConfig, report *AnalysisReport) []Finding

var heuristics = []heuristic{
    heurJunkNearWG,
    heurIPacketCluster,
    heurS4LowPadding,
    heurSPairTight,
    heurWGShapePadding,
    heurJunkWidthZero,
    heurHeadersNarrow,
    heurNoPeers,
    heurVanillaWG,
}

func scanRisks(cfg ServerConfig, _ [4]int, _ []PeerProfile) []Finding {
    var all []Finding
    for _, h := range heuristics {
        all = append(all, h(cfg, nil)...) // or pass report; keep the API uniform
    }
    return all
}
```

**Tests** — one table-driven test per heuristic, plus an integration test asserting all heuristics combined produce the expected set on a synthetic config.

## 10. Risks and Edge Cases

| Risk | Assessment / Mitigation |
| --- | --- |
| Config has zero peers (just-`init`-ed, no `add` yet) | Emit `no-peers` finding (INFO). Server-level sections still render. Verified by `TestFormatText_NoPeers`. |
| Config has 100+ peers (large fleet) | Default output gets long. Add a hint in `--help`: "for large fleets, use --peer NAME to narrow output". No truncation by default — users can pipe to `less` or `head`. |
| `--peer NAME` typo'd | Return `peer "X" not found in config` error from `Analyze` — surfaces as non-zero exit (the one place `analyze` is non-zero, and only on argument error). |
| User passes a *client* config, not server | `ParseServerConfig` reads only server fields; client configs partially parse without `Address`/`ListenPort`. Detect: if `cfg.Interface.PrivateKey == ""` AND `len(cfg.Peers) == 0`, return `config has no server data — did you pass a client config?`. |
| Vanilla WG config (no AWG fields) | Heuristic `vanilla-wg-config` emits WARN. All padded sizes equal raw sizes (S1=...=S4=0 → padded == raw). Junk range `[0..0]` → `width=0` triggers `junk-width-zero`. Output is correct, just predictably "vanilla shape". |
| Config with `Jmin > Jmax` (corrupt) | `ValidatePacketSizes` returns `ErrEmptyJunkRange`. `analyze` should not surface this as a hard error — emit a finding `WARN junk-range-malformed` and skip the junk-related heuristics. Document in `scanRisks`. |
| `--samples 1000` performance | `GenerateCPS` is dominated by `crypto/rand` calls (microseconds each). 1000 iterations × 5 slots × ~10 RNG calls ≈ 50k RNG draws ≈ tens of milliseconds. Acceptable. Cap at 1000 enforced. |
| Snapshot test reproducibility | Step F injects a `math/rand`-backed `io.Reader` via `AnalyzeOptions.Rand`. Unbounded byte stream, so over-provisioning is moot. No package-global mutation, so `t.Parallel()` is safe. |
| Seed exposure in CI logs | `--seed N` values are NOT cryptographic secrets — they only seed `math/rand`-backed I-packet sample generation for reproducibility. However, a seed combined with a real obfuscation config deterministically reveals one possible I-packet shape. **Document in `--seed` flag help text and in user-facing notes**: do not paste real configs to public forums alongside the seed used to generate the snapshot — the combination may leak peer-count or shape characteristics that an adversary could correlate. CI logs containing seed-only references (without the config) are harmless. |
| Distribution mode reproducibility | Use `--seed N` for reproducible distribution output (math/rand-backed). Without `--seed`, distribution is non-deterministic by design and uses `crypto/rand`. |
| JSON output schema breaking changes between versions | This PR establishes the JSON schema. Document it in `cobra.Command.Long`. Any future change to field names is a breaking change — bump `--api-version` flag if schema evolves (out-of-scope for this PR; flag the consideration in commit message). |
| Heuristics false-positives on legitimate configs | Severity is INFO/WARN, never ERROR. User can grep them out. Each heuristic has a stable `Code` for filtering: `analyze --output json | jq '.risks[] | select(.code != "s4-low-padding")'`. |
| Newly added WG message size (e.g. AWG 3.0 introduces a 5th type) | `wgInitiationSize=148, ...` are constants in `validation.go`. If AWG ever adds a 5th type, those need updating *throughout the codebase*, and `analyze`'s handshake table needs a 5th row. Out-of-scope; AWG 2.0 has four. |
| Output too wide for narrow terminals | `tabwriter` aligns columns; users on 80-col terminals see wrapping. Acceptable — this is a developer/operator tool, not a phone app. |
| Concurrent calls to `Analyze` | `Analyze` is pure (no shared state). `crypto/rand` is goroutine-safe. The injected `Rand io.Reader` is caller-owned — if shared across goroutines, the caller is responsible for thread-safety (typical seeded readers are not goroutine-safe). For tests, pass per-test seeded readers; do not share. Safe to call from parallel tests when each gets its own reader. |
| Nondeterministic `GenerateCPS` makes snapshot tests fragile | Mitigated by injecting a seeded `io.Reader` via `AnalyzeOptions.Rand` (Step F). No package-global mutation. If maintenance becomes painful, switch snapshot to assertion on *structure* (lengths in expected ranges) rather than bytes. Defer that to a follow-up if needed. |

## 12. Out of Scope (Explicit)

The following are **deliberately excluded** from this PR. If any come up during implementation, do not include them; note them for follow-up.

- **`--compare <preset>` mode** — depends on P1.5 preset library landing first. The `AnalysisReport` struct is designed so a future `Comparison` field is purely additive.
- **Graphical plotting beyond ASCII text tables** — no `gonum/plot`, no `chart` libraries. The distribution table (§ 8.3) is the upper bound on visualization in this PR. ASCII bar charts (e.g. for size histograms) are a possible follow-up but not required.
- **Live packet-capture verification** — no `tcpdump`, no `pcap`, no real socket binding. Analysis is purely arithmetic from the static config. Verification against actual on-the-wire bytes is a P3.x concern.
- **Machine-learning fingerprint scoring** — no entropy estimators beyond the simple cluster-width check, no statistical-distance metrics, no model loading. The heuristics in § 9 are deterministic rules.
- **Modifications to `validate` (P1.3)** — `analyze` *consumes* `ValidatePacketSizes` and `validateHeaderRange` as-is. If those need to be extended for any reason, that's a P1.3 follow-up, not this PR.
- **Peer config (`*.conf` exported via `export`) as input** — `analyze` accepts only server configs. Analyzing a client `.conf` requires a different parser (currently doesn't exist) and a different report shape (no `Peers` array). Defer to a future `analyze --client <peer.conf>` enhancement.
- **Schema versioning of JSON output** — no `--api-version` flag, no `"version": "1"` field in JSON. Document the schema in `cobra.Command.Long` and treat it as v1 implicitly. Schema changes will need explicit follow-up.
- **i18n / localised output** — text output is English-only.
- **YAML output** — JSON is sufficient; adding YAML pulls in `gopkg.in/yaml.v3`.
- **Custom heuristic plugins (third-party `.so` loading)** — `heuristics` is a package-private slice. Extending it requires a recompile.
- **Output piping integration with `validate`** — no `analyze | validate` UNIX shape; the two are separate commands with separate exit semantics by design.
- **Refactoring unrelated code** — `cps.go`, `generator.go`, `validation.go` stay untouched except for accessor exports if (and only if) needed.
